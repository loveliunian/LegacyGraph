**目标**：把 `lg_solution_step.codeSnippet` 注入 `ChangeTask.proposal`，生成可 `git apply` 的 patch。

**新增文件**：

- `dto/change/ProposalFile.java` — 单个文件的 patch 描述

**改造文件**：
- [SolutionToChangeTaskBridge.java](file:///Users/huymac/工作/数智/LegacyGraph/backend/src/main/java/io/github/legacygraph/service/solution/SolutionToChangeTaskBridge.java)
  - 注入 `SolutionStepRepository`
  - bridge 时读取所有 step 的 codeSnippet
  - 调用 `PatchComposer` 生成 diff
  - 组装 `ChangeTaskProposal` 写入 `ChangeTask.proposal`
- [ChangeTaskService.java](file:///Users/huymac/工作/数智/LegacyGraph/backend/src/main/java/io/github/legacygraph/service/change/ChangeTaskService.java)
  - `generatePatch()` 方法支持从 proposal 读取（而非从 Agent 重新生成）
  - 新增 `applySolutionProposal(taskId, solutionId)` 入口

**proposal JSON 结构**：

```json
{
  "files": [
    {
      "filePath": "src/main/java/.../OrderService.java",
      "op": "MODIFY",
      "symbolName": "OrderService#exportLast30Days",
      "diff": "@@ -10,3 +10,5 @@ ...",
      "evidenceIds": ["ev-001"],
      "testDescription": "...",
      "rollbackDescription": "..."
    }
  ]
}
```

**PatchComposer 实现策略**：
- CREATE：codeSnippet 整体作为新文件内容，生成 `--- /dev/null +++ b/file` diff
- MODIFY：先用 `filePath` 在项目目录中找原文件，逐行匹配插入点；找不到原文件回退为 CREATE
- DELETE：生成 `--- a/file +++ /dev/null` diff 占位

**验收**：
- 3 step APPROVED 方案 bridge 后 `proposal.files.length == 3`
- MODIFY 步骤的 diff 通过 `git apply --check` 校验
- 单测：`PatchComposer` 对 CREATE/MODIFY/DELETE 三种操作各 1 组用例
