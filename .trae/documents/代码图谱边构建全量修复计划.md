# 代码图谱边构建全量修复计划

## 概述

修复代码图谱中 Controller → Service → Mapper → SQL → Table 完整链路的边构建缺陷，涵盖 P0 致命问题（5项）、P1 高风险问题（4项）、P2 中风险问题（关键项）。

## 当前状态分析

预期链路：`ApiEndpoint →(HANDLED_BY)→ Method →(CALLS)→ Method →(EXECUTES)→ SqlStatement →(READS/WRITES)→ Table →(HAS_COLUMN)→ Column`

实际链路断裂点：
1. **Method → SqlStatement (EXECUTES)**：并发时序导致 Method 节点未创建时边被跳过，无补偿
2. **Method → Method (CALLS)**：JavaMemberCallResolver 索引只覆盖 3 种类型 + resolveOne 无消歧 + @Resource 注入漏识别
3. **SqlStatement → Table (READS/WRITES)**：Table nodeKey 大小写不一致导致指向错误节点
4. **Table → Column (HAS_COLUMN)**：nodeKey 格式三处不统一
5. **BELONGS_TO / EXTENDS / IMPLEMENTS**：只覆盖 Controller/Service/Mapper 三类

## 修复项

### P0-1: Method → SqlStatement (EXECUTES) 边并发时序丢失

**文件**: `builder/GraphBuilder.java`、`task/ProjectScanner.java`

**问题**: `buildMapperSqlGraph` 在并发执行时，Mapper 接口的 Method 节点可能尚未创建，`findMapperMethodNode` 返回 null，EXECUTES 边被静默跳过。

**修复方案**: 在 MEMBER_CALL_RESOLVE 之后新增 `MAPPER_SQL_LINK` 后置阶段，重新执行 Method → SqlStatement 边创建。

**具体改动**:

1. **GraphBuilder.java** 新增方法（约第 500 行，`buildMapperSqlGraph` 之后）:
```java
/**
 * 后置连接：在所有适配器执行完成后，重新执行 Method → SqlStatement 的 EXECUTES 边创建。
 * 补偿并发执行时因 Method 节点尚未创建而遗漏的边。
 */
public int linkMapperMethodsToSqlStatements(String projectId, String versionId) {
    // 1. 查询所有 Mapper 节点
    // 2. 对每个 Mapper，查询其关联的 SqlStatement 节点（通过 CONTAINS 边）
    // 3. 对每个 SqlStatement，用 findMapperMethodNode 查找 Method 节点
    // 4. 若找到且 EXECUTES 边不存在，则创建
    // 返回新创建的边数
}
```

2. **ProjectScanner.java** 在第 669 行之后（MEMBER_CALL_RESOLVE 之后、EXTERNAL_VERIFY 之前）插入:
```java
// 3c. Mapper-SQL 后置连接：补偿并发时序导致的 Method→SqlStatement 边遗漏
if (isCancelled(versionId)) return;
if (shouldScanCode) {
    ScanTask linkTask = createTask(projectId, versionId, "MAPPER_SQL_LINK", "Mapper-SQL后置连接");
    try {
        int linked = graphBuilder.linkMapperMethodsToSqlStatements(projectId, versionId);
        completeTask(linkTask, "linked " + linked + " method-sql edges", null);
    } catch (Exception e) {
        log.warn("Mapper-SQL link failed (non-blocking): versionId={}, err={}", versionId, e.getMessage());
        completeTask(linkTask, "failed: " + e.getMessage(), e.getMessage());
    }
}
```

3. **GraphBuilder.findMapperMethodNode** 性能优化（第 2093-2122 行）:
   - 当前每次调用全量查询所有 Method 节点，改为在 `linkMapperMethodsToSqlStatements` 中一次性加载所有 Method 节点到内存索引，复用查找。

**文件**: `builder/JavaMemberCallResolver.java` 第 79 行

**问题**: `simpleNameToClassNodes` 和 `fqnToClassNode` 只索引三类节点，被调用方是 Component/Utils/Helper 等时 CALLS 边缺失。

**修复方案**: 扩展索引覆盖范围。

**具体改动**:

### P0-3: resolveOne 无消歧且不回退反向匹配

**文件**: `builder/JavaMemberCallResolver.java`

**问题**:
1. `resolveOne` 中 `candidates.size() > 1` 直接跳过，无消歧逻辑
2. `targetClass` 已解析但索引找不到时不回退到 `resolveByMethodName`

**修复方案**:

1. **第 208-210 行** — resolveOne 增加消歧（复用已有的 `disambiguateCandidates` 方法）:
```java
// 修改前:
if (candidates.size() > 1) {
    return null; // 歧义（同名多类），跳过，不留错边
}
GraphNode targetClassNode = candidates.get(0);

// 修改后:
GraphNode targetClassNode;
if (candidates.size() == 1) {
    targetClassNode = candidates.get(0);
} else {
    // 同名多类消歧：同包优先 → 语义关联
    targetClassNode = disambiguateClassCandidates(candidates, call.callerClass);
    if (targetClassNode == null) {
        return null; // 无法消歧，跳过
    }
}
```

2. 新增 `disambiguateClassCandidates` 方法（复用 `disambiguateCandidates` 的同包过滤 + 语义关联逻辑，但针对类节点而非方法节点）。

3. **第 147-158 行** — targetClass 已解析但 resolveOne 返回 null 时回退反向匹配:
```java
// 修改前:
if (!isBlank(call.targetClass)) {
    edge = resolveOne(...);
    if (edge != null) forwardResolved++;
} else {
    edge = resolveByMethodName(...);
    if (edge != null) reverseResolved++;
}

// 修改后:
if (!isBlank(call.targetClass)) {
    edge = resolveOne(...);
    if (edge != null) {
        forwardResolved++;
    } else {
        // 正向匹配失败（索引找不到或歧义），回退到反向方法名匹配
        edge = resolveByMethodName(...);
        if (edge != null) reverseResolved++;
    }
} else {
    edge = resolveByMethodName(...);
    if (edge != null) reverseResolved++;
}
```

### P0-5: isControllerFile 路径判断与 supports 内容判断不一致

**文件**: `extractors/adapter/JavaCodeAdapter.java` 第 152-156 行、`extractors/adapter/JavaServiceCallAdapter.java` 第 56 行

**问题**: `isControllerFile` 按路径名判断，`supports` 按文件内容判断，导致部分文件被两个适配器都跳过或都处理。

**修复方案**: `JavaServiceCallAdapter.supports` 中改用与 `JavaCodeAdapter.supports` 相同的内容判断逻辑排除 Controller 文件。

**具体改动**:

`JavaServiceCallAdapter.java` 第 55-58 行:
```java
// 修改前:
// Controller 文件交给 JavaCodeAdapter，避免重复
if (JavaCodeAdapter.isControllerFile(asset.getRelativePath())) {
    return false;
}

// 修改后:
// Controller 文件交给 JavaCodeAdapter，避免重复（按内容判断，与 JavaCodeAdapter.supports 一致）
String content = asset.getCachedContent();
if (content != null) {
    if (content.contains("@RestController") || content.contains("@Controller")) {
        return false;
    }
} else if (asset.getFile() != null && Files.isReadable(asset.getFile())) {
    try (var lines = Files.lines(asset.getFile())) {
        if (lines.anyMatch(line -> line.contains("@RestController") || line.contains("@Controller"))) {
            return false;
        }
    } catch (IOException e) {
        // 读取失败时不排除，让 JavaServiceCallAdapter 处理
    }
}
```

### P1-1: Table nodeKey 大小写不一致

**文件**: `builder/GraphBuilder.java`

**问题**:
- DB 元数据路径（第 552 行）: `nodeKey = schema.tableName`（原始大小写）

**修复方案**: DB 元数据路径的 Table nodeKey 统一为小写。

**具体改动**:

// 修改前:
String tableKey = tableMeta.getTableSchema() + "." + tableMeta.getTableName();
// 修改后:
String tableKey = (tableMeta.getTableSchema() + "." + tableMeta.getTableName()).toLowerCase();

2. 第 562 行: 同样修改为 `.toLowerCase()`

3. 第 553 行 `tableNameToKey` map 的 value 自动随 tableKey 变更，无需额外修改

4. `addTableNodeIfAbsent` 方法（第 510-523 行）: 确保创建新 Table 节点时 nodeKey 也用小写。当前已用 `tableName.toLowerCase()`，保持一致。但需要确保 `seenNodeKeys` 中的 key 也是小写格式（因为 DB 路径的 tableKey 现在也是小写了）。

## 假设与决策

1. **NodeType 扩展决策**: NodeType 枚举中没有 `Component` 类型。大部分 Utils/Helper 类在 `inferNodeType` 中被归为 `Service`。因此 P0-2 的扩展只增加 `ConfigItem` 和 `ExternalSystem`，不做枚举新增。
2. **Table nodeKey 格式决策**: 统一保留 `schema.tableName.toLowerCase()` 格式（DB 元数据路径），SQL 解析路径因不知道 schema 仍用 `tableName.toLowerCase()`，但通过 `tableNameToKey` map 桥接。若 map 中找不到（表不在 DB 元数据中），则创建不带 schema 的小写 Table 节点。
3. **后置连接阶段位置**: 放在 MEMBER_CALL_RESOLVE 之后，因为此时所有 Method 节点已创建完成。
4. **isControllerFile 方法保留**: 不删除该方法（可能有其他调用方），但 JavaServiceCallAdapter 不再使用它。
5. **不处理的低优先级问题**: P2-5（链式调用 scope 解析）、P2-6（动态 SQL JSqlParser 干扰）等需要较大改动的中风险问题暂不处理，避免过度工程化。
