```
[1] Input           →   [2] Extraction        →   [3] Graph Build        →   [4] Graph Sync & Query     →   [5] Summary Doc
(代码/数据库/文档)     (抽取器/适配器)            (GraphBuilder/FrontendGraphBuilder/BusinessGraphBuilder)  (Neo4j/PG + 查询接口)             (报告/RAG 上下文)
```

## 2. 节点 / 边 schema 与存储一致性

### 2.1 NodeType（49 种）/ EdgeType（59 种）枚举与实际使用

| 评估点 | 代码位置 | 状态 |
|---|---|---|
| `NodeType` 完整覆盖业务/功能/代码/需求/方案/验收/变更闭环 | `common/NodeType.java` | ✅ 涵盖：Project / System / BusinessDomain / Process / Object / Rule / Role / FeatureModule / Feature / Menu / Page / Button / Permission / ApiEndpoint / Controller / Service / Method / Mapper / SqlStatement / Table / Index / Column / ConfigItem / ScheduledJob / MQConsumer / MQTopic / ExternalSystem / TestCase / Assertion / Requirement / Solution / ChangeTask / Patch / PR … |
| `EdgeType` 与方法论 §5.2 对应率 | `common/EdgeType.java` | ✅ 涵盖 24+ 主边 + P1-2 新增 READS_DB/WRITES_DB/CALLS_DB + 需求结构化 HAS_ITEM/HAS_ACCEPTANCE_CRITERION + 方案 SATISFIES/STEP_OF/VALIDATED_BY |
| 索引/约束自动建 | `dao/Neo4jSchemaRepository.createConstraints/createIndexes` | ✅ 每个 NodeType label：`id UNIQUE` + `(projectId, versionId, nodeKey) IS UNIQUE`；每个 label 建 `(projectId,versionId)` 与 `nodeKey` 两个索引 |
| **唯一性问题** | `GraphBuilder.buildApiNodes` | ⚠️ `ApiNodeKey = normalizeApiKey(METHOD, fullPath)`，**同一 METHOD+PATH 重载无版本号**；重载方法会得到同一 `ApiEndpoint` nodeKey，后写的会覆盖前一个 |

**结论**：骨架扎实；唯一性唯一风险在 `ApiEndpoint` 重载层面，但 controller 上 path/method 组合唯一时无问题。

### 2.2 `lg_fact` ↔ `lg_graph_node` 一致性

| 评估点 | 代码位置 | 状态 |
|---|---|---|
| `Fact` 表保留事实原文 JSON | `entity/Fact.java` + Flyway `V1__initial_schema.sql` / `V5__create_missing_tables.sql` | ✅ 唯一约束 `(project_id, version_id, fact_type, fact_key)` 防重复抽取 |
| `GraphNode` 写入走 `EvidenceGraphWriter.upsertNode` | `builder/EvidenceGraphWriter.java` | ✅ 消除简单的去重竞争 |
| **`saveFact` 中 `source_type` 总是 `CODE_AST`**（不管 `fact_type` 是 `MAPPER` / `FRONTEND_PAGE` / `FRONTEND_API`） | `task/ProjectScanner`（推断），详见 `三类图谱获取实现说明.md §8.1` | ⚠️ `Fact.source_type` 与 `GraphNode.source_type` **取值不一致**：图节点正确（由各自 builder 写），但事实表层错。因此从 `lg_fact` 反查事实来源时一律 `CODE_AST`，对文档/数据库类事实无差别意义 |
| **`GraphNode.verifiedScore` 持久化** | `entity/GraphNode.java`（`@TableField(exist = false)`）+ 运行链路 `markRuntimeVerified` | ⚠️ 字段声明为非持久化，运行验证回写存在但**不写入数据库**（详见 `三类图谱获取实现说明.md §8.10`） |

1. `saveFact` 的 `source_type` 写死 `CODE_AST`
2. `verifiedScore` 标记无法回写到 PG，导致闭环看板的"运行时校验"维度统计可能为空

### 3.2 [Extraction] 抽取器 / Adapter

#### 3.2.1 后端：Controller / Service / Mapper 抽取

| 抽出器 | 文件 | 适配性 | 评估 |
|---|---|---|---|
| **JavaControllerExtractor**（被 adapter 包装） | `JavaCodeAdapter` + `JavaControllerExtractor` | @RestController + 6 种 @*Mapping | ✅ |
| **JavaStructureExtractor** | `extractors/JavaStructureExtractor.java` | Class/Method/Field/接口/注解/继承关系；**Lombok 虚拟方法生成**（@Data/@Getter/@Setter/@Builder） | ✅ |
| **ServiceCallExtractor** | `extractors/ServiceCallExtractor.java`（本文已深读） | 字段注入(@Autowired/@Inject/@Resource) + 构造器注入 + Lombok `@RequiredArgsConstructor` final 字段推断 + JavaParser SymbolSolver（带 sourceRoot 时） + ThreadLocal JavaParser | ★★★★☆ |
| **JavaMemberCallResolver** | `builder/JavaMemberCallResolver.java`（本文已深读） | **二次扫描**：建立全局 (FQN|methodName) 索引；god-node 守卫（歧义/缺失一律跳过）；同包优先 → 参数个数 → 签名精确 → import 语义 | ★★★★★ **本项目当前最强的静态调用解析** |

**实测（用 LegacyGraph 自身的代码做被扫描对象）**：
- Controller 48 → ✅ 平台应该全识别（与 README 一致）
- Service interface 103 个 → ✅ 被识别为 Service 类节点
- 全部 0 个 `@Service` 注解 → **所有 Service 都是接口 + 实现类分离**，Extractor 通过文件名（`Service.java` 后缀）匹配
- 0 个 XML Mapper、0 个 `@Mapper` 注解、8 个 `extends IService / BaseMapper`、53 个 `@Select/@Insert/@Update/@Delete` 注解 → **没有 XML 数据可扫，但有 53 个 MyBatis-Plus 注解方法**

**🔴 硬断裂 #1 — MyBatis 注解方法被全部忽略**
- 位置：`builder/GraphBuilder.buildMapperSqlGraph` 只接受 `MapperSqlFact`（来自 XML 抽取器），而本仓库与绝大多数新型 Spring Boot 项目用 `extends BaseMapper` + `@Select/@Insert/@Update/@Delete` 注解式 Mapper
- 影响：`SqlStatement` 节点 ≈ 0；`Mapper -EXECUTES-> SqlStatement` ≈ 0；`SqlStatement -READS/WRITES-> Table` ≈ 0
- 这是**对 LegacyGraph 自身**也是最大的链路断点；用 `doc/三类图谱的方法论.md` "推荐的 MyBatis XML 解析规则" 套到注解式 Mapper 上，命中率近零
- 修复建议：增加 `JavaMyBatisAnnotationExtractor`：扫 `@Select/@Insert/@Update/@Delete` + `extends BaseMapper` 接口，把注解式 SQL 也喂给 `SqlTableExtractor` (JSqlParser 已支持 SQL 字符串解析)

**🔴 硬断裂 #2 — 实现类（XxxServiceImpl）的不对称命名**
- Service interface 与实现类文件命名（interface 文件匹配 `*Service.java`，Impl 文件不匹配）
- 实测：`find backend/src/main/java -name "*ServiceImpl.java"` = 0（本项目使用 interface + impl，但 Impl 文件名不一定是 Impl，详见接口实现习惯分歧）
- `ServiceCallExtractor` 用 `isServiceOrMapperFile` 匹配文件名包含/结尾 `Service`、`Mapper`、`Dao`——**所有命中**
- 但 `GraphBuilder.buildServiceCallGraph` 如何对 interface ↔ impl 做关联？需要在 `JavaMemberCallResolver` 的实现类解析下落到接口
- 修复建议：在 `JavaMemberCallResolver` 跑完后，新增 `JavaInterfaceImplLinker`：扫类文件，把 `class A extends B<C>` / `class A implements I` / 字段注入类型与实现类建立链接，更新 `Service` nodeKey 包含所有 `Impl`

#### 3.2.2 前端：路由 / API 调用 / Button / Permission

`FrontendGraphBuilder`（已深读）+ `VueRouteExtractor` + `FrontendApiExtractor`：
- ✅ Page / Menu / Button / Permission 节点建立
- ✅ 前后端 API 匹配基础分数：method(+0.3) + path(+0.5) + normalized path(+0.4) + param-count match(+0.1)
- ⚠️ **method/param/response 字段相似度未实现打分**（设计文档 §13.4 提到的 `request 字段相似 +0.1 / response 字段相似 +0.1` **未在 `FrontendGraphBuilder.calculateMatchScore` 内落地**）
- 🟠 **`button.getApiUrl()` 硬编码 `"POST " + normalizedPath`**（`FrontendGraphBuilder.java:232`），未解析 button 上的真实 HTTP method，导致 button 命中 GET 类接口时**必失败**
- ⚠️ 依赖 `node_modules` 路径过滤（`FrontendApiExtractor` 含 npm 路径正则），但 `package.json` 中转译/动态 URL（`/api/${prefix}/${x}`）仍需 `normalizePath`

**修复建议**：
1. 实现 method/param/response 字段相似度计算（增加 `requestFields.json` 抽象 + LCS 或 Jaccard 相似度）
2. button 的 HTTP method 从 button.attributes（如 `:http-method`、`<el-button http-method>`、自定义 prop）或父级 handler 中提取
3. 把 frontend API 节点的 `frontend:` 前缀和后端 `ApiEndpoint` 节点的归一 key 关系在 `findBackendApi` 中显式 handle

#### 3.2.3 业务事实抽取

`DocUnderstandingAgent` + `BusinessGraphBuilder.buildBusinessGraph`：
- 业务域 / 业务流程（同时建一个 `Feature` 节点）/ 业务对象 / 业务规则 / 角色 / 直接列出的 features 都建节点
- ⚠️ **「BusinessProcess 不再自动归到 BusinessDomain」** 是有意为之（`BusinessGraphBuilder.java:154-156` 注释：因为 LLM 输出 process 没有 domain 字段）；导致业务图谱中大量 Process 孤立节点
- ⚠️ `DocumentParsingRouter.threeLayerEnabled=false` 默认关闭 → Layout/OCR 解析走 Fast 兜底；扫描件 PDF/图片文档提取质量差

**修复建议**：
1. 加一个 AI 步骤：`BusinessProcessToDomainStep`，输入所有 `BusinessDomain` + `BusinessProcess` 名称，LLM 决策归类并建边
2. 三层解析默认改为 true（已经设计了 3 个 Strategy，不开等于浪费）
3. 把 `extractDocFacts` 编排进 `AiScanOrchestrator` 主流水（确认已经接，看 `task/step/` 下相关 step executor）

#### 3.2.4 Service ↔ Mapper CALLS 边的真实可达性

`ServiceCallExtractor` + `JavaMemberCallResolver` 二次解析，能给出 **Service -CALLS-> Service.method(s)** 和 **Method -CALLS-> Mapper.method**。但当目标 Mapper 节点不存在（因为 Mapper 是接口且 XmlExtractor 没扫到）时，`Map<String, List<GraphNode>> methodIndex` 拿不到 Mapper 节点，**mapping 会回退到类级 CALLS**，缺少 `Method -CALLS-> Mapper Method` 细粒度。

## 12. 结论与下一步

1. **总体设计完整**：业务/功能/代码三类图谱从资料到 Neo4j 的整条链路**有完整骨架**，6 个抽取器 + 3 个 Builder + 5 个 AI 映射函数 + 2 个跨语言实体合并函数 + 1 个 Neo4j schema 自初始化 + 18+ 查询端点 + 5 维质量度量 + Graphify 交叉验证管线接口
2. **有 5 处硬断裂 / 关键缺陷**（§3 + §6.1），可在 1–2 周内修完，主要是：
   - P0：注解式 MyBatis Mapper 不识别（影响 SQL→表 链路）
   - P0：Service interface ↔ impl 链路脆
   - P0：`getFeatureView`/`getBusinessView` 看不出节点（关键属性没写）
   - P0：Neo4j 只写 CONFIRMED → 业务视图落不到 Neo4j
   - P0：Cypher 路径查询未按 versionId 过滤
3. **图谱质量门禁** 依赖外部工具（Graphify/JOERN/Semgrep/PM4Py）和 LLM 可用性，目前都是设计级；建议在所有修复完成后做一次"对着 LegacyGraph 自己跑全量"作为基准：
   - LegacyGraph 自报数：48 Controller / 103 Service / 74 Mapper / 8 ext-BaseMapper / 53 MyBatis 注解
   - Graphify 已有数：10 865 节点 / 29 409 边
   - **目标数**：48 API / 103 Service 类 / 8 Mapper（XML + 注解）/ 53 SqlStatement（注解式）/ 表 N / 数千 Page（前端）
4. **下游可消费文档**：方法论、设计、当前实现、QA 评测、Graphify 报告四类文档已经齐了 §10.1；§10.2 的补全项也有对应代码预留位置
5. **80% 结论 / 20% 落地**：现状是"方法论 + 设计文档 + 当前实现文档" 三层齐备，但**实际把 ExternalVerification / Graphify / PM4Py 跑起来** 的开关化与默认启用，是下一阶段最值钱的工程动作

**给"后面业务（QA 问答，测试，制定方案）"的明确建议**：

- **QA 问答**：在 P0 修复前，**避免问"某个接口读写哪些表"、"这条 SQL 来自哪个 Mapper"，回答常常不准确**；修完后依赖 `ServiceCallExtractor → JavaMemberCallResolver → MyBatis 注解/Xml 抽取 → SqlTableExtractor` 链路一次性打通
- **自动化测试**：测试生成可以继续做（图谱覆盖已经在 QA 评测服务里），但**测试结果回写到图谱的 `confidence` 字段**目前不完整，要先解决 `verifiedScore` 持久化
- **方案 / 迁移**：基于现有 `ImpactSubgraphService / MigrationAgent / PatchPlanAgent` 已经能给初步方案；要做精度更高的"新老双写一致性"才需要 PM4Py 与跨仓联邦
