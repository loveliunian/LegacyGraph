## Part A — 人读 FAQ 卡片

> 每条回答附 `证据来源`，对齐 `01` §7 证据卡片字段。`sourceType`：CODE / DB / DOC / TEST / RUNTIME。

### A1. 系统总览类

**Q1. LegacyGraph 是做什么的？**
A：企业级系统分析与知识图谱平台，把代码库、数据库、文档、前端页面连接成统一知识网络，再投影出业务/功能/代码三类图谱。核心理念：静态分析给事实，LLM 做归纳补全，自动测试反证，人工审核兜底。
证据来源：`README.md` §项目简介（DOC）。

**Q2. 三类图谱分别是什么？数据来源？**
A：

| 图谱 | 内容 | 数据来源 |
|---|---|---|
| 代码图谱 | Controller/Service/Mapper/SQL/表/字段调用链 | Java AST + MyBatis XML + JDBC 元数据 |
| 功能图谱 | 页面/按钮/API/权限/菜单 → 后端接口映射 | Vue 组件 + axios + Controller 注解 |
| 业务图谱 | 业务域/流程/角色/对象/规则/状态流转 | 文档 AI 抽取 + 人工确认 |

证据来源：`README.md` §三类图谱（DOC）；`common/NodeType.java`（CODE）。

**Q3. 图谱节点和边存在 Neo4j 还是 PostgreSQL？**
A：**Neo4j** 是图谱节点/边的唯一读写存储（`Neo4jGraphDao` 幂等 MERGE）。**PostgreSQL** 只存证据关联（`lg_evidence` / `lg_node_evidence` / `lg_edge_evidence`）和审核记录。`lg_graph_node` / `lg_graph_edge` 表已废弃，仅保留 Bean 兼容，**当前业务代码不再写入**。
证据来源：`doc/整体技术文档/架构设计文档.md` §图谱模块（DOC）；`dao/Neo4jGraphDao.java`、`builder/EvidenceGraphWriter.java`（CODE）。

**Q4. 节点类型和关系类型有多少种？以什么为准？**
A：节点类型 49 种（`NodeType.java`），关系类型 59 种（`EdgeType.java`）。所有节点带 `evidence`、`confidence`(0–1)、`status`(`CONFIRMED`/`PENDING_CONFIRM`/`REJECTED`)。
证据来源：`common/NodeType.java`、`common/EdgeType.java`（CODE）。

**Q5. 扫描涉及哪些数据库表？**
A：`lg_scan_version`（扫描版本）、`lg_scan_task`（扫描子任务）、`lg_source_asset_snapshot`（资产快照/增量）、`lg_ai_scan_job`（AI 扫描异步任务）、`lg_document`（文档元数据）、`lg_doc_chunk`（文档分片）。版本是图谱可追溯的边界，所有事实/任务/证据/节点/边/Claim/Gap 都挂在同一 `project_id + version_id` 下。
证据来源：`doc/整体技术文档/数据库设计文档.md` §表总览（DOC）；Flyway V20/V24/V28（DB）。

**Q6. QA 问答用到哪些表？**
A：`lg_qa_conversation`（会话）、`lg_qa_message`（消息，含 evidences/confidence）、`lg_qa_feedback`（反馈，含 used_evidence_ids）、`lg_semantic_cache`（语义缓存）、`lg_vector_document`（向量）、`lg_doc_chunk`（文档片段）、`lg_knowledge_claim`（GraphRAG Claim 输入）。
证据来源：Flyway V29/V30（DB）；`agent/EnhancedQaAgent.java`（CODE）。

**Q7. 资料扫描的完整流程是什么？**
A：`ProjectScanner.runScanBody` → `ScanScopeResolver` 解析范围 → `AssetDiscoveryService` 发现资产 → 5 类 Adapter（Java/Service/MyBatis/Vue/Doc）+ `DatabaseMetadataScanService` 抽取 → `GraphBuilder`/`FrontendGraphBuilder` 经 `EvidenceGraphWriter` 写 Neo4j+PG → `AiScanOrchestrator` 编排 AI（文档抽取/代码抽取/功能映射/测试生成/审核准备/缺口发现）。
证据来源：`doc/项目升级计划/资料扫描到三类图谱构建流程与AI优化研究.md` §2（DOC）；`task/ProjectScanner.java`、`task/AiScanOrchestrator.java`（CODE）。

**Q8. 业务、功能、代码、数据四层怎么串起来？**
A：核心链路 `BusinessDomain CONTAINS Feature EXPOSED_BY ApiEndpoint HANDLED_BY Controller/Method IMPLEMENTED_BY Service CALLS Mapper EXECUTES SqlStatement READS/WRITES Table HAS_COLUMN Column`。任一层可沿此链路向前追到业务、向后追到数据。
证据来源：`01-关系总览与映射框架.md` §3、§4（DOC，基于 `EdgeType.java`）。

### A2. 业务域类（每域一问）

**Q9. 项目与数据源管理域的功能/代码/数据？**
A：功能 `ProjectController`(`/lg/projects`)、`SourceController`(`/sources`)、`AuthController`(`/lg/auth`)；前端 `project`/`source`/`login`/`dashboard`；代码 `ProjectService`/`SourceService`/`AuthService`/`JwtUtil`；数据 `lg_project`/`lg_code_repo`/`lg_db_connection`/`lg_document`/`lg_sys_user`/`lg_sys_role`。
证据来源：`02` §1（DOC，基于 Controller 基路径 + Flyway）。

**Q10. 图谱构建与查询域的关键类？**
A：`EvidenceGraphWriter`（统一写入器）、`Neo4jGraphDao`（Neo4j 唯一入口）、`GraphBuilder`/`FrontendGraphBuilder`/`BusinessGraphBuilder`、`GraphQueryService`、`GraphWriteIntentService`+`GraphWriteIntentWorker`（outbox）。写入同时落 Neo4j + PG，失败标 `INCOMPLETE`，由 `GraphWriteReconciler` 对账。
证据来源：`02` §3（DOC）；`dao/Neo4jGraphDao.java`、`builder/EvidenceGraphWriter.java`（CODE）。

**Q11. QA 问答的意图类型和图谱深度？**
A：`FACT_LOOKUP`(深度1,不用Planner)、`STRUCTURAL`(2,用)、`RELATIONAL`(3,用)、`COMPARATIVE`(2,不用)、`TEMPORAL`(1,不用)、`EXPLANATION`(2,不用)。简单事实查询不触发 GraphRAG Planner，避免延迟和噪声。
证据来源：`doc/项目升级计划/QA问答优化方案.md` §4.1（DOC）；`agent/EnhancedQaAgent.java`（CODE）。

**Q12. 测试闭环的断言分几层？置信度怎么回写？**
A：四层断言：协议(HTTP/JSONPath) + 业务(状态/角色) + 数据库(表/行) + 图谱(`Feature→API→Function→Table` 路径)。回写：PASS +0.10、FAIL −0.20、人工审核通过 +0.15。
证据来源：`02` §6（DOC）；`README.md` §测试闭环（DOC）。

**Q13. Graphify 集成是默认开启的吗？数据存哪？**
A：**默认关闭**（`legacygraph.graphify.enabled`）。可选集成外部 Graphify CLI，生成 `graph.json` 经校验/映射导入 Neo4j。**无专属表**，复用 `lg_scan_version`/`lg_ai_scan_job`/Neo4j。
证据来源：`README.md` §Graphify 集成（DOC）；`02` §10（DOC）。

**Q14. LLM 调用怎么缓存？审计在哪？**
A：`LlmGateway` 统一调用，支持多模型路由、Prompt 模板渲染、PII 脱敏、结构化校验、`lg_prompt_run` 审计(token/latency/inputHash)、Redis 结果缓存(TTL 7d)。Redis 不可用时静默降级回源。
证据来源：`README.md` §LLM Agent 体系、§Redis 缓存体系（DOC）。

**Q15. 插件和配额有专属表吗？**
A：**没有**。插件（`plugin/PluginRegistry`/`BuiltinPluginRegistrar`）基于内存注册；配额（`tenant/TenantQuotaManager`）基于内存 + Redis/`lg_sys_config`。
证据来源：`02` §12（DOC）；`plugin/`、`tenant/` 包（CODE）。

### A3. 缺口与边界类

**Q16. 哪些问题不能高置信回答？**
A：
- 956 个孤立节点（`recordToolUse.sh`、`OperationType`、`EdgeType`、`SourceType` 等 ≤1 连接）的关系——⚠ 缺口。
- 223 条 `getMessage()` 推断边（INFERRED）——不能作为 `CONFIRMED` 事实引用。
- Service Layer / Frontend 社区内聚度低（cohesion 0.05/0.04）——"功能→代码"映射边界模糊。
- 仅文档存在、仅运行时存在、静态有但运行时无观测的三类差异——须显式标记，不能偷偷覆盖。
证据来源：`graphify-out/GRAPH_REPORT.md` §Knowledge Gaps、§Suggested Questions（DOC/RUNTIME）；`01` §6.3、§8（DOC）。

**Q17. AI 抽取的事实能直接当确认事实用吗？**
A：**不能**。AI 来源默认 `PENDING_CONFIRM`，只能进 `lg_knowledge_claim` 作候选。须经 `KnowledgeCompiler` 编译、冲突检测、评分，或经代码/DB/运行时/测试/人工审核补证后才升为 `CONFIRMED`。AI 输出回答时只能引用证据卡片中的事实，无证据的内容必须进"推断/待确认"章节。
证据来源：`01` §6.4、§7（DOC）；`service/KnowledgeClaimService.java`、`service/KnowledgeCompiler.java`（CODE）。

## Part C — 与内置 QA 的对接说明

### C1. 如何把本底座喂给 `EnhancedQaAgent`

| 路径 | 方式 | 字段对齐 |
|---|---|---|
| 检索增强 | Part B 表内容向量化入 `lg_vector_document` / `lg_doc_chunk`，供 `HybridRetrievalService` 召回 | `sourceType`/`sourcePath`/`nodeKey`/`excerpt` |
| GraphRAG Claim | Part B 每行作为 `KnowledgeClaim` 写入 `lg_knowledge_claim`，供 `GraphRagPlanExecutor` 按 `subjectKeys` 检索 | `claimId`/`confidence`/`status` |
| 语义缓存 | Part A FAQ 的 question embedding 入 `lg_semantic_cache`，相似问题直接命中 | question_embedding/answer/evidence |
| 证据卡片 | 回答时展开为 `GraphRagEvidenceCard`，`relationTypes`/`pathNodeKeys` 来自 Part B2 | 全字段对齐 `01` §7 |

### C2. 回答约束（底线）

1. **有证据才回答**：引用 Part B 行或证据卡片；无证据进"推断/待确认"章节。
2. **置信度标注**：`CONFIRMED` 直接答；`PENDING_CONFIRM` 标注"待确认"；缺口表内的标"⚠ 暂无法高置信回答"。
3. **来源差异显式化**：文档/代码/DB/运行时说法矛盾时，列全来源，不偷偷覆盖。
4. **追溯可展开**：任一结论能沿 Part B2 链路展开节点/关系序列。

### C3. 与既有 QA 优化方案的关系

本底座是 `doc/项目升级计划/QA问答优化方案.md` 中"GraphRAG Claim 相关性筛选"和"评测集"的事实输入：

## Part E — H28: QA 评测集 v2 基线

> 评测集：`backend/src/test/resources/qa-evaluation/test-set-v2.json`（100 题，5 类意图 × 20 题）
> 详细基线文档：`doc/图谱构建完整性与准确性评估.md §13`

### E1. 基线目标

| 意图分类 | 题数 | 基线目标 | 说明 |
|---|---|---|---|
| FACT_LOOKUP | 20 | 95% | 事实查找：节点基本信息查询 |
| RELATIONSHIP | 20 | 90% | 关系查询：节点间边/路径 |
| INFERENCE | 20 | 80% | 推断：基于图谱的逻辑推断 |
| CHANGE_IMPACT | 20 | 70% | 变更影响：字段/接口变更的影响范围 |
| ENUMERATION | 20 | 95% | 列举：满足条件的节点集合 |
| **合计** | **100** | — | |

### E2. 评测接口

- `POST /lg/projects/{projectId}/qa/eval-v2/run` — 运行 v2 评测集
- `GET /lg/projects/{projectId}/qa/eval-v2/baseline` — 查询基线目标
- `GET /lg/projects/{projectId}/qa/eval-runs` — 查询历史评测运行

### E3. CI 集成

- 每日 UTC 18:00（北京时间 02:00）cron 跑评测
- 回归 > 5% 输出 GitHub Actions warning
- 评测结果落库 `lg_qa_evaluation_run` 表
