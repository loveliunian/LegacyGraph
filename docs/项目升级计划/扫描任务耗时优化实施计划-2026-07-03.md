# 扫描任务耗时优化 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 将 LegacyGraph 扫描任务从“长时间黑盒运行”优化为可观测、可控、可增量、可分阶段验收的扫描流水线，并优先压缩代码扫描、数据库扫描和 AI 后处理耗时。

**Architecture:** 先修复进度与阶段耗时可观测性，再收敛 `ProjectScanner` 主链路到统一任务记录器和并发 Adapter 执行器，随后把高频逐条写库/写图改成批处理，最后把 AI 与数据库重扫改为显式、增量、异步的后处理。所有改动以 `scanScope` 为控制面，以 `ScanTask`/`ScanVersion` 为观测面，以回归测试和一次真实小扫描为验收面。

**Tech Stack:** Java 21, Spring Boot 4, MyBatis-Plus, PostgreSQL/Flyway, Neo4j Java Driver, Redis cache, Vue 3/Element Plus.

### 1.3 立即暴露的问题

1. 运行库 `flyway_schema_history` 有 V23、V25，但没有 V24；本地 V24 是 `lg_scan_task` 进度列迁移，线上表缺 `total_items / processed_items / current_item`。
2. `ScanVersionService.getScanProgress` 只把 `SUCCESS` 算完成，`WARNING` 不计入；关闭 AI 时会生成 `AI_ORCHESTRATION:WARNING`，导致终态仍显示 80%。
3. `ScanScopeResolver` 读 `aiEnabled`，`AiScanConfig` 和前端读写 `enableAi`，同一个扫描范围里存在两个 AI 开关字段。
4. `AdapterExecutionService.executeScan` 已有虚拟线程并发，但主链路仍走 `ProjectScanner.scanDiscoveredAssetsWithAdapters` 的顺序循环。
5. Adapter 内部大量逐条 `FactRepository.upsert`、逐节点/逐边写 Neo4j，远程 PostgreSQL/Neo4j 环境下会把文件级循环放大成大量网络往返。
6. `ProjectScanner` 和 `AiScanOrchestrator` 仍各自保留私有 `createTask/completeTask`，已存在的 `ScanTaskRecorder` 没有统一接管主链路。

## 2. 范围与非目标

### 本计划覆盖

- 扫描任务进度、阶段时间、终态进度修复。
- AI 开关字段统一和兼容。
- Adapter 主链路并发化。
- 确定性抽取写入批处理。
- 数据库扫描增量化。
- AI 后处理默认可跳过、可异步、可单独触发。
- 前端扫描进度展示的数据契约修正。

### 本计划不覆盖

- 不重做三类图谱模型。
- 不切换到新的图数据库。
- 不把所有 Builder 一次性改成 Claim Compiler 投影。
- 不要求在第一阶段重构全部 1500 行扫描器；先按可观测、可测、可回滚的小步收敛。

## 3. 目标指标

| 阶段 | 指标 |
|---|---|
| P0 可观测性 | 终态任务进度必须为 100%；所有子任务有 `started_at/finished_at`；进度列存在并可写入 |
| P1 开关与范围 | `enableAi=false` 时不跑 AI heavy 阶段；`scanTypes`、`maxFiles`、`incremental` 均由一个解析器消费 |
| P2 代码扫描 | 当前 `game-01` 代码-only、AI 关闭扫描控制在 5 分钟内；中型仓库不因单文件失败中断 |
| P3 批处理 | 同等仓库 Adapter 阶段耗时降低 50% 以上；Neo4j/PG 写入往返数可通过日志采样看到下降 |
| P4 数据库扫描 | 未变更 schema 的数据库扫描跳过或在 1 分钟内结束；显式 `DB_SCAN` 时才扫库 |
| P5 AI 后处理 | 默认扫描不被 AI 阻塞；AI 增强可独立排队、可取消、可重试 |

## 5. 分阶段实施

### Phase 0：先修复观测面

目标：扫描终态和进度可信，阶段耗时可查。

- 运行库即使已经跳到 V25，也能通过 V26 补齐扫描任务进度列。
- `WARNING` 作为终态完成计入整体进度。
- 版本已经 `SUCCESS / FAILED / CANCELLED / PAUSED` 时，整体进度不再停在中间值。
- 所有新建子任务都通过 `ScanTaskRecorder`，确保 `startedAt` 一致写入。

### Phase 1：统一 scanScope 与任务记录

目标：同一个 scope 字段只在一个地方解释，AI 开关不再错读。

- `ScanScopeResolver` 兼容 `enableAi` 和旧 `aiEnabled`，内部统一输出 `enableAi` 语义。
- `AiScanConfig` 兼容旧 `aiEnabled`，但前端和新接口只写 `enableAi`。
- `ProjectScanner` 和 `AiScanOrchestrator` 删除私有重复 `createTask/completeTask`，统一注入 `ScanTaskRecorder`。

### Phase 2：主链路接入并发 Adapter

目标：把当前顺序处理文件的 `scanDiscoveredAssetsWithAdapters` 收敛到一个并发执行器。

- `AdapterExecutionService` 支持接收已发现资产列表，避免重复 walk。
- 并发度用配置控制，默认 8，最大不超过数据库连接池可承受范围。
- 单文件失败记录 warning，不中断全局扫描。
- 取消信号在文件处理前和批次之间检查。

### Phase 3：批量写 PostgreSQL 与 Neo4j

目标：减少网络往返，解决 Adapter 阶段 30 分钟级耗时。

- `FactPersister` 增加 `saveFacts`。
- `FactRepository` 增加 `batchUpsert`。
- `Neo4jGraphDao` 增加 `mergeNodesBatch`、`mergeEdgesBatch`。
- GraphBuilder 从“循环中即时写”逐步改成“先收集 NodeUpsert/EdgeUpsert，再批写”。

### Phase 4：数据库扫描增量化

目标：显式 `DB_SCAN` 才扫库，未变更 schema 不重复扫。

- 为每个连接记录 schema fingerprint。
- include/exclude tables 在扫描层强制生效。
- 扫描结果表数、列数、跳过原因写入 `ScanTask.outputSummary`。
- 数据库 LLM enrichment 继续异步，但必须有独立任务记录和失败可见性。

### Phase 5：AI 后处理拆出异步增强任务

目标：默认扫描快速完成，AI 增强作为独立任务继续运行或手动触发。

- `enableAi=false` 时不创建 heavy AI 子任务，只记录一个 `AI_ORCHESTRATION:SKIPPED` 或 `WARNING`。
- `enableAi=true` 时扫描主任务结束后排队 `AiScanJob`，不阻塞 `ScanVersion` 的基础扫描完成。
- AI 子任务可以单独取消、重试、查看 token/latency。

### Phase 6：前端展示修正

目标：前端看到的是可解释的阶段时间线，而不是只看粗略百分比。

- 版本列表展示扫描基础阶段和 AI 增强阶段两个状态。
- 进度详情展示每个 task 的 `startedAt/finishedAt/seconds/totalItems/processedItems/currentItem`。
- 终态进度为 100%，但保留 WARNING 数量和 warning 摘要。

```java
                                   List<SourceAsset> assets,
                                   ScanTask task,
                                   java.util.function.Supplier<Boolean> cancelChecker,
                                   boolean incremental,
                                   AssetDiscoveryService assetDiscoveryService) {
    if (context == null || assets == null || assets.isEmpty() || adapterRegistry == null) {

    }

    int total = assets.size();
    taskRecorder.logProgress(task, 0, total, "adapter candidate files");
    int maxConcurrency = Math.max(1, Integer.getInteger("legacy-graph.scan.adapter-concurrency", 8));
    Semaphore semaphore = new Semaphore(maxConcurrency);
    AtomicInteger visited = new AtomicInteger(0);
    AtomicInteger processed = new AtomicInteger(0);

    List<Callable<Void>> calls = assets.stream()
            .<Callable<Void>>map(asset -> () -> {
                if (cancelChecker != null && cancelChecker.get()) {
                    return null;
                }
                semaphore.acquire();
                try {
                    if (incremental && assetDiscoveryService != null
                            && assetDiscoveryService.isIncrementalSkip(asset, context.getProjectId(), context.getVersionId())) {
                        return null;
                    }
                    var adapter = adapterRegistry.selectAdapter(context, asset);
                    if (adapter.isPresent()) {
                        ExtractionResult result = adapter.get().extract(context, asset);
                        processed.addAndGet(result != null && result.getProcessedAssets() > 0
                                ? result.getProcessedAssets() : 1);
                    }
                } catch (Exception ex) {
                    log.warn("Adapter failed for discovered asset {}: {}",
                            asset.getRelativePath(), ex.getMessage());
                } finally {
                    semaphore.release();
                    int done = visited.incrementAndGet();
                    taskRecorder.logProgress(task, done, total, "adapter candidate files");
                }
                return null;
            })
            .toList();

    try (ExecutorService executor = Executors.newVirtualThreadPerTaskExecutor()) {
        executor.invokeAll(calls);
    } catch (InterruptedException ex) {
        Thread.currentThread().interrupt();
    }
    return processed.get();
}
```

## 9. 风险与回滚

| 风险 | 触发点 | 回滚策略 |
|---|---|---|
| 并发写 Neo4j/PG 导致连接池耗尽 | Adapter 并发度过高 | 配置 `legacy-graph.scan.adapter-concurrency=1` 回到串行 |
| 批写改变幂等语义 | batch upsert SQL 冲突键错误 | 保留单条 `saveFact`，按 Adapter 灰度切换 |
