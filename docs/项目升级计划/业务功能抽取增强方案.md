# 业务/功能内容抽取增强方案

> **目标：** 解决"全量扫描后业务图谱（业务域/流程/功能/对象/角色/规则）经常为空"的问题，让图谱不仅有技术节点（Controller/API/Mapper/Table），也稳定产出业务与功能语义。
>
> **涉及项目：** `LegacyGraph/backend`
>
> **核心文件：** `task/AiScanOrchestrator.java`、`builder/BusinessGraphBuilder.java`、`agent/DocUnderstandingAgent.java`、`agent/CodeFactAgent.java`、`common/SourceType.java`、`common/EdgeType.java`
>
> **最后更新：** 2026-07-02 — 代码复核 + 全部修复 + 测试补充完成（507 tests / 0 failures）
> 
> **主要变更：** SourceType 参数化、Feature key 归一化、enableAi 可观测性、P1-C 路径对齐、P2 实体对齐扩展、FeatureIdentityNormalizer 整合、AgentEnvelope 重命名、GraphQueryService 编译修复、8 个新测试用例

---

## 一、背景与目标

业务图谱为空的根因是**业务事实抽取过度依赖三个前提同时成立**：

1. `enableAi=true`（AI 编排开关）
2. 项目里发现到文档（PRD/需求文档）
3. LLM 抽出了结构化的业务内容

任何一个不满足，图谱即为空 —— 且大多只留一行 `log.info`，排查困难。

增强方向：

- **代码来源**：让 Service/Controller 的方法名、注释、枚举也能产出业务语义，摆脱对文档的强依赖。
- **多层 Feature**：无论 LLM 输出的是流程步骤、直接功能清单还是代码事实，都稳定落为 Feature 图节点。
- **可观测性**：AI 未启用 / 无文档 / 无代码候选 / LLM 无结果四类空图原因，在 `scan_task` 中显式可见。
- **来源追溯**：文档来源 vs 代码来源在 Fact 表和图节点上正确区分。
- **业务-技术连通**：BusinessObject 桥接到 Table + Service + Mapper，消除业务层与技术层的割裂。

## 三、增强方案与实现

### P0-A｜代码侧业务事实抽取

**问题：** 业务事实 100% 依赖文档；无文档项目业务图谱必为空。

**方案：**
- 在 `AiScanOrchestrator` 新增子任务 `runCodeExtract(projectId, versionId)`，任务类型 `AI_CODE_EXTRACT`。
- 查询 Neo4j 中已有的 Service/Controller 节点（上限 30 个），读取源码 → `CodeFactAgent.extractFacts()` → LLM 抽取业务语义。
- 抽取结果桥接为 `BusinessFactExtraction.features`，经 `persistAndBuildCodeFacts()` 分别落 Fact 表和图节点。

**实现细节：**

```
runCodeExtract()
 ├─ 查询 Service + Controller 节点（按 sourcePath 去重）
 ├─ 逐个读取源文件内容（上限 8000 字符）
 ├─ CodeFactAgent.extractFacts() → FactExtractionResult
 └─ persistAndBuildCodeFacts()
     ├─ saveAiFact(..., SourceType.CODE_AI)   # CODE_FEATURE 事实，来源正确标记
     └─ buildBusinessGraph(..., SourceType.CODE_AI)  # Feature 图节点，来源正确标记
```

**效果：** 即使项目零文档，也能从代码产出 Feature 节点，且 `sourceType=CODE_AI` 可追溯到代码来源。

**已知限制：** 当前 `FactExtractionResult.FactItem` 仅桥接为 Feature；扩展为 BusinessDomain / BusinessProcess / BusinessObject / BusinessRule 需升级 `code-fact-extraction` 模板输出结构（⏳ 后续）。

### P0-B｜features 清单落图 + Feature key 归一化

**问题：** `persistBusinessFacts` 已把 `extraction.getFeatures()` 落成 FEATURE 事实，但 `buildBusinessGraph` 此前不消费它 —— 事实表有、业务图谱无。此外，不同来源的 Feature 使用不同 nodeKey 格式（`process.getName()` / `"feature:" + name` / `"code-feature:" + name`），相同语义可能产生重复节点。

**方案：**

1. `buildBusinessGraph` 遍历 `facts.getFeatures()` 创建 Feature 图节点（与流程派生 Feature 独立）。
2. 新增 `normalizeFeatureKey()` 方法，统一去前缀（`feature:` / `code-feature:` / `code_feature:`）、中文标点半角化（全角括号/冒号→半角）、合并连续空白，所有 Feature 节点创建前先归一化 key。

**实现代码（BusinessGraphBuilder）：**

```java
// P0-B：消费 LLM 直接给出的功能清单
if (facts.getFeatures() != null) {
    for (String feature : facts.getFeatures()) {
        if (feature == null || feature.isBlank()) continue;
        String key = normalizeFeatureKey(feature);  // 归一化去重
        findOrCreateNode(projectId, versionId, NodeType.Feature.name(),
                key, feature, feature, null,
                sourceType, sourcePath, null, null,
                BigDecimal.valueOf(0.7), NodeStatus.PENDING_CONFIRM);
    }
}
```

**效果：** 事实表与业务图谱两层一致；多来源同语义 Feature 合并为一个节点。

**后续可升级：** 整合已有 `FeatureIdentityNormalizer` 类，增加来源优先级排序（如 DOC_AI > CODE_AI）。

### P1-A｜流程 Feature 解耦于 steps

**问题：** 只有 `BusinessProcess.steps` 非空才建 Feature 节点；LLM 未吐 steps 时，流程存在但一个 Feature 都没有。

**方案：**
- `BusinessProcess` 本身即落一个粗粒度 Feature（流程即功能），不再要求 `steps` 非空。
- `steps` 存在时再派生细粒度子 Feature，均建 `CONTAINS` 边。

**实现代码（BusinessGraphBuilder）：**

```java
// P1-A：流程本身 = 粗粒度 Feature
GraphNode processFeatureNode = findOrCreateNode(
        projectId, versionId, NodeType.Feature.name(),
        normalizeFeatureKey(process.getName()),  // 归一化 key
        process.getName(), process.getName(), process.getDescription(),
        sourceType, sourcePath, null, null,
        BigDecimal.valueOf(process.getConfidence() * 0.9), NodeStatus.PENDING_CONFIRM);
createEdge(..., EdgeType.CONTAINS, ...);  // 流程 CONTAINS 功能

// steps 存在时派生细粒度 Feature
if (process.getSteps() != null) {
    for (String step : process.getSteps()) {
        // ... 同逻辑，nodeKey = normalizeFeatureKey(process.getName() + "/" + step)
    }
}
```

### P1-B｜可观测性增强：静默 → 显式 WARNING

**问题：** `enableAi=false` 时仅 `log.info` + `return`，scan_task 表无任何记录，无法区分"没扫到"与"没开扫"。空结果任务仍标记 `SUCCESS`，淹没在正常任务中。

**方案（两处改动）：**

1. **enableAi=false 创建跳过任务：**

```java
// AiScanOrchestrator.orchestrate()
if (config == null || !config.isEnableAi()) {
    ScanTask skipTask = createTask(projectId, versionId, "AI_ORCHESTRATION", "AI 编排");
    completeTask(skipTask,
            "⚠ AI 编排已跳过：enableAi=false（未启用 AI 归纳）。"
            + "业务图谱将不会生成。如需生成，请在 scanScope 中设置 enableAi=true。",
            null);
    return;
}
```

2. **空结果标 WARNING 状态：**

```java
// completeTask() — 摘要以 ⚠ 开头 → taskStatus = "WARNING"
if (error != null) {
    task.setTaskStatus("FAILED");
} else if (summary != null && summary.startsWith("⚠")) {
    task.setTaskStatus("WARNING");   // 与 SUCCESS 区分
} else {
    task.setTaskStatus("SUCCESS");
}
```

3. **文档抽取摘要区分场景：**

```java
// buildDocExtractSummary()
if (docCount == 0)  → "⚠ 未发现任何文档 —— 业务事实 0 条。请确认..."
if (factCount == 0) → "⚠ 扫描文档 N 个，但未抽取到业务事实..."
else                → "AI 抽取业务事实 N 条，扫描文档 N 个"
```

**效果：** 四类空图原因（AI 未启用 / 无文档 / 无代码候选 / LLM 无结果）在 `scan_task.outputSummary` 中均可区分，WARNING 状态可筛选。

**待定：** `enableAi` 默认值是否改为 `true`（⏳ 待产品决策，需考虑 LLM 调用成本）。

### P1-C｜落库路径对齐 + Feature-代码映射

**问题（双面）：**

- **路径不一致：** 手动 `FactController.extractDocFacts()` 会调用 `mapFeaturesToCode()` 建 Feature↔Page/API 桥接边，自动 `runDocExtract()` 不调用 —— 自动 Feature 成孤岛。
- **映射范围窄：** `mapFeaturesToCode()` 只查询 `sourceType=DOC_AI` 的 Feature，代码抽取的 Feature 永远不会被映射。

**方案：**

1. **新增 `runFeatureCodeMapping` 步骤**（`AI_FEATURE_CODE_MAPPING`），编排中 `runCodeExtract()` 之后自动执行：

```java
private void runFeatureCodeMapping(String projectId, String versionId) {
    businessGraphBuilder.mapFeaturesToCode(projectId, versionId);       // Feature → Page/API
    businessGraphBuilder.mapBusinessObjectsToTables(projectId, versionId); // BusinessObject → Table/Service
}
```

2. **`mapFeaturesToCode` 移除 sourceType 过滤**，查询所有 Feature 节点（含 DOC_AI + CODE_AI）。

3. **`runFeatureMapping` 以 Feature 为锚点**：查询已有 Feature 节点作为 `productDoc` 上下文传入 LLM，让 LLM 以已有 Feature 为坐标系做 Page/API 映射。

**效果：** 自动 + 手动路径的 Feature 对齐行为一致；代码 Feature 也能接到 Page/API。

**后续可升级：** 抽出 `BusinessFactIngestionService` 统一四条路径（文档自动/文档手动/代码自动/代码手动），固定执行：事实落库 → 图构建 → Feature 对齐 → 审核生成（⏳ 架构重构）。

### P2｜业务-技术实体对齐

**问题：** 文档抽出的 `BusinessObject`（如"订单"）与 DB 的 `orders` 表、代码的 `OrderService` 之间没有桥接边，业务层与技术层是两张不连通的图。

**方案：** `mapBusinessObjectsToTables()` 将 BusinessObject 按名称相似度同时对齐到三类技术实体：

| 技术实体 | 节点类型 | 建立的边 | 阈值 |
|---------|---------|---------|------|
| 数据库表 | `Table` | `MAPS_TO` | 0.6 |
| 服务类 | `Service` | `IMPLEMENTED_BY` | 0.65 |
| 数据访问层 | `Mapper` | `IMPLEMENTED_BY` | 0.65 |

名称相似度处理：转小写、去表前缀（`t_/tb_/sys_/biz_`）、去下划线、Jaccard 词级匹配 + 包含关系 bonus。

**效果：** 业务对象可沿 `MAPS_TO` / `IMPLEMENTED_BY` 边直接导航到数据库表和代码实现，业务视图不再割裂。

---

### 补充修复｜SourceType 参数化

**问题（代码复核发现）：** `saveAiFact()` 和 `buildBusinessGraph()` 内部硬编码 `SourceType.DOC_AI`。代码事实经 `persistAndBuildCodeFacts()` 写入时，Fact 表和图节点的 `sourceType` 都被错误标记为 `DOC_AI`。而 `SourceType.CODE_AI` 枚举早已存在但从未使用。

**修复：**

- `saveAiFact(..., String sourceType)` — 新增参数，代码路径传 `SourceType.CODE_AI.name()`，文档路径传 `SourceType.DOC_AI.name()`。
- `buildBusinessGraph(..., String sourceType)` — 新增 4-param 重载；原 2-param/3-param 重载保持兼容，默认传 `DOC_AI`。
- `persistAndBuildCodeFacts()` — 两处调用均传入 `CODE_AI`。

**回溯兼容：** 文档抽取路径（`persistBusinessFacts` / `buildBusinessGraph(doc)`) 行为不变，仍为 `DOC_AI`。

## 四、当前编排流程

修复后的 `orchestrate()` 执行 6 步流水线：

```
orchestrate()
 ├─ (enableAi=false?) → 创建 AI_ORCHESTRATION 跳过任务(WARNING) → return
 ├─ 1. AI_DOC_EXTRACT        # 文档 → LLM → BusinessFactExtraction → Fact + 图节点
 ├─ 2. AI_CODE_EXTRACT       # Service/Controller 源码 → CodeFactAgent → CODE_FEATURE + Feature 图节点
 ├─ 3. AI_FEATURE_CODE_MAPPING  # Feature → Page/API 名称匹配 + BusinessObject → Table/Service/Mapper 对齐
 ├─ 4. AI_FEATURE_MAPPING    # 已有 Feature + Page/API → LLM 补充映射
 ├─ 5. AI_TEST_GENERATE      # (可选) 高价值 API 节点 → 测试用例
 └─ 6. AI_REVIEW_PREPARE     # 低置信节点 → 人工审核记录
```

**子任务类型与状态一览：**

| 步骤 | taskType | 空结果/跳过时的行为 |
|------|----------|-------------------|
| 编排入口 | `AI_ORCHESTRATION` | enableAi=false → WARNING + "enableAi=false..." |
| 文档抽取 | `AI_DOC_EXTRACT` | 0 文档 → WARNING + "未发现任何文档..." |
| 代码抽取 | `AI_CODE_EXTRACT` | 无 Service/Controller 或未抽到代码事实 → WARNING + 可见原因 |
| Feature-代码映射 | `AI_FEATURE_CODE_MAPPING` | 未建立 Feature/Page/API 或 BusinessObject/技术实体映射 → WARNING + 可见原因 |
| 功能映射 | `AI_FEATURE_MAPPING` | 无 Page/API → SUCCESS + "跳过" |
| 测试生成 | `AI_TEST_GENERATE` | 仅 `autoGenerateTestCase=true` 时创建 |
| 审核准备 | `AI_REVIEW_PREPARE` | 无低置信节点 → SUCCESS + "0 条" |
