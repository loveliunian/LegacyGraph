# LegacyGraph 需求到编码闭环实施方案

> 基于《LegacyGraph 与图谱及需求到编码工具的差距调研报告》与当前代码现状
> 版本：v1.0 | 日期：2026-07-13

---

## 一、现状定位

### 1.1 已有能力（调研报告确认）

| 能力模块 | 当前状态 | 证据 |
|---------|---------|------|
| 需求结构化 | 完整 | `Requirement`/`RequirementItem`/`AcceptanceCriterion` 实体 + 表 |
| 方案生成 | 完整 | `Solution`/`SolutionStep` + `SolutionController` + `SolutionPlanner` |
| 方案审批 | 完整 | 状态机 `APPROVED`/`REJECTED` + `SolutionAudit` |
| 影响分析 | 完整 | `ChangeImpactAgent` + `ImpactSubgraph` + `ImpactSubgraphService` |
| 方案→变更桥接 | 完整 | `SolutionToChangeTaskBridge` + `ChangeTask` 状态机 |
| 变更任务管理 | 完整 | `ChangeTaskService` + `ChangeTaskController` |
| 补丁草案生成 | 完整 | `PatchPlanAgent` + `PatchPlan` DTO + 多种 Adapter |
| 补丁校验 | 完整 | `PatchPlanValidator` 三类校验（范围/格式/证据）|
| 验证门禁 | 完整 | `ValidationGateRunner` + STATIC/UNIT/API/DB/E2E/MIGRATION |
| PR 编排 | 存在 | `PrOrchestrator` + `PrTask` 实体 |
| 图谱质量门禁 | 完整 | `GraphQualityAssessor` + `GraphReleaseStatus` |

### 1.2 缺失能力（调研报告指出）

| 优先级 | 缺失项 | 影响 |
|--------|--------|------|
| P0 | **安全基线治理** | 无法进入企业试点 |
| P1 | **执行沙箱** | 补丁无法自动验证可执行性 |
| P1 | **PR 真正落地** | PR 创建但未真正推送到远程仓库 |
| P1 | **需求入口连接器** | 需求需手工粘贴，无法对接 Jira/Confluence |
| P2 | **补丁生成能力强化** | PatchPlan Agent 覆盖面有限 |
| P2 | **跨仓影响分析** | 只能分析单个仓库 |
| P3 | **Benchmark 评测体系** | 无法量化证明系统价值 |

### 1.3 当前 ChangeTask 完整状态机

```
OPEN → IMPACT_READY → PATCH_DRAFTED/REVIEW_PENDING → VALIDATING
     → VALIDATION_PASSED/VALIDATION_FAILED → PR_READY → PR_CREATED
     → MERGED/REJECTED/ROLLED_BACK
```

**调研报告结论**：系统"已经具备'需求到方案'的中间层能力，但尚未完成'方案到补丁/PR/可部署变更'的工程闭环"。

## 三、分阶段实施方案

### 阶段一：安全止血与评测基线（P0，第 1-2 月）

#### 1.1 安全基线治理

**目标**：消除最严重安全阻断项，否则无法进入企业试点。

| 任务 | 具体动作 | 涉及文件 |
|------|---------|---------|
| 密钥清理 | 删除 `.env.local` 中的密钥，重写 Git 历史 | `.env.local` |
| 默认配置收口 | `application.yml` 默认 profile 改为 `prod`，移除 fallback secret | `application.yml` |
| CORS 收紧 | `SecurityConfig` 中 `allowedOriginPatterns("*")` 改为白名单 | `SecurityConfig.java` |
| 公开端点清理 | Swagger/QA 流式端点默认加认证 | `SecurityConfig.java` |
| JWT 密钥 | 生成强随机密钥，支持密钥轮换 | `application.yml` |
| Secret 扫描 | 接入 Git hooks / CI secret 扫描（如 `trufflehog`）| - |

**关键代码改动点**：

```java
// SecurityConfig.java 目标配置
.allowedOriginPatterns("https://legacygraph.example.com")  // 白名单
.allowCredentials(true)
.maxAge(3600)

// application.yml 目标配置
spring:
  profiles:
    active: prod  // 默认 prod 而非 dev
```

#### 1.2 Benchmark 评测体系

**目标**：把"看起来很强"变成"可以量化对比"。

| 指标类别 | 具体指标 | 评测方式 |
|---------|---------|---------|
| 需求抽取 | 需求结构化 F1、开放问题识别率 | 黄金数据集 + 人工标注 |
| 图谱链接 | Impact 精确率/召回率 | 影响分析结果对比人工分析 |
| 影响分析 | Precision@K（K=1,3,5）| 受影响文件对比 |
| 方案质量 | 方案可执行率、人工修改率 | 开发者评分 |
| 补丁质量 | Patch 构建通过率、测试通过率 | 自动验证 |
| 端到端 | 任务完成时间、返工次数 | A/B 测试 |

**落地建议**：
- 建立 `eval/` 包：参考 `backend/src/test/java/io/github/legacygraph/eval/`
- 创建黄金数据集：6 类场景（新增字段、API 变更、权限变更、前端改动、跨仓依赖、旧表迁移）
- 接入 CI：每次 PR 自动运行评测

### 阶段二：方案粒度下沉与 Patch Draft 中间层（P1，第 3-4 月）

#### 2.1 方案输出升级为文件级变更计划

**目标**：从"方案摘要"升级成"开发可执行任务单"。

当前 `SolutionStep` 已有 `filePath`、`actionType`、`codeSnippet` 字段，需强化：

```java
// SolutionStep 增强
public class SolutionStep {
    private String filePath;           // 已有
    private String actionType;         // 已有：CREATE/MODIFY/DELETE
    private String codeSnippet;       // 已有
    private String symbolName;        // 新增：影响的符号（类名/方法名）
    private String changeScope;       // 新增：FULL/PARTIAL（完整替换 vs 部分修改）
    private String rollbackPlan;      // 新增：回滚计划
    private List<String> testFiles;   // 新增：需新增/修改的测试文件
    private List<String> evidenceIds; // 新增：支撑此步骤的证据 ID
}
```

#### 2.2 新增 PatchDraftService 中间层

**意义**：把"方案"与"真正写代码"之间强制分层，让每一步都有证据、审计、验证和回放能力。

```java
/**
 * 补丁草案服务 — 需求到编码的关键中间层。
 * <p>
 * 将已审批的 Solution 转换为可验证的 PatchDraft，
 * 每个 Draft 包含文件级 diff、证据引用、风险评估。
 * </p>
 */
public interface PatchDraftService {
    /**
     * 从 Solution 创建补丁草案。
     */
    PatchDraft createDraft(String projectId, String versionId, String solutionId);

    /**
     * 校验草案：范围、格式、证据完整性。
     */
    PatchValidationReport validateDraft(String patchDraftId);

    /**
     * 物化草案：生成 unified diff 并持久化。
     */
    ChangePackage materialize(String patchDraftId);
}
```

**PatchDraft 实体设计**：

```java
@Data
public class PatchDraft {
    private String id;
    private String solutionId;
    private String projectId;
    private String versionId;
    private String status; // DRAFT / VALIDATED / MATERIALIZED / EXPIRED
    private List<DraftFile> files;
    private String riskLevel;
    private String generatedBy; // llm / manual
    private LocalDateTime createdAt;
    private LocalDateTime validatedAt;
}

@Data
public class DraftFile {
    private String filePath;
    private String changeType; // CREATE / MODIFY / DELETE
    private String originalContent; // 可选：原始内容
    private String newContent;   // 期望的新内容
    private String diff;         // unified diff
    private List<String> evidenceIds;
    private List<String> symbolNames; // 影响的符号
    private String validationStatus; // PENDING / PASS / FAIL
    private String validationMessage;
}
```

#### 2.3 强化 Bugfix 补丁生成

当前 `PatchPlanAgent` 主要覆盖 BUGFIX，需扩展到更多场景：

| 任务类型 | 当前状态 | 需增强 |
|---------|---------|--------|
| BUGFIX | 可用 | 增强证据聚合能力 |
| REFACTOR | 可用 | `RefactorAgentAdapter` 已存在 |
| UPGRADE | 可用 | `MigrationAgentAdapter` 已存在 |
| ADD_COLUMN | 可用 | `AddColumnPatchAgentAdapter` 已存在 |
| API_CHANGE | 缺失 | 新增 `ApiChangePatchAgent` |
| FRONTEND_CHANGE | 缺失 | 新增 `FrontendPatchAgent` |

### 阶段三：执行沙箱与 PR 流水线（P1-P2，第 5-6 月）

#### 3.1 执行沙箱层

**目标**：自动运行编译、测试、lint、契约验证、回归校验。

```
验证门禁类型：
├── STATIC     → 静态分析（SpotBugs/PMD/CheckStyle）
├── UNIT       → 单元测试（Maven/Gradle）
├── API        → API 测试（REST Assured）
├── DB         → 数据库验证（DbAssertionExecutor）
├── E2E        → 端到端测试（Playwright）
├── MIGRATION  → 数据库迁移验证
└── MANUAL     → 人工复审
```

**关键实现**：

```java
/**
 * 沙箱执行服务 — 在隔离环境中验证补丁。
 * <p>
 * 支持两种模式：
 * - LOCAL：本地执行（适用于有 localPath 的 CodeRepo）
 * - CONTAINER：容器化执行（适用于隔离验证）
 * </p>
 */
public interface SandboxExecutor {
    SandboxResult execute(SandboxRequest request);
}

@Data
public class SandboxRequest {
    private String changeTaskId;
    private List<DraftFile> files;
    private List<String> gateTypes; // STATIC, UNIT, API, DB, E2E
    private String workingDir;      // 命令类门禁工作目录
    private String environment;      // test / staging
}

@Data
public class SandboxResult {
    private boolean success;
    private String buildOutput;
    private List<TestResult> testResults;
    private List<Violation> violations;
    private Map<String, String> metrics;
}
```

**与当前系统的对接**：

- 复用 `ValidationGateRunner` 作为底层执行器
- 复用 `TestExecutionScheduler` 作为测试调度
- 新增 `SandboxEnvironmentManager` 管理隔离环境

#### 3.2 PR 流水线完整实现

**当前状态**：`PrOrchestrator` + `PrTask` 已存在，但未真正推送到远程仓库。

**目标**：打通 `PrTask` → 远程仓库 PR 的完整链路。

```java
/**
 * PR 落地编排服务 — 打通补丁到 PR 的最后一公里。
 * <p>
 * 流程：
 * 1. 创建分支（基于 targetBranch）
 * 2. 应用补丁（PatchApplicator）
 * 3. 提交并推送（GitCommandExecutor）
 * 4. 创建 PR（GitHub/GitLab API）
 * 5. 等待 CI 状态（Webhook 或轮询）
 * 6. 回写结果到图谱
 * </p>
 */
public interface PrOrchestrator {
    /**
     * 创建 PR 草案并推送到远程仓库。
     */
    PrResult createAndPushPr(PrRequest request);

    /**
     * 查询 PR 状态。
     */
    PrStatus getPrStatus(String prTaskId);

    /**
     * 处理 PR 合并回调。
     */
    void onPrMerged(String prTaskId);
}

@Data
public class PrRequest {
    private String changeTaskId;
    private String sourceBranch;
    private String targetBranch;
    private String commitMessage;
    private String prTitle;
    private String prBody;
    private List<DraftFile> files;
    private Map<String, String> metadata;
}
```

**GitHub/GitLab 集成**：

```java
/**
 * Git 提供商适配器 — 支持 GitHub / GitLab / Gitea。
 */
public interface GitProviderAdapter {
    void createBranch(String repoUrl, String branchName, String baseBranch);
    void applyPatch(String repoUrl, String branchName, List<DraftFile> files);
    void pushBranch(String repoUrl, String branchName);
    String createPullRequest(CreatePrRequest request);
    PrStatus getPrStatus(String prUrl);
    void addReviewer(String prUrl, List<String> reviewers);
}
```

**与现有 CodeRepo 实体集成**：

```java
// CodeRepo 已有字段扩展
public class CodeRepo {
    private String remoteUrl;           // 已有
    private String localPath;           // 已有
    private String provider;             // 新增：github / gitlab / gitea
    private String accessToken;         // 已有但需加密存储
    private String defaultBranch;       // 新增
    private String reviewTeam;          // 新增：默认 review 团队
}
```

## 六、关键风险与缓解措施

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| 作用域膨胀 | 项目无法按时交付 | 严格按优先级控制范围，每个阶段交付可用功能 |
| 补丁质量不可控 | 生成的补丁破坏现有功能 | 强化验证门禁 + 沙箱 + 人工复审 |
| 图谱不完整 | 影响分析不准确 | 增量扫描 + 定期全量重扫 |
| 安全风险 | 密钥泄露、越权修改 | 安全基线治理先行，PR 流水线加权限控制 |
| 团队规模不足 | 4 人团队覆盖多个模块 | 优先交付核心闭环，其他模块可简化 |

---

## 七、结论

本方案基于调研报告的八环节评估（需求理解 → 需求建模 → 影响分析 → 方案生成 → 代码生成 → 可落地性 → 自动化 → 可解释性），结合 LegacyGraph 当前代码现状，提出了分五个阶段的实施路径。

**核心策略**：不追求"更会聊天"，而是"把已经很强的需求/图谱/证据层，继续打通到 patch/PR/验证流水线"。

**最短路径**：Phase 0-1（安全 + 评测）+ Phase 2（PatchDraft + Sandboxed Validation）即可形成最小可用闭环，验证需求到编码的完整链路。
