## 一、N+1 查询问题

### 1.1 JavaMemberCallResolver 循环查询 Neo4j

**位置**: `builder/JavaMemberCallResolver.java:78-103`

**问题**:
```java
// 当前：3 次独立查询
for (NodeType t : new NodeType[]{NodeType.Controller, NodeType.Service, NodeType.Mapper}) {
    List<GraphNode> classNodes = neo4jGraphDao.queryNodes(projectId, versionId, t.name(), ...);
    // ...
}
// 再次单独查 Method
List<GraphNode> methods = neo4jGraphDao.queryNodes(projectId, versionId, NodeType.Method.name(), ...);
```

**影响**: 4 次 Neo4j 查询可合并为 1 次，每次查询涉及网络往返 + 结果序列化

**修复方案**:
```java
// 一次查询所有类型
List<String> nodeTypes = List.of("Controller", "Service", "Mapper", "Method");
List<GraphNode> allNodes = neo4jGraphDao.queryNodesByTypes(projectId, versionId, nodeTypes, 0);

// 按类型分组
Map<String, List<GraphNode>> typeToNodes = allNodes.stream()
    .collect(Collectors.groupingBy(GraphNode::getNodeType));

List<GraphNode> classNodes = Stream.of("Controller", "Service", "Mapper")
    .flatMap(t -> typeToNodes.getOrDefault(t, List.of()).stream())
    .toList();
List<GraphNode> methods = typeToNodes.getOrDefault("Method", List.of());
```

**新增 DAO 方法**:
```java

    String cypher = """
        MATCH (n:Node {projectId: $projectId, versionId: $versionId})
        WHERE n.nodeType IN [%s]
        RETURN n
        """.formatted(typesParam);
```

## 二、连接池配置优化

### 2.1 HikariCP 调优

**位置**: `application.yml:24-32`

**当前配置**:
```yaml
hikari:
  minimum-idle: 5
  maximum-pool-size: 20
  idle-timeout: 30000
  max-lifetime: 1800000
  connection-timeout: 30000
```

**优化配置**:
```yaml
hikari:
  minimum-idle: 10          # 提升：减少冷启动连接创建延迟
  maximum-pool-size: 30     # 提升：扫描时并发高，20 连接可能成为瓶颈
  idle-timeout: 30000       # 保持
  max-lifetime: 1800000     # 保持
  connection-timeout: 10000 # 降低：30s 太长，快速失败好过长时间挂起
```

**理由**:
- `maximum-pool-size: 30`: 扫描时多线程并发写入，20 连接容易排队等待
- `minimum-idle: 10`: 预热连接，避免首次请求延迟
- `connection-timeout: 10000`: 10s 快速失败，便于监控发现问题

### 2.3 JDBC 批量重写

**位置**: `application.yml` 数据库连接 URL

**问题**: MyBatis-Plus 的 `saveBatch()` 本质上仍是逐条 `INSERT`，只是在同一事务中执行，未真正利用 PG 的批量协议。

**修复方案**: 在 JDBC URL 上加 `reWriteBatchedInserts=true`
```yaml
# 当前
url: jdbc:postgresql://localhost:5432/legacygraph

# 优化
url: jdbc:postgresql://localhost:5432/legacygraph?reWriteBatchedInserts=true
```

**效果**: 将多条 `INSERT INTO ... VALUES (?)` 重写为 `INSERT INTO ... VALUES (?), (?), (?)` 的批量形式，批量写入性能可提升 3-10x。配合 §4.1 的 `saveBatch()` 调用生效。

### 3.2 Neo4j 索引优化

**位置**: Neo4j 数据库（Cypher 查询层）

**问题**: 当前 Neo4j 未创建针对图谱查询热点模式的索引。从 [05-图谱查询与写图链路优化.md](./05-图谱查询与写图链路优化.md) 可知高频查询模式：

```cypher
-- 查询热点 1：按项目+版本+类型查节点
MATCH (n:Node {projectId: $p, versionId: $v}) WHERE n.nodeType IN [...]
-- 查询热点 2：按项目+版本+nodeKey 查节点
MATCH (n:Node {projectId: $p, versionId: $v, nodeKey: $k})
-- 查询热点 3：按项目+版本+类型查边
MATCH ()-[e:Edge {projectId: $p, versionId: $v}]->() WHERE e.edgeType = $t
```

**修复方案**:
```cypher
-- 节点复合索引（projectId + versionId + nodeType）
CREATE INDEX node_project_version_type IF NOT EXISTS
FOR (n:Node) ON (n.projectId, n.versionId, n.nodeType);

-- 节点复合索引（projectId + versionId + nodeKey）— 用于精确查找
CREATE INDEX node_project_version_key IF NOT EXISTS
FOR (n:Node) ON (n.projectId, n.versionId, n.nodeKey);

-- 边复合索引（projectId + versionId + edgeType）
CREATE INDEX edge_project_version_type IF NOT EXISTS
FOR ()-[e:Edge]-() ON (e.projectId, e.versionId, e.edgeType);
```

**预期收益**: 大图查询从全表扫描变为索引查找，查询延迟可能降低 1-2 个数量级。配合 [05-图谱查询与写图链路优化.md](./05-图谱查询与写图链路优化.md) 的窗口查询，效果叠加。

> **执行方式**: Neo4j 索引通过 `neo4j` shell 或 Cypher shell 手动执行一次即可；也可作为 Flyway 回调在应用启动时执行（`@PostConstruct` + Neo4j `Session`）。

### 3.3 Neo4j 连接池调优

**位置**: `application.yml` Neo4j driver 配置（如 `spring.neo4j.pool` 或 driver 自定义配置）

**问题**: Neo4j Bolt driver 的连接池默认值偏保守（`max-connection-pool-size` 默认 100 但实际取决于配置）。

**建议配置**:
```yaml
spring:
  neo4j:
    pool:
      max-connection-pool-size: 50      # 写图并发时有足够连接
      connection-acquisition-timeout: 10000  # 10s 超时，快速失败
      max-connection-lifetime: 3600s    # 1 小时
```

## 五、只读事务优化

### 5.1 查询方法加 @Transactional(readOnly = true)

**位置**: 所有 `*Service` / `*Repository` 中纯查询方法

**问题**: Spring Data JPA / Hibernate 默认开启脏检查（dirty checking），纯查询不需要此开销。不加 `readOnly = true` 时 Hibernate 会维护一级缓存快照用于 flush 时对比。

**修复方案**:
```java
// 类级别或方法级别
@Service
@Transactional(readOnly = true)  // 类级别默认只读
public class GraphQueryService {
    
    @Transactional(readOnly = true)  // 显式标注（覆盖类级别也可）
    public UnifiedGraph getUnifiedGraph(String projectId, String versionId) {
        // 纯查询逻辑
    }
}

// 写方法单独标注
@Transactional  // 默认 readOnly = false
public void saveGraph(GraphData data) {
    // 写入逻辑
}
```

**注意事项**:
- `@Transactional(readOnly = true)` 只需要在**直接操作数据库的 Service/Repository** 上使用
- 调用链上层不需要传播（Spring 默认 `Propagation.REQUIRED` 会复用已有事务）
- 如果 Service 内部分方法是查询、部分是写入，**类级别标 `readOnly = true`**，写入方法单独标 `@Transactional`

**预期收益**: 减少 Hibernate 一级缓存快照维护开销，查询吞吐提升 5-15%。
