## 一、文件遍历并行化

### 1.1 AssetDiscoveryService 文件遍历串行

**位置**: `task/AssetDiscoveryService.java:235-241`

**问题**: `Files.walk()` 遍历目录树是串行操作，对于大型项目（数千文件）耗时长

**修复方案**:
```java
// 当前
try (var stream = Files.walk(root)) {
    paths = stream.filter(Files::isRegularFile)
                  .filter(p -> !isExcludedDir(p, root))
                  .filter(p -> hasSupportedExtension(p))
                  .sorted()
                  .toList();
}

// 优化：并行流（移除 sorted，并行流不支持全局排序）
try (var stream = Files.walk(root).parallel()) {
    paths = stream.filter(Files::isRegularFile)
                  .filter(p -> !isExcludedDir(p, root))
                  .filter(p -> hasSupportedExtension(p))
                  .toList();
}
```

**预期收益**: 大项目（5000+ 文件）遍历速度提升 2-3x

> **注意**: `Files.walk()` 的目录遍历本身是串行的，`.parallel()` 只并行化下游的 filter/map 操作（CPU 部分），I/O 部分仍在单一线程中完成。实际收益来自 filter 阶段的并行，对 I/O 密集的大项目收益可能小于 2x。如果目录遍历本身是瓶颈，考虑用 `ForkJoinPool` + 递归目录遍历替代。

### 1.2 资产构建串行循环

**位置**: `task/AssetDiscoveryService.java:247-254`

**问题**: `for (Path p : paths)` 逐个构建 `SourceAsset`，包括计算 SHA-256（CPU 密集）

**修复方案**:
```java
// 当前
List<SourceAsset> assets = new ArrayList<>();
for (Path p : paths) {
    String relativePath = root.relativize(p).toString();
    SourceAsset asset = buildAsset(p, relativePath);
    if (shouldIncludeAsset(asset, scanCode, scanDocs)) {
        assets.add(asset);
    }
}

// 优化：并行流
List<SourceAsset> assets = paths.parallelStream()
    .map(p -> {
        String relativePath = root.relativize(p).toString();
        SourceAsset asset = buildAsset(p, relativePath);
        return shouldIncludeAsset(asset, scanCode, scanDocs) ? asset : null;
    })
    .filter(Objects::nonNull)
    .toList();
```

**预期收益**: SHA-256 计算并行化，CPU 密集任务提升 4-8x（取决于核心数）

### 2.1 单文件重复解析

    }

## 三、增量扫描

### 3.1 基于内容哈希的变更检测

**位置**: `task/AssetDiscoveryService.java:208-254` / `task/AiScanOrchestrator.java`

**问题**: 当前每次重新扫描遍历全部文件并重新 LLM 提取。对于只有少量文件变更的迭代项目，大量重复工作。

**修复方案**:
```java
public List<Path> findChangedFiles(Path projectRoot, String versionId) {
    // 1. 遍历当前文件，计算 content_hash
    Map<String, String> currentHashes = walkFiles(projectRoot).stream()
        .collect(toMap(this::relativePath, this::sha256));
    
    // 2. 查询上次扫描的快照
    Map<String, String> previousHashes = snapshotRepository.findByVersionId(versionId)
        .stream()
        .collect(toMap(SourceAssetSnapshot::getRelativePath, SourceAssetSnapshot::getContentHash));
    
    // 3. 对比差异
    return currentHashes.entrySet().stream()
        .filter(e -> !e.getValue().equals(previousHashes.get(e.getKey())))
        .map(e -> projectRoot.resolve(e.getKey()))
        .collect(toList());
    
    // 4. 新增文件 + 删除文件
    // - 新增：currentHashes 有但 previousHashes 没有 → 全量提取
    // - 删除：previousHashes 有但 currentHashes 没有 → 标记删除
}
```

**LLM 提取侧配合**:
```java
// AiScanOrchestrator
List<Path> changedFiles = assetDiscoveryService.findChangedFiles(projectRoot, versionId);
List<Path> newFiles = changedFiles.stream()
    .filter(f -> !previousHashes.containsKey(relativePath(f)))
    .toList();
List<Path> modifiedFiles = changedFiles.stream()
    .filter(f -> previousHashes.containsKey(relativePath(f)))
    .toList();

// 新增文件 → 全量 LLM 提取
// 修改文件 → 仅对修改部分做增量 LLM 提取（如果 LLM 能力足够做 diff 感知）
// 未变文件 → 复用上次提取结果
```

**预期收益**: 日常迭代（通常 < 5% 文件变更），重扫时间从完整耗时降到 5-10%。

### 3.2 扫描断点续传

**位置**: `task/AiScanOrchestrator.java`

**问题**: 大项目扫描可能持续数小时，如果 JVM crash / OOM / 手动中断，已完成的 LLM 提取全部丢失。

**修复方案**: 在 PG 中记录每个文件的提取状态：
```sql
-- 新增表
CREATE TABLE lg_extract_checkpoint (
    id          BIGSERIAL PRIMARY KEY,
    version_id  VARCHAR(64) NOT NULL,
    file_path   VARCHAR(1024) NOT NULL,
    status      VARCHAR(20) NOT NULL DEFAULT 'PENDING', -- PENDING/EXTRACTING/DONE/FAILED
    extracted_at TIMESTAMP,
    error_msg   TEXT,
    UNIQUE(version_id, file_path)
);
```

```java
// AiScanOrchestrator
public void extractWithCheckpoint(String versionId, List<Path> files) {
    List<Path> pending = checkpointRepository.findPendingFiles(versionId, files);
    
    for (Path file : pending) {
        checkpointRepository.markExtracting(versionId, file);
        try {
            ExtractResult result = doExtract(file);
            saveResult(result);
            checkpointRepository.markDone(versionId, file);
        } catch (Exception e) {
            checkpointRepository.markFailed(versionId, file, e.getMessage());
            // 失败文件可选重试，不影响其他文件
        }
    }
}
```

**预期收益**:
- 大项目中断后可续传，已完成文件不重复提取
- 失败文件单独追踪，便于排查和重试
- 扫描进度可精确到文件级别

## 四、线程池优化

### 4.1 虚拟线程执行器频繁创建销毁

**位置**: 
- `task/AdapterExecutionService.java:165`
- `task/ProjectScanner.java:1637`

**问题**: 每次调用 `executeScan()` 都创建新的 `Executors.newVirtualThreadPerTaskExecutor()`

**修复方案**: 复用全局虚拟线程执行器
```java
@Component
public class AdapterExecutionService {
    private final ExecutorService virtualThreadExecutor = Executors.newVirtualThreadPerTaskExecutor();
    
    @PreDestroy
    public void shutdown() {
        virtualThreadExecutor.shutdown();
        try {
            if (!virtualThreadExecutor.awaitTermination(30, TimeUnit.SECONDS)) {
                virtualThreadExecutor.shutdownNow();
            }
        } catch (InterruptedException e) {
            virtualThreadExecutor.shutdownNow();
        }
    }
}
```

### 4.2 AiScanOrchestrator 固定线程池过小

**位置**: `task/AiScanOrchestrator.java:149-151`

**问题**: `docExtractExecutor` 和 `codeExtractExecutor` 都是 `newFixedThreadPool(4)`，LLM 调用瓶颈时无法扩展

**修复方案**: 使用虚拟线程 + 可配置 Semaphore 控制 LLM 并发
```yaml
# application.yml
legacygraph:
  scan:
    llm-concurrency: 8          # LLM API 最大并发调用数
    code-extract-concurrency: 4  # 代码提取最大并发数
```

```java
@Value("${legacygraph.scan.llm-concurrency:8}")
private int llmConcurrencyLimit;

@Value("${legacygraph.scan.code-extract-concurrency:4}")
private int codeExtractConcurrency;

private final ExecutorService docExtractExecutor = Executors.newVirtualThreadPerTaskExecutor();
private Semaphore llmConcurrency;  // 在 @PostConstruct 中初始化

@PostConstruct
void init() {
    this.llmConcurrency = new Semaphore(llmConcurrencyLimit);
}

// 使用时
public CompletableFuture<ExtractResult> extractDocument(Document doc) {
    return CompletableFuture.supplyAsync(() -> {
        try {
            llmConcurrency.acquire();
            try {
                return doExtract(doc);
            } finally {
                llmConcurrency.release();
            }
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
            throw new RuntimeException(e);
        }
    }, docExtractExecutor);
}
```

> **注意**: 虚拟线程创建成本极低，不需要复用虚拟线程本身。真正需要限流的是 LLM API（受 API 限流和 Token 预算约束），因此用 Semaphore 控制即可。虚拟线程执行器本身可以安全复用（或按需创建）。

## 五、异步编排优化

### 5.1 嵌套 CompletableFuture 缺少异常处理

**位置**: `task/AiScanOrchestrator.java:400-454`

**问题**: 多个 `CompletableFuture.runAsync()` 嵌套在循环中，单个失败不影响其他，但 `allOf().join()` 会抛出第一个异常

**修复方案**: 添加 `exceptionally()` 处理
```java
futures.add(CompletableFuture.runAsync(() -> {
    // 任务逻辑
}, docExtractExecutor).exceptionally(ex -> {
    log.warn("Document extraction failed: {}", ex.getMessage());
    return null;
}));
```

### 5.2 EvidenceGraphWriter 分段锁粒度过粗

**位置**: `builder/EvidenceGraphWriter.java:65-136`

**问题**: 使用 `ReentrantLock[16]` 分段锁，但锁粒度基于 `nodeType::nodeKey`，可能导致热点节点锁竞争

**修复方案**: 增加锁分段数
```java
private static final int LOCK_STRIPES = 64;  // 增加分段数
private final ReentrantLock[] locks = new ReentrantLock[LOCK_STRIPES];

{
    for (int i = 0; i < LOCK_STRIPES; i++) {
        locks[i] = new ReentrantLock();
    }
}

private ReentrantLock getLock(String key) {
    int hash = Math.abs(key.hashCode());
    return locks[hash % LOCK_STRIPES];
}
```

## 六、进度反馈优化

### 6.1 进度日志间隔过大

**位置**: `task/AdapterExecutionService.java:319`

**问题**: `getProgressLogInterval()` 固定返回 10，对于 10000+ 文件的项目，进度更新频率太低

**修复方案**: 动态计算间隔
```java
private int getProgressLogInterval(int total) {
    if (total < 100) return 10;
    if (total < 1000) return 50;
    if (total < 10000) return 100;
    return 500;  // 大项目每 500 个文件报告一次
}
```
