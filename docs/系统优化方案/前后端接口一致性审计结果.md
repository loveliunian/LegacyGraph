## 2. 审计方法

### 后端端点提取
- 扫描 `backend/src/main/java/io/github/legacygraph/controller/` 下全部 50 个 `.java` 文件。
- 提取类级 `@RequestMapping` 与方法级 `@GetMapping/@PostMapping/@PutMapping/@DeleteMapping/@PatchMapping`，拼接完整路径。
- 路径变量归一化：`{projectId}` / `{id}` 等统一为 `{var}` 以便前后端对齐。

### 前端调用点提取
- 扫描 `frontend/src/` 下所有 `.ts` 和 `.vue` 文件（排除 `__tests__`、`node_modules`）。
- 覆盖以下调用模式：
  - `request.get/post/put/delete` + 泛型 `get<T>(url)` / `post<T>(url)`（含嵌套泛型 `get<PageResult<User>>`）
  - `get(url)` / `post(url)` / `del(url)` / `put(url)` 裸函数调用
  - `upload<T>(url, file)` 上传调用
  - `downloadFile(url)` / `download(url)` 下载调用
  - `fetch(url, { method })` 原生 fetch（含 `method` 上下文检测）
  - 模板字符串 URL `` `${apiBaseUrl}/lg/...` ``
  - 字符串拼接 URL `+ '/lg/...'`
  - 变量赋值 URL `const url = '/lg/...'`
- 路径变量归一化：`${projectId}` → `{var}`，`{encodeURIComponent(id)}` → `{var}`，剥离 `${apiBaseUrl}` 前缀。
- API 前缀白名单过滤：`/lg/`、`/qa/`、`/reports/`、`/agents/`、`/llm/`、`/change-tasks/`。

### 差集计算
- A 类 = 前端调用集 − 后端端点集（前端有 / 后端缺）
- B 类 = 后端端点集 − 前端调用集（后端有 / 前端缺）
- 以 `HTTP方法 + 归一化路径` 为去重键。

## 7. 结论与建议

### 总体评估

- **A 类归零**：前端所有 API 调用均有后端端点支撑，无 404 风险。
- **B 类从 29 增至 55**：上次报告的 29 项中 11 项已解决（知识工作台接入、Graphify 作业入口补齐、重复端点下线），但后端新增 65 个端点中仅 28 个被前端调用，37 个尚无前端入口。
- **新增 Controller 已被识别**：ProcessMiningController（2 端点）、QaAdminController（1 端点）、CrossRepoImpactController（1 端点未调用 `/diff`）均已纳入审计。

### 按优先级建议

| 优先级 | 分类 | 数量 | 建议 |
|---|---|---|---|
| 🔴 高 | 变更任务操作（assign/claim/merge） | 3 | 变更任务管理是核心流程，建议在任务详情页补分配/认领/合并按钮 |
| 🔴 高 | 需求验收标准（verify/fail/waive） | 3 | 验收闭环是需求管理关键环节，建议在需求详情页补验收操作 |
| 🟡 中 | 方案审批流程（approve/reject/bridge） | 4 | 方案审批涉及工作流，建议在方案详情页补审批入口 |
| 🟡 中 | 流程挖掘（ProcessMining） | 2 | 新增模块，建议新建 `process-mining.api.ts` + 流程挖掘视图 |
| 🟡 中 | QA 审计日志 | 1 | 新增模块，建议在 QA 管理页补审计日志 Tab |
