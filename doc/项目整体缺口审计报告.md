# LegacyGraph 项目整体缺口审计报告

> 检查日期：2026-06-27  
> 检查范围：后端、前端、测试、数据库脚本、部署配置、项目文档  
> 检查口径：以当前代码、配置和可执行验证结果为准

## 结论

当前项目不能按“功能完整、可上线运行”判断。代码中已有较完整的前后端目录、实体、Controller、页面、测试文件和文档框架，但存在多处阻断级缺口：后端 Maven 测试在读 POM 阶段失败，前端构建因依赖缺失无法启动，认证链路无法产生可被后端校验的 JWT，前后端 API 路径大量不一致，数据库初始化脚本缺表且存在 PostgreSQL 语法问题，部署配置也不能闭环。

已有 `doc/项目完整性检查报告.md` 和 `doc/未完成功能清单.md` 将项目描述为 100% 完成，但这个结论与当前代码不一致。后续状态文档应改为“已有框架和部分实现，尚未完成端到端可运行闭环”。

## 阻断级缺口

### 1. 后端当前无法构建和运行测试

执行 `mvn test` 直接失败：

```text
'dependencies.dependency.version' for org.springframework.boot:spring-boot-starter-aop:jar is missing.
```

相关位置：

- `backend/pom.xml:14-18` 使用 Spring Boot parent `4.0.6`。
- `backend/pom.xml:54-57` 引入 `spring-boot-starter-aop`，但 Maven 未解析到版本。

影响：所有后端测试文件都不能作为“已验证通过”的依据，Docker 后端构建中的 `mvn clean package -DskipTests` 也有同类风险。

### 2. 前端当前无法构建，且入口路由有硬错误

执行 `npm run build` 失败：

```text
vue-tsc && vite build
sh: vue-tsc: command not found
```

原因和证据：

- 仓库没有 `node_modules`，也没有 `package-lock.json`、`pnpm-lock.yaml`、`yarn.lock`。
- `frontend/package.json:7` 的 build 脚本依赖 `vue-tsc`。
- `frontend/src/router/index.ts:9` 引用 `@/views/login/Login.vue`，实际目录只有 `LoginPage.vue`。
- `frontend/src/router/index.ts:211-217` 使用 `userStore.token`、`userStore.user`、`userStore.fetchUser()`，但 `frontend/src/stores/user.ts` 暴露的是 `accessToken`、`userInfo`、`fetchCurrentUser()`。

影响：前端首屏和鉴权守卫在当前代码下不能可靠运行。

### 3. 登录认证链路不可用

后端安全配置要求除登录/刷新等少数接口外全部认证：

- `backend/src/main/java/io/github/legacygraph/config/SecurityConfig.java:34-45`

但登录接口返回的是随机 UUID 字符串，而不是 `JwtUtil.generateToken(...)` 生成的 JWT：

- `backend/src/main/java/io/github/legacygraph/controller/AuthController.java:90-91`

同时数据库初始化脚本创建的是 `sys_user`：

- `docs/sql/init.sql:360-386`

认证 Controller 使用的实体却绑定到 `lg_user`：

- `backend/src/main/java/io/github/legacygraph/entity/User.java:10-12`

`docs/sql/init.sql` 中没有 `lg_user` 表。影响：默认数据库初始化后登录查询表不存在；即使表存在，登录返回 token 也无法通过 JWT 过滤器校验，后续受保护接口会失败。

### 4. 前后端 API 路径大量不一致

典型不一致项：

- 当前用户接口：前端 `frontend/src/api/auth.api.ts:32-33` 调 `/lg/auth/current`，后端是 `AuthController` 的 `/lg/auth/me`。
- 数据源接口：前端 `frontend/src/api/source.api.ts:39-87` 使用 `/code-repos`、`/db-connections`、`/documents`，后端 `SourceController` 的基路径是 `/lg/projects/{projectId}/sources`，实际路径是 `/sources/repos`、`/sources/databases`、`/sources/documents` 等。
- 系统管理：前端 `frontend/src/api/system.api.ts:40`、`:61`、`:82` 使用 `/lg/system/users`、`/dictionaries`、`/configs`，后端 `SystemController` 使用 `/users/list`、`/dicts/list`、`/configs/list`。
- 报告下载：前端 `frontend/src/api/report.api.ts:36-39` 调 `/lg/reports/{id}/download`，后端是 `/lg/projects/{projectId}/reports/{reportId}/download`。
- 测试运行：前端 `frontend/src/api/test-run.api.ts:45-63` 声明了测试运行列表、详情、结果、日志接口，后端 `TestCaseController` 只提供 start、rerun、cancel、stats，没有这些 GET 端点。

影响：多数页面即使渲染出来，也无法与后端真实接口连通。

### 5. 数据库初始化脚本不完整且有执行风险

`docs/sql/init.sql:7-8` 包含：

```sql
CREATE DATABASE IF NOT EXISTS legacy_graph;
\c legacy_graph;
```

PostgreSQL 不支持 `CREATE DATABASE IF NOT EXISTS`。同时 docker-compose 已通过 `POSTGRES_DB=legacy_graph` 创建并连接初始化数据库，脚本再建库/切库不适合作为容器 init 脚本。

此外，当前 SQL 没有创建下列代码实际依赖的表：

- `lg_user`：`backend/src/main/java/io/github/legacygraph/entity/User.java:10-12`
- `lg_code_repo`：`backend/src/main/java/io/github/legacygraph/entity/CodeRepo.java:10-12`
- `lg_db_connection`：`backend/src/main/java/io/github/legacygraph/entity/DbConnection.java:10-12`
- `lg_document`：`backend/src/main/java/io/github/legacygraph/entity/Document.java:10-12`

影响：认证和资料接入相关功能在真实数据库上会直接失败。

## 后端功能缺口

### 1. 全量扫描只覆盖 Java Controller 和 MyBatis XML

`ProjectScanner.startFullScan` 中前端扫描和数据库元数据扫描仍是 TODO：

- `backend/src/main/java/io/github/legacygraph/task/ProjectScanner.java:77-96`

虽然类内有 `scanDatabaseMetadata(...)` 方法：

- `backend/src/main/java/io/github/legacygraph/task/ProjectScanner.java:194-205`

但全量扫描没有调用它，也没有接入已配置的数据源。影响：前端图谱、数据库图谱和完整业务链路无法通过一次扫描生成。

### 2. 事实抽取接口是空成功返回

- `backend/src/main/java/io/github/legacygraph/controller/FactController.java:148-155`
- `backend/src/main/java/io/github/legacygraph/controller/FactController.java:158-165`

代码和文档事实抽取接口没有调用 `CodeFactAgent`、`DocUnderstandingAgent`，也没有异步任务或落库。影响：API 调用显示成功，但没有实际事实产生。

### 3. 图查询存在占位视图

- `backend/src/main/java/io/github/legacygraph/service/GraphQueryService.java:85-96` 功能视图只是返回 `module/versionId`。
- `backend/src/main/java/io/github/legacygraph/service/GraphQueryService.java:102-106` 业务视图只是返回 `domain/versionId`。

影响：文档中描述的“功能图谱/业务图谱完整链路”尚未实现。

### 4. 资料接入和解析存在假成功

- 仓库连接测试固定返回成功：`backend/src/main/java/io/github/legacygraph/controller/SourceController.java:237-256`
- 仓库拉取只是状态置为 READY：`backend/src/main/java/io/github/legacygraph/controller/SourceController.java:266-292`
- 数据库连接测试固定返回成功：`backend/src/main/java/io/github/legacygraph/controller/SourceController.java:465-484`
- 数据库扫描用随机表数量：`backend/src/main/java/io/github/legacygraph/controller/SourceController.java:500-522`
- 文档解析用随机 factCount：`backend/src/main/java/io/github/legacygraph/controller/SourceController.java:683-711`

影响：前端会看到成功状态，但没有真实 git clone、数据库连通性验证、schema 抽取、文档解析。

### 5. 审核与测试闭环仍有模拟数据

- 审核待办为空时造 mock 数据：`backend/src/main/java/io/github/legacygraph/controller/ReviewController.java:63-79`
- 审核历史为空时造 mock 数据：`backend/src/main/java/io/github/legacygraph/controller/ReviewController.java:116-136`
- 测试用例列表为空时造 mock 数据：`backend/src/main/java/io/github/legacygraph/controller/TestCaseController.java:61-88`
- 测试用例生成使用随机数量和空壳数据：`backend/src/main/java/io/github/legacygraph/controller/TestCaseController.java:144-169`
- 单用例执行随机通过/失败：`backend/src/main/java/io/github/legacygraph/controller/TestCaseController.java:172-193`
- 测试结果回调未保存结果、未更新图谱置信度：`backend/src/main/java/io/github/legacygraph/controller/TestCaseController.java:200-209`

影响：测试验证闭环不能支撑迁移置信度。

### 6. 报告能力未闭环

- 测试覆盖率报告把 coveredNodes/coveredEdges 固定为 0：`backend/src/main/java/io/github/legacygraph/service/ReportingService.java:305-323`
- 导出 PDF/Excel 时仍返回 JSON 字节：`backend/src/main/java/io/github/legacygraph/service/ReportingService.java:457-486`
- `saveReport` 只保存记录，没有上传 MinIO：`backend/src/main/java/io/github/legacygraph/service/ReportingService.java:489-504`

影响：报告列表、下载、覆盖率结论都不能作为正式交付依据。

## 前端功能缺口

### 1. 页面与后端真实数据连接不足

静态搜索显示多个页面仍有 TODO、`setTimeout`、随机数或静态图数据，典型位置：

- 迁移风险详情确认/忽略未调用接口：`frontend/src/views/migration/RiskDetail.vue:135`、`:141`
- 审核列表仍标记 API 分页 TODO：`frontend/src/views/review/ReviewList.vue:118`
- 图分析面板多处用 `setTimeout` 模拟结果：`frontend/src/components/graph/GraphAnalysisPanel.vue`
- 审计日志列表生成随机日志：`frontend/src/views/audit/LogList.vue`
- 数据源连接测试、扫描、文档解析页面使用 `setTimeout`/随机数：`frontend/src/views/source/*`
- `BusinessGraph.vue`、`FeatureGraph.vue` 内存在硬编码示例节点和边。

影响：很多页面只是“可展示”，不是“已接入业务后端”。

### 2. PWA 配置缺资源

`frontend/vite.config.ts` 配置了 `favicon.ico`、`robots.txt`、`pwa-192x192.png`、`pwa-512x512.png`，但 `frontend/public/` 当前为空。影响：PWA manifest 和静态资源构建存在缺口。

### 3. 代码规范脚本缺配置

`frontend/package.json:9` 有 `npm run lint`，但前端根目录没有 `.eslintrc*` 或 `eslint.config.*`。影响：lint 脚本的规则来源不明确，无法作为统一质量门禁。

## 测试缺口

### 当前测试资产

- 后端测试文件存在，约 187 个 `@Test` 声明。
- 前端 unit/e2e 测试文件存在，约 97 个测试声明。

### 当前不可作为通过依据

- 后端 `mvn test` 在 POM 解析阶段失败，未进入测试执行。
- 前端 `npm run build` 因未安装依赖失败；同理 Vitest/Playwright 也无法在当前工作区直接运行。
- 后端没有 `backend/src/test/resources`，测试环境数据库、profile、H2/Testcontainers 等隔离配置缺失。
- 测试没有覆盖最关键的端到端问题：登录后 token 校验、前后端 API 路径一致性、SQL 初始化、Docker 部署、真实扫描/解析/测试执行。

## 部署与运维缺口

- 前端 Dockerfile 使用 `npm ci`：`frontend/Dockerfile:7-8`，但仓库没有 lockfile，镜像构建会失败。
- Nginx 配置只有静态文件和 history fallback：`frontend/nginx.conf:1-23`，没有 `/api` 代理；生产前端默认 `baseURL=/api` 将无法访问后端。
- 后端 Dockerfile 依赖 Maven 构建，当前 Maven POM 失败会阻断镜像。
- docker-compose 将 SQL 目录挂载为初始化脚本：`deploy/docker-compose.yml:17-19`，但 `init.sql` 自身有建库语法风险。
- docker-compose 设置 `SPRING_NEO4J_*` 环境变量：`deploy/docker-compose.yml:84-86`，而自定义 `Neo4jConfig` 读取的是 `neo4j.uri/username/password`：`backend/src/main/java/io/github/legacygraph/config/Neo4jConfig.java:13-20`。`application-docker.yml:11-15` 写的是 `spring.neo4j`，不匹配自定义配置。

## 文档缺口

- `doc/项目完整性检查报告.md` 写“功能完整，可上线运行”和整体完成度 100%，但当前构建、认证、数据库、前后端联调都不成立。
- `doc/未完成功能清单.md` 一方面写“项目整体完成度 100%”，另一方面末尾又列出性能、功能、测试、文档待优化项，状态口径冲突。
- README 声称项目采用 Apache 2.0 并引用 `LICENSE`，但根目录没有 `LICENSE` 文件。
- README 和文档中的 API 示例、资料接入路径、系统管理路径没有和当前后端 Controller 对齐。
- 数据库设计文档没有反映 `lg_user`、`lg_code_repo`、`lg_db_connection`、`lg_document` 等实际实体依赖表缺失问题。

## 建议修复优先级

### P0：先恢复可构建、可登录、可初始化

1. 修复 `backend/pom.xml` 依赖管理，让 `mvn test` 至少能进入编译/测试阶段。
2. 为前端补 lockfile，统一包管理器，修复 `Login.vue` 路由和 user store 字段。
3. 统一认证模型：改登录为真实 JWT，或同步安全过滤逻辑；统一 `lg_user` 与 `sys_user`。
4. 修复 `docs/sql/init.sql`，补齐实体依赖表，并用一次干净数据库初始化验证。
5. 对齐前端 API 与后端 Controller 路径。

### P1：补齐核心业务闭环

1. 全量扫描接入前端扫描和数据库元数据扫描。
2. 事实抽取接口接入 Agent、任务状态和结果落库。
3. 资料接入实现真实 git clone/pull、数据库连接测试、schema 抽取、文档解析。
4. 测试用例生成/执行/回调改为真实调度和结果持久化。
5. 图谱视图返回真实节点边和链路，而不是占位字段。

### P2：补质量门禁与部署闭环

1. 加后端 test profile 或 Testcontainers，明确测试数据库和初始化脚本。
2. 前端补 ESLint 配置、PWA 资源，跑通 `npm run build`、`npm run test`、`npm run test:e2e`。
3. 修复前端 Nginx `/api` 代理和 Dockerfile `npm ci` 前置条件。
4. 把旧文档中的 100% 完成结论改为当前真实状态，并维护一份“已验证/未验证/未完成”清单。
