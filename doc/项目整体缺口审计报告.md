## 一、总体结论

经过本轮“先保证能跑起来”修复后，项目已从“前后端都不能构建”推进到“基础构建/打包可通过，Docker Compose 配置可解析”的状态。但仍不能按“功能完整、可上线运行”判断。仓库已经具备较完整的前后端目录、Controller、页面、实体、测试文件和文档框架，但仍存在端到端闭环缺口：

- 后端 `mvn -DskipTests compile`、`mvn -DskipTests package` 已通过，可生成 jar。
- 前端 `npm run build` 已通过，可生成 `dist` 产物；但 `vue-tsc` 类型检查已拆出为 `npm run type-check`，还没有作为通过项确认。
- `deploy/docker-compose.yml` 已能通过 `docker compose config` 解析，前端 Nginx `/api` 代理已对齐后端 context-path。
- 数据库初始化脚本与当前实体字段不一致，多个核心接口即使启动也会在真实数据库上报错。
- 前后端部分 API 路径已修复（test-run、数据库连接测试、文档上传），但数据结构仍有未对齐项。
- 报告、测试、审核、扫描等核心闭环里仍有递归风险、mock 数据、随机结果或 TODO。
- 既有文档仍保留“100% 完成 / 可上线运行”的旧结论，与当前验证结果冲突。

因此，项目当前状态更准确地描述为：**基础运行门槛已恢复，但尚未形成可初始化、可联调、可测试、可验证的端到端业务闭环。**

## 四、P0 阻断级缺口

### P0-1：后端曾无法进入编译和测试（本轮已修复到可编译/可打包）

修复前执行 `rtk mvn test` 失败于 POM 解析阶段：

```text
'dependencies.dependency.version' for org.springframework.boot:spring-boot-webmvc-test:jar is missing.
```

证据：

- `backend/pom.xml:192` 引入 `spring-boot-starter-test`
- `backend/pom.xml:197` 引入 `spring-boot-webmvc-test`
- Maven 对 `spring-boot-webmvc-test` 没有从 parent 中解析到版本

影响：

- 当前 `rtk mvn -DskipTests compile` 已通过。
- 当前 `rtk mvn -DskipTests package` 已通过。
- 完整 `mvn test` 仍未作为通过项确认，测试数据库/profile 仍需单独治理。

### P0-2：前端曾无法构建（本轮已修复到可产物构建）

修复前执行 `rtk npm run build` 失败，既有 TypeScript 编译问题，也有真实 Vite/SFC 构建问题。

典型错误：

- `frontend/src/api/report.api.ts:2` 引用不存在的 `@/types/report`
- `frontend/src/api/index.ts` 重复导出名为 `User` 的成员
- `GraphViewer.vue` 使用的 `@vue-flow/core` 导出与当前版本类型不匹配
- 多处 `ElMessage`、`computed`、`watch`、`ref` 未导入
- 多处前端类型与 `src/types/index.ts` 不一致

影响：

- 当前 `rtk npm run build` 已通过。
- 类型检查已拆出为 `npm run type-check`，历史 TypeScript 严格错误尚未按“已通过”确认。
- 前端测试仍未恢复，不能以测试通过作为质量判断。

### P0-3：数据库脚本与实体字段严重不一致

`docs/sql/init.sql` 中虽然补了多张表，但字段仍和当前实体不一致。

典型差异：

| 表 | SQL 现状 | 实体现状 | 影响 |
| --- | --- | --- | --- |
| `lg_code_repo` | `repo_url` | `CodeRepo.java:19` `repoName`、`:23` `gitUrl` 等 | 创建仓库会写不存在字段 |
| `lg_db_connection` | 缺 `schema_name`、`password`、`readonly`、`include_tables`、`exclude_tables`、`table_count` 等 | `DbConnection.java:29`、`:37` 等字段存在 | 数据库连接创建/扫描会失败 |
| `lg_document` | `file_url`、`status` | `Document.java:23` `fileType`、`:29` `parseStatus` 等 | 上传和解析文档会失败 |
| `lg_review_record` | `review_status`、`reviewer`、`review_comment` | `ReviewRecord.java:23` `targetName`、`:39` `reviewedBy` 等 | 审核列表/确认会失败 |
| `lg_test_case` | 缺 `deleted` | `TestCase.java:47` 有 `@TableLogic deleted` | MyBatis-Plus 逻辑删除查询可能引用不存在列 |

影响：

- 即使 Docker 数据库初始化成功，后端真实 CRUD 也会因列名不存在或类型不匹配失败。
- 数据库设计文档仍按旧字段描述，无法作为实现依据。

### P0-4：报告生成曾存在递归调用风险（本轮已修复递归点）

`ReportingService` 的生成方法会调用 `saveReport(...)`：

- `ReportingService.java:196`
- `ReportingService.java:279`
- `ReportingService.java:363`
- `ReportingService.java:474`

但 `saveReport(...)` 内又再次调用生成方法：

- `ReportingService.java:541` 调 `generateMigrationReport(projectId)`

影响：

- 当前 `saveReport(...)` 不再回调生成方法，避免报告生成时递归。
- `exportReport(...)` 的 PDF/Excel 仍是 TODO：`ReportingService.java:510`。

### P0-5：前后端测试运行 API 曾未对齐（本轮已修复前端路径）

前端 `frontend/src/api/test-run.api.ts` 当前调用：

- `/lg/test-runs/${runId}`
- `/lg/test-runs/start`

影响：

## 六、P2 质量、测试与部署缺口

### P2-1：前端测试配置误收 Playwright e2e 文件

`frontend/vitest.config.ts:10` 配置为：

```ts
exclude: [...configDefaults.exclude, 'e2e/*']
```

但当前 e2e 文件在 `tests/e2e/*`，因此 `npm run test -- --run` 会把 Playwright 的 `test.describe(...)` 当成 Vitest 文件加载，导致 11 个 e2e suite 直接失败。

影响：

- Vitest 单测和 Playwright e2e 没有隔离。
- 测试统计结果失真。

### P2-2：前端单测环境不完整

`frontend/tests/setup.ts` 只 stub 了 `teleport`，没有全局注册或 stub Element Plus 组件/指令。

实际失败表现：

- `el-tag`、`el-card`、`el-form-item`、`v-loading` 等无法解析。
- API 单测期望 `VITE_API_BASE_URL`，但当前未定义。
- 部分视图测试没有 router 注入，直接触发 Vue Router 警告和断言失败。

### P2-3：文档状态与当前代码冲突

当前文档仍有明显过度结论：

- `doc/项目完整性检查报告.md:3` 写“功能完整，可上线运行”
