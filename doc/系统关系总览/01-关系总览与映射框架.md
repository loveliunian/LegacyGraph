# 01 — 业务/功能/代码/数据 关系总览与映射框架

> 文档定位：**结论性框架**。定义 LegacyGraph 系统中"业务、功能、代码、数据"四层的内涵、层间映射规则、追溯路径与证据约束。
> 本文档给出**可复用的通用框架**；LegacyGraph 自身的实例映射见 `02-分层映射详解.md`；可直接问答的事实见 `03-QA事实底座.md`。
> 口径：以当前代码实现为准。节点类型以 `common/NodeType.java`（38 种）为锚，关系类型以 `common/EdgeType.java`（26 种）为锚，证据卡片以 `dto/rag/GraphRagEvidenceCard.java` 为锚。

---

## 1. 为什么是"四层"而不是"三类图谱"

LegacyGraph 对外呈现**三类图谱**（业务图谱 / 功能图谱 / 代码图谱），但它们本质是同一张统一图谱的**投影视图**。从"关系"视角看，真正稳定的骨架是**四层**：

| 层 | 回答的问题 | 投影对应的图谱 |
|---|---|---|
| 业务层 | 系统**为什么**存在、为谁、以什么流程和规则工作 | 业务图谱 |
| 功能层 | 用户/系统**如何触发**、看到什么入口与结果 | 功能图谱 |
| 代码层 | 具体由**哪些技术单元**实现、如何调用 | 代码图谱 |
| 数据层 | 落到**哪些表/字段**、谁读写、什么约束 | 贯穿三类图谱 |

> 数据层不单独投影成一张图谱，而是**贯穿**三类图谱：业务对象映射到表（`MAPS_TO`）、功能验证落到表断言、代码 SQL 读写表。这也是为什么"数据"必须从三类图谱里抽出来作为独立一层来描述关系。

三类图谱是"读模型"，四层是"关系模型"。本文档与 `02` 用四层组织关系，是为了让任何一层的事实都能追溯到其他三层——这正是 QA 问答和影响分析所依赖的连通性。

---

## 2. 四层定义

### 2.1 业务层（Business）

系统为谁解决什么问题、以什么流程和规则工作。主要来自文档 AI 抽取 + 代码 AI 抽取 + 数据库语义增强，默认 `PENDING_CONFIRM`。

| 节点类型（NodeType） | 含义 |
|---|---|
| `BusinessDomain` | 业务域 |
| `BusinessProcess` | 业务流程 |
| `BusinessObject` | 业务对象 |
| `BusinessRule` | 业务规则 |
| `Role` | 业务角色 |

典型关系：`BusinessDomain CONTAINS Feature`、`BusinessObject MAPS_TO Table`、`Feature HAS_RULE BusinessRule`、`Role REQUIRES_PERMISSION Permission`（间接）。

### 2.2 功能层（Feature）

用户/系统从哪个入口触发、经过哪些页面与接口、受什么权限约束、由什么测试验证。来自前端 Vue 组件 + axios 调用 + Controller 注解 + AI 功能映射。

| 节点类型 | 含义 |
|---|---|
| `FeatureModule` | 功能模块 |
| `Feature` | 功能点 |
| `Menu` | 菜单 |
| `Page` | 页面 |
| `Button` | 按钮 |
| `Permission` | 权限 |
| `ApiEndpoint` | 接口 |

典型关系：`Feature EXPOSED_BY Page/ApiEndpoint`、`Page/Button CALLS ApiEndpoint`、`Feature REQUIRES_PERMISSION Permission`、`Feature VERIFIED_BY TestCase`。

### 2.3 代码层（Code）

具体由哪些技术单元实现、如何调用与执行 SQL。来自 Java AST + MyBatis XML + JDBC 元数据，确定性来源，易成为 `CONFIRMED`。

| 节点类型 | 含义 |
|---|---|
| `Controller` | 控制器 |
| `Service` | 服务类 |
| `Method` | 方法 |
| `Mapper` | Mapper |
| `SqlStatement` | SQL 语句 |
| `ScheduledJob` | 定时任务 |
| `MQConsumer` / `MQTopic` | 消息消费者 / 主题 |
| `ExternalSystem` | 外部系统 |

典型关系：`ApiEndpoint HANDLED_BY Controller/Method`、`Controller/Service CALLS Method/Service`、`Mapper/Method EXECUTES SqlStatement`、`Method CALLS_EXTERNAL ExternalSystem`、`ScheduledJob TRIGGERS Method`、`MQConsumer CONSUMES MQTopic`。

### 2.4 数据层（Data）

落到哪些表、字段、索引、约束。来自 JDBC 元数据 + SQL 解析。

| 节点类型 | 含义 |
|---|---|
| `Table` | 数据库表 |
| `Column` | 字段 |
| `Index` | 索引 |
| `ConfigItem` | 配置项 |

典型关系：`Table HAS_COLUMN Column`、`Table HAS_INDEX Index`、`Column UNIQUE_ON`（唯一约束）、`Table REFERENCES Table`（外键）、`SqlStatement READS/WRITES Table`、`SqlStatement JOINS Table`。

### 2.5 横切层：测试与变更闭环

不属于四层但与四层都相关，作为验证与治理的横切关注点。

| 节点类型 | 含义 | 典型关系 |
|---|---|---|
| `TestCase` / `Assertion` / `Evidence` | 测试用例 / 断言 / 证据 | `Feature/ApiEndpoint VERIFIED_BY TestCase`、`TestCase ASSERTS Assertion`、节点/边 `HAS_EVIDENCE Evidence` |
| `ChangeTask` / `Patch` / `PullRequest` / `Dependency` / `VersionRisk` | 变更任务 / 补丁 / PR / 依赖 / 版本风险 | `ChangeTask AFFECTS 节点`、`ChangeTask FIXED_BY Patch`、`Patch MIGRATES_TO`、`Dependency DEPENDS_ON` |

## 3. 核心映射链路

四层之间最核心的一条贯穿链路：

```text
BusinessDomain / BusinessProcess
   └─ CONTAINS ─→ Feature
                    └─ EXPOSED_BY ─→ Page / ApiEndpoint
                                       └─ HANDLED_BY ─→ Controller / Method
                                                          └─ IMPLEMENTED_BY ─→ Service / Method
                                                                                └─ CALLS ─→ Mapper / Method
                                                                                              └─ EXECUTES ─→ SqlStatement
                                                                                                               └─ READS / WRITES ─→ Table
                                                                                                                                      └─ HAS_COLUMN ─→ Column
```

一句话：**业务域包含功能 → 功能由页面/接口暴露 → 接口由控制器处理 → 控制器调用服务 → 服务调用 Mapper → Mapper 执行 SQL → SQL 读写表 → 表拥有字段。**

沿这条链路，任意一层都可向前追到业务、向后追到数据，且每一段都带证据与置信度。这条链路也是：
- **QA 问答**的主路径（"用户注册涉及哪些表"= 从 Feature 走到 Table）；
- **影响分析**的反向路径（"改这张表影响哪些功能"= 从 Table 反走到 Feature/BusinessDomain）；
- **测试生成**的出发点（从 Feature 派生 `Feature → API → Function → Table` 断言链）。

## 4. 层间映射关系表（以 `EdgeType.java` 为准）

> 26 种 EdgeType，按"层间映射"分组。所有关系均带 `evidence`、`confidence`、`status`。

### 4.1 业务层内部 & 业务→功能/数据

| 关系 | 方向 | 含义 |
|---|---|---|
| `CONTAINS` | Domain/Process → Feature/子域 | 包含 |
| `HAS_RULE` | Feature/Method → BusinessRule | 拥有/执行业务规则 |
| `MAPS_TO` | BusinessObject → Table/Feature | 业务对象对应到表或功能 |
| `BELONGS_TO` | 节点 → Domain/Project | 属于 |

### 4.2 功能层内部 & 功能→代码

| 关系 | 方向 | 含义 |
|---|---|---|
| `EXPOSED_BY` | Feature → Page/ApiEndpoint | 功能由页面/接口暴露 |
| `CALLS` | Page/Button/Method → ApiEndpoint/Method | 前端调用接口 / 代码调用方法 |
| `REQUIRES_PERMISSION` | Feature → Permission | 功能需要权限 |
| `HANDLED_BY` | ApiEndpoint → Controller/Method | 接口由控制器/方法处理 |
| `IMPLEMENTED_BY` | Feature/ApiEndpoint → Service/Method | 由服务/方法实现 |
| `USES` | Feature/Service → 资源 | 使用 |

### 4.3 代码层内部 & 代码→数据

| 关系 | 方向 | 含义 |
|---|---|---|
| `EXECUTES` | Mapper/Method → SqlStatement | 执行 SQL |
| `READS` | SqlStatement/Method → Table | 读表 |
| `WRITES` | SqlStatement/Method → Table | 写表 |
| `JOINS` | SqlStatement → Table | JOIN 表 |
| `CALLS_EXTERNAL` | Method → ExternalSystem | 调用外部系统 |
| `TRIGGERS` | ScheduledJob → Method | 定时任务触发方法 |
| `CONSUMES` | MQConsumer → MQTopic | 消费消息 |

### 4.4 数据层内部

| 关系 | 方向 | 含义 |
|---|---|---|
| `HAS_COLUMN` | Table → Column | 拥有字段 |
| `HAS_INDEX` | Table → Index | 拥有索引 |
| `UNIQUE_ON` | Table → Column | 唯一约束字段 |
| `REFERENCES` | Table → Table | 外键引用 |

### 4.5 测试与证据（横切）

| 关系 | 方向 | 含义 |
|---|---|---|
| `VERIFIED_BY` | Feature/ApiEndpoint → TestCase | 由测试验证 |
| `ASSERTS` | TestCase → Assertion | 断言 |
| `HAS_EVIDENCE` | 节点/边 → Evidence | 拥有证据 |

### 4.6 变更闭环（横切）

| 关系 | 方向 | 含义 |
|---|---|---|
| `AFFECTS` | ChangeTask/VersionRisk → 节点 | 影响 |
| `FIXED_BY` | ChangeTask → Patch | 由补丁修复 |
| `MIGRATES_TO` | Patch/节点 → 节点 | 迁移到 |
| `DEPENDS_ON` | Dependency/节点 → 节点 | 依赖于 |

## 5. 追溯路径（从任一层追到其他三层）

四层互通的追溯能力是关系文档的核心价值。下表给出从某一层出发的典型追问与路径：

| 出发层 | 典型追问 | 正向路径 | 反向路径 |
|---|---|---|---|
| 业务 | "这个业务域由哪些功能实现？落到哪些表？" | Domain `CONTAINS` Feature `EXPOSED_BY` Api `HANDLED_BY` Method `EXECUTES` SQL `READS/WRITE` Table | BusinessObject `MAPS_TO` Table |
| 功能 | "这个功能改了会影响哪些表？谁在用？" | Feature `EXPOSED_BY` Api `HANDLED_BY` Method … `READS/WRITE` Table | Feature `VERIFIED_BY` TestCase；Page/Button `CALLS` Api |
| 代码 | "这个 Service 被哪些接口调用？读写哪些表？" | Service `EXECUTES` SQL `READS/WRITE` Table | Api `HANDLED_BY` Method `CALLS` Service |
| 数据 | "这张表被哪些功能/业务读写？" | Table `READS/WRITE`← SQL ← Mapper ← Service ← Api ← Feature ← Domain | BusinessObject `MAPS_TO` Table |

> 追溯的每一段都附带 `evidence`（文件/行号/SQL/文档片段）、`confidence`（0–1）、`status`（`CONFIRMED` / `PENDING_CONFIRM` / `REJECTED`）。回答时必须能展开为证据卡片，见第 7 节。

## 6. 证据与置信度规则

### 6.1 来源优先级（确定性优先，AI 候选待确认）

| 来源 | 建议权重 | 默认状态 | 说明 |
|---|---:|---|---|
| TEST / RUNTIME | 0.90 | CONFIRMED | 真实执行/测试反证最强 |
| CODE / DB / SQL | 0.80 | CONFIRMED | 静态确定性事实 |
| DOC | 0.60 | PENDING_CONFIRM | 文档可能过期 |
| AI_INFERENCE / DOC_AI / CODE_AI | 0.40 | PENDING_CONFIRM | 只能作为候选 |
| HUMAN_REVIEW | 0.95 | CONFIRMED | 人工确认可强制升权，留审计 |

### 6.2 置信度回写（测试闭环）

| 事件 | 置信度变化 |
|---|---|
| 测试 PASS | +0.10 |
| 测试 FAIL | −0.20 |
| 人工审核通过 | +0.15 |

### 6.3 三类来源差异（必须显式区分）

- **文档有但代码/运行时没有** → 标记 `planned_or_stale`（计划中或已过期）。
- **代码有但运行时长期未观测** → 标记 `static_only_candidate`（静态候选）。
- **运行时有但静态图没有** → 标记 `dynamic_only_candidate`（反射/动态 SQL/插件/抽取漏检）。

这三类差异不能偷偷覆盖，必须作为冲突或缺口显式记录（见第 8 节）。

### 6.4 Claim 层约束

- AI 来源只能进 `lg_knowledge_claim` 且默认 `PENDING_CONFIRM`，不能直接写 `CONFIRMED` 边。
- 同一 `subject + predicate + object` 幂等去重，重复写入合并 evidenceIds 并取更高 confidence。
- Claim 经 `KnowledgeCompiler` 编译、冲突检测、评分后才投影为图谱节点/边。

## 7. 证据卡片格式（对齐 `GraphRagEvidenceCard`）

任何关系结论在被 QA 引用时，都应能展开为如下证据卡片：

| 字段 | 说明 |
|---|---|
| `evidenceId` | 证据 ID |
| `sourceType` | `CODE` / `DOC` / `DB` / `TEST` / `RUNTIME` |
| `sourcePath` | 来源文件/路径 |
| `startLine` / `endLine` | 行号 |
| `nodeKey` | 关联节点 key |
| `claimId` | 关联 Claim ID |
| `excerpt` | 内容摘录 |
| `confidence` | 置信度（0–1） |
| `status` | `CONFIRMED` / `PENDING_CONFIRM` / `REJECTED` |
| `relationTypes` | 路径上出现的关系类型（distinct） |
| `pathNodeKeys` | 路径经过的节点 key 序列 |

> 规则：**AI 输出报告/回答时只能引用证据卡片中的事实；没有证据卡片的内容必须进入"推断/待确认"章节。** 这条约束是 QA 事实底座可信度的底线。

## 8. 缺口与冲突在关系中的体现

关系不是"全连通"的，缺口与冲突本身就是关系文档的一部分：

| 形态 | 表现 | 落地 |
|---|---|---|
| 孤立节点 | 某层节点与其他层无连接（如某 `OperationType` 仅 1 连接） | graphify 标记 isolated；进 `lg_gap_task` |
| 薄弱社区 | 社区内聚度低（如 Service Layer cohesion 0.05） | graphify 标记 thin community |
| 推断边未验证 | INFERRED 边需复核（如 223 条 `getMessage()` 推断边） | 进 Review 队列 |
| 来源冲突 | 文档/代码/DB 说法矛盾 | `lg_evidence_conflict` 记录 |
| 知识缺口 | Claim 层缺失关键谓词 | `lg_gap_task`（7 类缺口类型） |

> LegacyGraph 自身扫描已发现的缺口见 `02-分层映射详解.md` §"已知缺口"（956 孤立节点、30 个薄弱社区、若干 god nodes）。这些缺口要被 QA 系统识别为"不能高置信回答"的边界。

## 9. 三类图谱与四层关系的对应

| 投影图谱 | 主要层 | 典型节点 | 回答 |
|---|---|---|---|
| 业务图谱 | 业务层（+ 数据层映射） | BusinessDomain / Process / Object / Rule / Role | "为什么存在" |
| 功能图谱 | 功能层（+ 代码层入口 + 测试） | Feature / Page / Button / Permission / ApiEndpoint / TestCase | "如何触发" |
| 代码图谱 | 代码层 + 数据层 | Controller / Service / Method / Mapper / SqlStatement / Table / Column | "由什么实现、读写什么数据" |

三者价值来自**可追溯连通**，而非各自完整。一条关键路径应能贯穿三图：`BusinessCapability → Feature → API → Function → Table`。

## 10. 本框架的使用方式

- **写 QA 事实底座**（`03`）：按四层 + 核心链路组织 FAQ，每条结论附证据卡片字段。
- **回答 QA 问题**：沿第 5 节追溯路径检索，用第 7 节证据卡片作答，按第 6 节规则标注置信度。
- **做影响分析**：反向走核心链路，用 `AFFECTS` / `DEPENDS_ON` 扩展变更闭环。
- **识别边界**：用第 8 节缺口/冲突判断"能否高置信回答"，不能就进"推断/待确认"。
