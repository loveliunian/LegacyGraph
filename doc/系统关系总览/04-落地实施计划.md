# 04 — 落地实施计划：系统关系总览能力

> 文档定位：把 `01/02/03` 描述的四层关系能力，从**文档结论**落地为**系统可自动产出、可查询、可问答**的功能。
> 基于当前代码现状（V1–V37，43 张活跃表，34 Controller，24 Agent）制定，每阶段给出具体新建/修改类与复用点。
> 日期：2026-07-06

---

## 0. 目标与范围

**总目标**：让 LegacyGraph 对任意被分析项目都能——
1. **自动产出**业务-功能-代码-数据四层关系总览；
2. **可查询**关系（按业务域/层/链路）；
3. **可问答**（EnhancedQaAgent 真正能回答关系问题并带证据）；
4. **可导出**为报告。

**范围**：4 个阶段，P0→P2。本计划只规划落地路径，不包含三类图谱方法论本身的变更。

## 3. 实施方案（4 阶段）

### 阶段 1（P0）：QA 事实底座接入 — 最快见效

**目标**：把 `02/03` 结构化关系写入向量/Claim/语义缓存，让 EnhancedQaAgent 真正能回答关系问题。EnhancedQaAgent 已接 `loadRelevantClaims`，**底座填充后即自动生效，无需改 Agent**。

**新建**：
| 文件 | 职责 |
|---|---|
| `service/systemoverview/SystemOverviewIngestService.java` | 解析 `02/03` 结构化表 → 生成 VectorDocument + KnowledgeClaimDraft + 语义缓存条目 |
| `dto/systemoverview/SystemOverviewIngestRequest.java` | 导入请求（projectId/versionId/数据源：md文件或内联结构化表） |
| `controller/SystemOverviewIngestController.java` | `POST /lg/system-overview/ingest`（触发导入，幂等） |

**复用（不改）**：
- `VectorRetrievalService.batchUpsertVectors` — 向量化关系表行
- `KnowledgeClaimService.upsertDrafts` — 批量写 Claim（每行映射 = 一条 Claim：`subject=业务能力`，`predicate=MAPS_TO/IMPLEMENTED_BY`，`object=Controller/表`）
- `SemanticCache.put(projectId, question, answer, evidence)` — FAQ 入缓存（内部自动 embedding question）
- `LlmGateway`（Embedding）— 生成向量

**数据流**：
```text
02/03 结构化表(业务域→功能→代码→数据)
  → SystemOverviewIngestService
    ├ VectorRetrievalService.batchUpsertVectors → lg_vector_document(chunkType=SYSTEM_OVERVIEW)
    ├ KnowledgeClaimService.upsertDrafts → lg_knowledge_claim(sourceType=DOC)
    │     · 默认 status=PENDING_CONFIRM（DOC 来源，对齐 01 §6.1）
    │     · 仅经代码核查的确定性映射（Controller/Service/表名）标 CONFIRMED，附 evidenceId 指向代码位置
    │     · AI 抽取项（业务域/规则）保持 PENDING_CONFIRM
    └ SemanticCache.put → lg_semantic_cache(FAQ question→answer+evidences)
  → EnhancedQaAgent.loadRelevantClaims / HybridRetrievalService 自动检索（无需改）
```

**Claim 谓词映射**（对齐 `01` §4）：
| 结构化表行 | Claim predicate | subject | object |
|---|---|---|---|
| 业务域→Controller | `IMPLEMENTED_BY` | 业务能力 | Controller |
| Controller→Service | `HANDLED_BY`/`IMPLEMENTED_BY` | ApiEndpoint | Service |
| Service→表 | `READS`/`WRITES` | Service | Table |
| Feature/Service→资源 | `USES` | Feature/Service | 资源节点 |
| 业务域→表 | `MAPS_TO` | BusinessObject | Table |

**验收**：
- 导入后 `lg_vector_document` 新增 `chunkType=SYSTEM_OVERVIEW` 记录
- `lg_knowledge_claim` 新增关系 Claim
- 问"三类图谱存哪""扫描涉及哪些表""QA 用到哪些表"，EnhancedQaAgent 带证据回答（`sourceType=DOC`，`status=CONFIRMED`）
- 重复导入幂等（Claim 去重，向量 upsert）

**工作量**：1–2 天。**风险低**（只新增 ingest 服务，不改现有 Agent）。

### 阶段 2（P1）：系统自分析功能 — 核心价值

**目标**：让 LegacyGraph 对任意被分析项目自动产出四层关系总览，可查询、可导出。

**新建**：
| 文件 | 职责 |
|---|---|
| `service/systemoverview/SystemOverviewService.java` | 从 Neo4j 图谱 + Claim + 表结构投影 12 业务域/四层映射/贯穿链路 |
| `dto/systemoverview/SystemOverviewDTO.java` | 四层关系总览结构（对齐 `01` §2 + `02` §0.1） |
| `dto/systemoverview/LayerMappingDTO.java` | 单业务域的业务↔功能↔代码↔数据映射 |
| `controller/SystemOverviewController.java` | `GET /lg/projects/{projectId}/system-overview`（按域/层/链路查询） |
| `frontend/src/views/workbench/SystemOverviewWorkbench.vue` | 四层关系可视化 + 链路下钻 |

**修改**：
| 文件 | 改动 |
|---|---|
| `service/report/ReportExportService.java` | `ReportType` 枚举加 `SYSTEM_OVERVIEW`；`exportReport` 加分支 |
| `service/report/ScanResearchReportService.java` | 复用其 md 生成模式，新增 `generateSystemOverviewMarkdown` |
| `plugin/BuiltinPluginRegistrar.java` | 注册 `system-overview-scanner`(SCANNER)、`system-overview-view`(GRAPH_VIEW) |

**复用（不改）**：
- `GraphQueryService` / `Neo4jGraphDao` — 查图谱节点边
- `KnowledgeClaimService.listClaimsBySubjects` — 查 Claim
- `NodeType`/`EdgeType` — 投影规则
- `FeatureSliceSynthesizer`（若已接入）— 功能切片合成

**投影规则**（对齐 `02` §0.1）：
```text
业务域识别：按 Controller 类名→业务域映射（不用纯前缀，因 /lg/projects/{projectId} 下有多个 Controller 共享前缀）
  映射表（对齐 02 §0.1，硬编码于 SystemOverviewService）：
    AuthController→项目数据源·认证  ScanController→资料扫描
    GraphQuery/GraphDiff→图谱构建查询  Fact/Knowledge/Review/EvidenceConflict→事实证据
    EnhancedQa/GraphQa→QA问答  TestCase/Validation→测试闭环
    CodeUnderstanding→代码理解  ChangeTask/CrossRepoImpact→变更迁移
    Report/ReportExport→报告度量  Graphify*→Graphify集成
    LlmProvider/PromptTemplate/LlmAgent/AgentRun→LLM/Agent  其余→系统管理
四层映射：
  业务层 = BusinessDomain/Process/Object/Rule 节点（AI 抽取）
  功能层 = Feature/Page/Button/ApiEndpoint + 前端 views 对应
  代码层 = Controller/Service/Method/Mapper（从图谱 CALLS/HANDLED_BY 链）
  数据层 = Table/Column（从 READS/WRITES + JDBC 元数据）
贯穿链路 = 按 01 §3 核心链路从 BusinessDomain 走到 Column
```

**接口**：
```http
GET /lg/projects/{projectId}/system-overview                    # 全量总览
GET /lg/projects/{projectId}/system-overview/domains/{domainId}  # 按业务域
GET /lg/projects/{projectId}/system-overview/paths?from=Feature&to=Table  # 贯穿链路
GET /reports/system-overview/{projectId}/{versionId}             # 导出 md
```

**验收**：
- 对任意已扫描项目能生成四层关系总览
- 前端可按业务域查看、按链路下钻
- 可导出 md 报告（结构对齐 `02`）
- 业务域识别准确率（按 ApiEndpoint 前缀，可人工校验）

**工作量**：3–5 天。**风险中**（投影规则需调试，可先做后端 API 再做前端）。

### 阶段 3（P2）：KnowledgeCompiler 切主链路 + 双轨配置

**目标**：从 dry-run 切到实际写入，实现 `write-mode` 配置，收敛双写。

**新建/修改**：
| 文件 | 改动 |
|---|---|
| `config/GraphWriteConfig.java`（新建） | `@ConfigurationProperties("legacygraph.graph")`，`writeMode`(direct/shadow/claim-compiler) |
| `service/KnowledgeCompiler.java` | `compile` 增加 `dryRun=false` 分支：调 `EvidenceGraphWriter.writeIntent` 实际写图 |
| `task/AdapterExecutionService.java` | 按配置选直接写图或 Claim 编译；`shadow` 模式双写 + 对比报告 |
| `application.yml` | `legacygraph.graph.write-mode: shadow`（默认） |

**复用**：
- `GraphWriteIntentService` + `GraphWriteIntentWorker`（outbox 已就绪）
- `GraphWriteReconciler`（对账已就绪）
- `EvidenceGraphWriter.writeIntent`

**模式语义**：
| 模式 | 行为 |
|---|---|
| `direct` | 现状：Adapter 直接写图（默认兼容） |
| `shadow` | 直接写图 + Claim 编译 dry-run 对比（安全过渡） |
| `claim-compiler` | Claim 编译实际写图，Adapter 只产 Claim Draft |

**验收**：
- `shadow` 模式产出对比报告（直接写图 vs Claim 编译的节点边差异）
- 节点边数量无大幅回退（<5%）
- `claim-compiler` 模式下图谱可正常构建
- 配置可热切换（重启生效）

**工作量**：3–4 天。**风险高**（动扫描主链路，必须有开关和回滚）。**默认 `shadow`，不破坏现状**。

### 阶段 4（P2）：graphify 自身导入 + 插件增强

**目标**：把 graphify 扫描数据导入系统图谱，接入 CodeGraph MCP/动态插件。

**4.1 graphify 自身导入**
- 复用 `GraphifyImportService` 把 `graphify-out/graph.json`（8573 节点）导入 Neo4j
- 新建 `task/SelfAnalysisBootstrapService.java`：应用启动时（可选）或手动触发，把 LegacyGraph 自身作为"被分析项目"导入
- 数据：复用 `lg_project`（自身项目）+ `lg_scan_version` + Neo4j

**4.2 CodeGraph MCP 接入 ToolRouter**
- 修改 `understanding/ToolRouter.java`：`codegraph-tool` 真正路由到 CodeGraph MCP 调用（`codegraph_explore`/`codegraph_callers` 等）
- 复用 `ToolRunRecorder` 记录工具运行

**4.3 插件动态加载**
- 修改 `plugin/PluginRegistry.java`：支持外部 MCP 插件动态注册（`registerExternal(McpPluginDescriptor)`）
- 新建 `dto/plugin/ExternalPluginDescriptor.java`：含 MCP endpoint、协议、鉴权
- 接口：`POST /lg/plugins/register`（动态注册外部 MCP 插件）

**验收**：
- 自身 8573 节点进 Neo4j，QA 可检索
- ToolRouter 调 CodeGraph 返回结果，落 `lg_tool_run`
- 外部 MCP 插件可动态注册并调用

**工作量**：4–5 天。**风险中**（MCP 接入需调试）。

### 11.1 测试策略（每阶段配套，现有 223 测试类）
| 阶段 | 测试类 | 覆盖点 |
|---|---|---|

| 3 | `KnowledgeCompilerTest`（扩展） | 非 dry-run 分支、shadow 对比报告 |
| 4 | `SelfAnalysisBootstrapServiceTest`、`ToolRouterCodeGraphTest` | 自身导入、CodeGraph 调用 |
验收命令：`mvn test -Dtest=<测试类>`（以 `mvn` 为准，无 wrapper）。

### 11.2 权限与安全
- 现有 Controller **不用 `@PreAuthorize`**，沿用 `SecurityConfig` + JWT 角色模式。
- **高危接口限 admin**：`POST /lg/system-overview/ingest`、`POST /lg/plugins/register`。
- 查询接口（`GET system-overview`）沿用项目级权限。
- 数据按 `projectId + versionId` 隔离。

### 11.3 审计
- 导入/插件注册/报告导出走现有 AOP/拦截器记 `lg_sys_operation_log`；**需确认 AOP 覆盖新 Controller**，若不覆盖则显式调 `AuditService`。
- Agent 调用记 `lg_agent_run`；阶段4 工具调用记 `lg_tool_run`（复用 `ToolRunRecorder`）。

### 11.4 多租户
- 数据按 `projectId` 隔离（ingest / system-overview 均带 projectId）。
- 插件注册是**全局操作**，限 admin + 记审计。
