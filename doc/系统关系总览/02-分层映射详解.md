> 文档定位：把 `01` 的四层框架落到 **LegacyGraph 系统自身**，给出 12 个业务域的 业务↔功能↔代码↔数据 实际映射，附扫描证据。

## 1. 业务域 1：项目与数据源管理

**业务能力**：纳管被分析的老项目，配置其代码仓库、数据库连接、文档资料，并提供用户认证与项目概览。

| 层 | 实际节点 |
|---|---|
| 功能（前端） | `project` `source` `login` `dashboard` |
| 功能（API） | `AuthController` `/lg/auth`；`ProjectController` `/lg/projects`；`ProjectOverviewController` `/lg/projects/{projectId}`；`SourceController` `/lg/projects/{projectId}/sources` |
| 代码 | `service/ProjectService` `service/SourceService` `service/AuthService`；`util/JwtUtil`；`repository/*Repository`；`config/SecurityConfig` |
| 数据 | `lg_project` `lg_code_repo` `lg_db_connection` `lg_document` `lg_sys_user` `lg_sys_role` `lg_sys_user_role` |

核心链路：`项目(Project) BELONGS_TO 业务域 → 仓库/连接/文档 作为数据源 → SourceController 配置 → ScanController 触发扫描`。

## 2. 业务域 2：资料扫描

**业务能力**：把代码仓库、数据库、文档资料转成结构化事实与图谱节点边。含适配器抽取、数据库元数据扫描、扫描后 AI 编排、增量扫描。

| 层 | 实际节点 |
|---|---|
| 功能（前端） | `scan` `source` |
| 功能（API） | `ScanController` `/lg/projects/{projectId}/scan-versions`（start/resume/cancel/pause/logs/progress/performance-report） |
| 代码 | `task/ProjectScanner`（主入口）；`task/AiScanOrchestrator`（AI 编排）；`task/ScanScopeResolver` `task/AssetDiscoveryService`；`extractors/adapter/*`（`JavaCodeAdapter` `JavaServiceCallAdapter` `MyBatisXmlAdapter` `VueFrontendAdapter` `DocumentAdapter`）；`extractors/DatabaseMetadataScanService` `extractors/DatabaseConstraintExtractor` |
| 数据 | `lg_scan_version` `lg_scan_task` `lg_source_asset_snapshot` `lg_ai_scan_job` `lg_document` `lg_doc_chunk` |

适配器→图谱产出（`01` §3 链路的入口）：

| Adapter | 处理对象 | 产出节点/关系 |
|---|---|---|
| `JavaCodeAdapter` | `@RestController/@Controller` | `Controller` `Method` `ApiEndpoint` `Permission`；`HANDLED_BY` |
| `JavaServiceCallAdapter` | 非 Controller Java | `Service` `Method`；`CALLS` `SERVICE_CALL` fact |
| `MyBatisXmlAdapter` | Mapper XML | `Mapper` `SqlStatement`；`EXECUTES` `READS/WRITES` `JOINS` |
| `VueFrontendAdapter` | `.vue/.jsx/.tsx` | `Page` `Button`；`CALLS`（前端→Api） |
| `DocumentAdapter` | `.md/.pdf/.docx/...` | `DOC` fact → 供 AI 抽业务事实 |

数据库扫描四层（结构+约束+样本+语义）：`DatabaseMetadataScanService` 抽表/列/主键 → `DatabaseConstraintExtractor` 抽 FK/索引/唯一约束（`REFERENCES` `HAS_INDEX` `UNIQUE_ON`）→ `DbSchemaAnalysisAgent` 语义增强。

## 3. 业务域 3：图谱构建与查询

**业务能力**：把扫描产出统一写入图谱（Neo4j 主存储 + PG 证据），并支持图谱查询、版本差异、写入对账。

| 层 | 实际节点 |
|---|---|
| 功能（前端） | `graph` `workbench` |
| 功能（API） | `GraphQueryController` `/lg/projects/{projectId}`；`GraphDiffController` `/lg/graph/diff` |
| 代码 | `builder/GraphBuilder` `builder/FrontendGraphBuilder` `builder/BusinessGraphBuilder`；`builder/EvidenceGraphWriter`（统一写入器）；`dao/Neo4jGraphDao`（Neo4j 唯一入口）；`service/GraphQueryService`；`service/GraphWriteIntentService` `task/GraphWriteIntentWorker`；`common/NodeType`（38）`common/EdgeType`（26） |
| 数据 | **Neo4j**（图谱节点/边主存储）；`lg_evidence` `lg_node_evidence` `lg_edge_evidence` `lg_graph_write_intent` |

关键事实（QA 高频）：
- 图谱节点/边存 **Neo4j**，`Neo4jGraphDao.mergeNode/mergeEdge` 幂等写入。
- 证据存 **PostgreSQL**（`lg_evidence` + 关联表）。
- `lg_graph_node/lg_graph_edge` **已废弃**，仅保留 Bean 兼容。
- `EvidenceGraphWriter` 同时写 Neo4j + PG，失败标记 `INCOMPLETE`，由 `GraphWriteReconciler` 对账。
- `GraphWriteIntent` 走 outbox 模式（`lg_graph_write_intent`，幂等 + 重试）。

## 4. 业务域 4：事实/证据/知识断言

**业务能力**：管理原始事实、证据、知识断言（Claim）、缺口、领域本体、人工审核、证据冲突。这是"知识编译器"的主场。

| 层 | 实际节点 |
|---|---|
| 功能（前端） | `fact` `review` |
| 功能（API） | `FactController` `/lg/projects/{projectId}`；`KnowledgeController` `/lg/projects/{projectId}/knowledge`；`ReviewController` `/lg/projects/{projectId}/reviews`；`EvidenceConflictController` `/lg/evidence-conflicts` |
| 代码 | `service/KnowledgeClaimService`（Claim 幂等 upsert）；`service/KnowledgeCompiler`（Claim→图谱 dry-run 编译）；`service/GapFinderService` `agent/GapFinderAgent`；`service/ReviewService`；`agent/CodeFactAgent` `agent/DocUnderstandingAgent` `agent/GraphMergeAgent` |
| 数据 | `lg_fact` `lg_evidence` `lg_node_evidence` `lg_edge_evidence` `lg_knowledge_claim` `lg_gap_task` `lg_domain_ontology_term` `lg_domain_ontology_relation` `lg_review_record` `lg_evidence_conflict` |

Claim 规则：AI 来源默认 `PENDING_CONFIRM`；`CODE/DB/RUNTIME/TEST` 且置信度足够可 `CONFIRMED`；同 `subject+predicate+object` 去重合并 evidenceIds 取更高 confidence。冲突不自动解决，进 `lg_evidence_conflict` + Review。

## 5. 业务域 5：QA 问答

**业务能力**：自然语言问答，五段式 RAG（查询改写/HyDE → 向量召回 → 图谱邻域扩展 → 重排序 → LLM 生成），支持多轮、流式、证据链接、反馈、GraphRAG。

| 层 | 实际节点 |
|---|---|
| 功能（前端） | `graph`（`GraphQa.vue`） |
| 功能（API） | `EnhancedQaController` `/qa` `/qa/ask/stream`（SSE）；`GraphQaController` `/qa` |
| 代码 | `agent/EnhancedQaAgent`（主流程）；`agent/GraphRagPlannerAgent` `service/GraphRagPlanExecutor`；`agent/HyDEGenerator`；`service/QueryIntentClassifier` `service/QueryRewriter`；`service/HybridRetrievalService` `service/ReRankingService`（`KeywordReranker`/`CrossEncoderReranker`）；`service/ConversationContextManager`；`dto/rag/GraphRagEvidenceCard` |
| 数据 | `lg_qa_conversation` `lg_qa_message` `lg_qa_feedback` `lg_semantic_cache` `lg_vector_document` `lg_doc_chunk` `lg_knowledge_claim` |

意图→图谱深度（`01` §5 追溯路径的触发点）：

| 意图 | 图谱深度 | 用 Planner |
|---|---:|---|
| `FACT_LOOKUP` | 1 | 否 |
| `STRUCTURAL` | 2 | 是 |
| `RELATIONAL` | 3 | 是 |
| `COMPARATIVE`/`TEMPORAL`/`EXPLANATION` | 1–2 | 否 |

证据卡片由 `GraphRagPlanExecutor` 产出，字段对齐 `GraphRagEvidenceCard`（`01` §7）。回答必须能展开为证据卡片。

## 13. 贯穿链路实例（业务→功能→代码→数据）

### 13.1 链路 A：资料扫描 → 图谱构建

```text
[业务] 资料扫描业务域
  └ ScanController(/scan-versions) 启动扫描
    └ ProjectScanner.runScanBody
      └ ScanScopeResolver → AssetDiscoveryService
        └ Adapter(Java/Service/MyBatis/Vue/Doc) + DatabaseMetadataScanService
          └ GraphBuilder/FrontendGraphBuilder
            └ EvidenceGraphWriter → Neo4jGraphDao.mergeNode/mergeEdge
              ├ Neo4j(节点/边)
              └ lg_evidence / lg_node_evidence / lg_edge_evidence
    └ AiScanOrchestrator → DocUnderstandingAgent/CodeFactAgent
      └ lg_knowledge_claim / lg_gap_task
```

### 13.2 链路 B：QA 问答

```text
[业务] QA 问答业务域
  └ EnhancedQaController /qa/ask/stream (SSE)
    └ EnhancedQaAgent.answerStream
      ├ QueryIntentClassifier → (FACT_LOOKUP/STRUCTURAL/RELATIONAL/...)
      ├ QueryRewriter + HyDEGenerator → HybridRetrievalService
      │   ├ lg_vector_document(向量) + lg_doc_chunk + Neo4j(图邻域)
      │   └ ReRankingService
      ├ (复杂意图) GraphRagPlannerAgent → GraphRagPlanExecutor
      │   └ lg_knowledge_claim + Neo4j 路径 → GraphRagEvidenceCard
      ├ LlmGateway.callStream → LLM
      └ lg_qa_conversation / lg_qa_message / lg_semantic_cache / lg_qa_feedback
```

### 13.3 链路 C：测试生成与回写

```text
[业务] 测试闭环业务域
  └ TestCaseController
    └ TestCaseAgent 从 Feature 出发生成用例
      └ lg_test_case / lg_test_assertion
    └ ApiTestExecutor / E2eTestExecutor / DbTestExecutor 执行
      └ lg_test_run / lg_test_result
    └ 回写图谱置信度(PASS+0.10/FAIL−0.20) → Neo4j + lg_knowledge_claim
```

### 13.4 链路 D：变更影响分析

```text
[业务] 变更影响业务域
  └ ChangeTaskController /change-tasks/{id}/impact
    └ ChangeImpactAgent 沿图谱依赖链反向遍历
      └ Neo4j: Table ←READS/WRITE← SQL ←EXECUTES← Mapper ←CALLS← Service ←HANDLED_BY← Api ←EXPOSED_BY← Feature
      └ lg_change_task / lg_patch_file / lg_pr_task / lg_migration_risk
```

### 13.5 链路 E：Graphify 导入

```text
[业务] Graphify 集成业务域
  └ GraphifyJobController /graphify/jobs
    └ GraphifyRunner(CLI) → graph.json
      └ Parser → Compatibility 校验 → Import 映射
        └ GraphMergeAgent → EvidenceGraphWriter → Neo4j
    └ GraphifyDiffController 差异 / rollback
```

| 主要社区 | Service Layer、Frontend、Controller、Graph Builder、AI Agent、DAO、Task/Orchestrator、Extractor、Config、Understanding/Tool、LLM、Testing、Common、Event、DTO、Entity |

| 知识缺口 | 956 孤立节点、30 个薄弱社区（<3 节点） |
