# LegacyGraph 开发规范文档

## 📋 概述

本文档定义 LegacyGraph 项目的代码风格、命名规范、Git 提交规范等开发约定。

---

## 🎯 Java 开发规范

### 包命名规范

- 全部小写，使用点分隔
- 遵循：`io.github.legacygraph.功能层`

**包名约定：**

| 包名 | 用途 |
|------|------|
| `io.github.legacygraph.controller` | HTTP 控制器 |
| `io.github.legacygraph.service` | 业务服务 |
| `io.github.legacygraph.entity` | 数据库实体 |
| `io.github.legacygraph.repository` | MyBatis-Plus Mapper |
| `io.github.legacygraph.dto` | 请求/响应 DTO |
| `io.github.legacygraph.common` | 通用工具 |
| `io.github.legacygraph.exception` | 异常 |
| `io.github.legacygraph.config` | 配置类 |
| `io.github.legacygraph.filter` | Servlet 过滤器 |
| `io.github.legacygraph.aspect` | AOP 切面 |
| `io.github.legacygraph.annotation` | 自定义注解 |
| `io.github.legacygraph.llm` | LLM 相关 |
| `io.github.legacygraph.llm.agent` | LLM Agent |
| `io.github.legacygraph.util` | 工具类 |
| `io.github.legacygraph.test` | 测试工具 |

### 类命名规范

- **大驼峰命名** (PascalCase)
- **后缀约定：**

| 后缀 | 用途 | 示例 |
|------|------|------|
| `Controller` | HTTP 控制器 | `ProjectController` |
| `Service` | 业务服务 | `ProjectService` |
| `Repository` / `Mapper` | 数据访问 | `ProjectRepository` |
| `Entity` | 实体 | （不使用后缀，直接命名）`Project` |
| `DTO` / `Request` / `Response` | 数据传输对象 | `LoginRequest`, `LoginResponse` |
| `Exception` | 异常 | `BusinessException` |
| `Config` | 配置类 | `SecurityConfig` |
| `Aspect` | AOP 切面 | `LogAspect` |
| `Agent` | LLM Agent | `CodeFactAgent` |
| `Util` / `Utils` | 工具类 | `JwtUtil` |

### 变量命名规范

- **局部变量/参数：** 小驼峰 (camelCase)
- **常量：** 全大写下划线分隔 (UPPER_SNAKE_CASE)
- **静态变量：** 小驼峰

**示例：**
```java
// 正确
private final int MAX_PAGE_SIZE = 100;
private String userName;
public void getUserById(String userId) { }

// 错误
private int maxpagesize = 100;
private String UserName;
```

### 方法命名规范

- 小驼峰命名
- 动词开头

**常用动词约定：**

| 动词 | 用途 | 示例 |
|------|------|------|
| `get` | 获取单个对象 | `getById`, `getByUsername` |
| `list` / `query` | 查询列表 | `listAll`, `queryByProjectId` |
| `create` / `add` | 创建 | `create`, `addItem` |
| `update` | 更新 | `update`, `updateStatus` |
| `delete` / `remove` | 删除 | `deleteById`, `removeItem` |
| `save` | 保存 | `save` |
| `is` / `check` | 判断/检查 | `isSensitiveParam`, `checkPermission` |
| `handle` / `process` | 处理 | `handleException`, `processRequest` |
| `convert` / `map` | 转换 | `convertToDTO`, `mapToEntity` |

### 注释规范

- 类必须添加 Javadoc 注释，说明用途
- 公共方法建议添加 Javadoc 说明参数和返回值
- 复杂业务逻辑需要添加行内注释说明

**示例：**
```java
/**
 * 用户服务 - 处理用户相关业务
 */
@Service
public class SysUserService {
    
    /**
     * 根据用户名查询用户
     * @param username 用户名
     * @return 用户实体，不存在返回 null
     */
    public SysUser getByUsername(String username) {
        // 查询逻辑
    }
}
```

### 格式化

- 使用 4 空格缩进（不使用 Tab）
- 大括号不换行，K&R 风格
- 每行长度建议不超过 120 字符

## 🎯 TypeScript / Vue 开发规范

### 文件命名

- **组件文件：** PascalCase，例如 `ScanVersionList.vue`
- **工具函数/Type：** kebab-case，例如 `use-breakpoint.ts`
- **页面文件：** PascalCase，和组件一致

**示例：**
```
# 正确
components/GraphViewer.vue
composables/usePerformance.ts
utils/request.ts

# 错误
components/graphViewer.vue
components/graphviewer.vue
```

### 组件命名

- 组件名使用 PascalCase
- 单文件组件文件名与组件名一致
- 组合式 API 函数使用 `use` 前缀

**示例：**
```typescript
// 正确
const usePerformance = () => { }
const submitForm = () => { }

// 正确 - 组件
<script setup lang="ts">
const ScanVersionList = () => { }
</script>
```

### 响应式 API 规范

- 使用 `<script setup lang="ts">` 语法糖
- 使用 Composition API，不使用 Options API
- 使用 `ref` / `reactive` / `computed` / `watch`

### 样式规范

- 总是使用 `<style scoped>` 避免样式污染
- 使用 CSS 变量支持主题切换
- 类名使用 kebab-case

## 🔄 Git 提交规范

### Commit Message 格式

```
<type>(<scope>): <subject>
```

### type 类型

| type | 说明 |
|------|------|
| `feat` | 新增功能 |
| `fix` | 修复 Bug |
| `docs` | 文档修改 |
| `style` | 代码格式修改，不影响代码逻辑 |
| `refactor` | 重构（既不新增功能，也不修复 Bug） |
| `perf` | 性能优化 |
| `test` | 增加测试 |
| `chore` | 构建/工具链/辅助工具变动 |
| `revert` | 回滚提交 |

### scope 范围

可选，说明提交影响的范围，例如：
- `project` - 项目模块
- `scan` - 扫描模块
- `graph` - 图谱模块
- `agent` - LLM Agent 模块
- `frontend` - 前端整体
- `backend` - 后端整体
- `docs` - 文档

### subject 描述

- 使用祈使语气，第一人称现在时
- 首字母小写
- 结尾不加句号
- 长度不超过 50 字符

**示例：**
```
feat(system): add user management controller
fix(aspect): fix audit log saving null pointer
docs(readme): update project introduction
refactor(graph): optimize node filtering logic
```

### 分支命名规范

- `main` - 主分支，生产环境代码
- `develop` - 开发分支
- `feature/xxx` - 功能开发
- `fix/xxx` - 缺陷修复
- `docs/xxx` - 文档更新

**示例：**
```
feature/system-management
fix/audit-log-npe
docs/update-db-design
```

## 🗄️ 数据库命名规范

### 表名

- 全小写，下划线分隔
- 系统表前缀 `sys_`
- 业务表前缀 `lg_` （LegacyGraph 缩写）

**示例：**
```sql
-- 正确
sys_user
sys_operation_log
lg_project
lg_scan_version

-- 错误
SysUser
LGProject
scanVersion
```

### 字段名

- 全小写，下划线分隔
- 不使用保留字作为字段名

**示例：**
```sql
-- 正确
user_name
created_at
last_login_ip

-- 错误
userName
createAt
`order`
```

### 主键外键命名

- 主键统一命名 `id`
- 外键遵循 `表名_id`，例如 `project_id`

## 🧪 测试规范

### 后端测试

- 测试类命名：`被测试类Test`，例如 `ProjectControllerTest`
- 放在 `src/test/java` 对应包下
- 使用 JUnit 5
- 每个方法测试一个行为

### 前端测试

- 测试文件命名：`{name}.test.ts`
- 放在 `tests/unit/` 对应目录
- 使用 Vitest + Vue Test Utils

### 测试覆盖率目标

- 核心 Controller：100% 覆盖
- 核心 Service：80% 覆盖
- 工具类：尽可能覆盖

## 📝 注释指南

### 需要注释的场景

1. **复杂算法** - 说明思路
2. **非常规逻辑** - 说明为什么这么做
3. **对外 API** - 说明用法
4. **TODO/FIXME** - 标注待改进

### 不需要注释的场景

1. 自解释的代码（好代码本身就是注释）
2. 简单 getter/setter
3.  obvious 的逻辑

## 🔐 安全规范

1. **密码** - 永远不要明文存储，必须 BCrypt 加密
2. **敏感信息** - 日志中自动过滤密码、token、密钥
3. **SQL 注入** - 使用 MyBatis-Plus 参数绑定，不拼接 SQL
4. **XSS** - 前端输出内容自动转义
5. **文件上传** - 文件类型白名单、大小限制、重命名存储
6. **权限** - 需要认证的接口都要有检查
