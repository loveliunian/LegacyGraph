# LegacyGraph 前后端架构整体分析与改进建议

> 分析范围：`backend/src/main/java/io/github/legacygraph/`（226 个 Java 源文件，约 26643 行）与 `frontend/src/`（Vue 3 + TypeScript + Vite）。
> 分析日期：2026-07-01
> 分析方式：只读静态分析，未修改任何代码。所有问题均标注文件路径与行号。

---

## 一、总体评价

### 1.1 做得好的部分（应保持）

**后端：**
- 分层骨架完整：`controller / service / repository / dao / builder / llm / agent / task` 包划分意图清晰，依赖方向单向（controller → service/builder → dao/repository），无循环依赖。
- 统一返回结构 `Result<T>` + `GlobalExceptionHandler` 覆盖 BusinessException / 参数校验 / 认证 / 权限 / 通用异常，参数校验 `@Valid` 普遍使用。
- JWT 鉴权（`JwtAuthenticationFilter` + `SecurityConfig`）落地完整。
- LLM 集成：Prompt 走 DB 驱动（`PromptTemplateLoader`）未硬编码、API Key 存 DB、`PiiMaskingService` 日志脱敏、Provider 动态切换、结果缓存（inputHash 去重 7 天 TTL）、PromptRun 审计。这部分设计是全项目最规范的。
- `EvidenceGraphWriter`（351 行）集中了节点/边/证据的去重、状态裁决、证据继承，是架构最合理的模块。
- 异步线程池配置（`AsyncConfig`）合理：task/io/test 三池 + `CallerRunsPolicy` 背压 + `AsyncExceptionHandler`。
- 日志规范统一：全项目 `@Slf4j` + Lombok，结构化带参数上下文。**全项目无 TODO/FIXME/HACK**。
- 配置走 `${ENV_VAR:default}`，敏感信息环境变量注入，JPA 自动配置已正确排除（用 MyBatis-Plus）。

**前端：**
- 技术栈完整：Vue 3 + Pinia + vue-router + Element Plus + AntV G6/ECharts + axios + i18n + PWA + Vitest/Playwright 测试工具链。
- 目录骨架清晰（`api/ views/ components/ stores/ composables/ utils/ types/ router/ locales/`）。
- 路由全部懒加载。
- Pinia 持久化、主题 CSS 变量方案、axios 拦截器封装均存在。

### 1.2 核心判断

项目处于 **"半重构 / 半完成"状态**：新旧两套实现并存（API 风格、主题系统、GraphViewer 组件），大量弃用代码未清理，类型安全与权限层名存实亡。后端存在若干**严重正确性 Bug**（跨存储事务、Neo4j elementId、图谱合并查询）和**韧性缺口**（LLM 无超时/重试）。改进应分优先级推进，先止血（修 Bug + 加固 LLM），再治本（分层强制 + 清死代码 + 统一类型/权限）。

## 四、跨端共性问题

1. **半重构状态**：后端（`@Transactional` 与 Neo4j / DAO 与原生 Cypher 双轨）、前端（新旧 API、新旧主题、双 GraphViewer、双 NodeDetailDrawer）都存在新旧两套实现并存，应尽快收敛下线旧实现。
2. **分层边界未强制**：后端 Controller 越层直连 DAO/Repository，前端 view 越层直调 request。两端都缺**架构约束机制**（后端 ArchUnit、前端 eslint 规则 + 目录 lint）。
3. **类型契约不对齐**：后端 DTO 与前端 `types/` 无自动同步，前端 `User` 三份、`any` 194 处。建议引入 OpenAPI/Springdoc 自动生成前端类型。
4. **异常/错误处理不统一**：后端 Controller try-catch 绕过全局处理，前端 401 处理重复 + 硬刷新。
5. **韧性缺口**：后端 LLM 无超时/重试/熔断；前端无 token 刷新、轮询清理不统一。
