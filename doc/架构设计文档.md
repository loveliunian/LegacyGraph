# LegacyGraph 架构设计文档

## 📋 概述

**LegacyGraph** 是一个基于知识图谱和 LLM 的遗留系统迁移分析平台。

**目标：** 通过 AI 自动理解遗留系统代码、文档、数据库结构，构建统一知识图谱，辅助迁移决策和测试生成。

## 🔄 核心数据流

### 1. 扫描流程

```
用户创建扫描
    ↓
ScanController 创建 ScanVersion 和 ScanTask
    ↓
异步任务执行：
  ├─ 代码扫描 → JavaParser 解析 AST → 提取 API/Service/方法信息
  ├─ 数据库扫描 → 读取 information_schema → 提取表/字段/约束
  ├─ 文档解析 → Tika 解析 → 分块 → LLM 提取事实
  └─ 图谱构建 → 创建节点和关系 → 写入 PostgreSQL
              ↓
            写入 Neo4j
              ↓
          生成事实和证据
              ↓
          任务完成，通知前端
```

### 2. LLM Agent 流程

```
用户触发 / 自动触发
    ↓
LlmGateway 统一入口
    ↓
选择对应的 Agent（6种）
    ↓
构建 Prompt 模板
    ↓
调用 OpenAI API
    ↓
解析 JSON 响应
    ↓
后处理 / 写入数据库
    ↓
返回结果
```

### 3. 图谱查询流程

```
前端请求图谱数据
    ↓
GraphQueryController
    ↓
GraphQueryService
    ↓
从 PostgreSQL 查询节点和关系
    ↓
组装成前端需要的图谱格式
    ↓
返回给前端，VueFlow 渲染
```

### 4. 人工审核流程

```
用户看到待审核节点/关系
    ↓
点击确认/拒绝
    ↓
ReviewController 接收
    ↓
ReviewService 更新状态
    ↓
记录 ReviewRecord 变更历史
    ↓
返回成功
```

## 🧩 核心模块设计

### 1. 项目管理模块

**职责：** 管理用户的项目，项目是最高层级的组织单位

**主要组件：**
- `ProjectController` - HTTP 接口
- `ProjectService` - 业务逻辑
- `ProjectRepository` - 数据访问
- `Project` - 实体

---

### 2. 扫描模块

**职责：** 扫描代码、数据库、文档，提取事实

**主要组件：**
- `ScanController` - HTTP 接口
- `ScanVersionService` - 扫描版本管理
- `ScanVersion` - 扫描版本实体
- `ScanTask` - 单个扫描任务实体
- **扫描器：**
  - `CodeScanner` - Java 代码扫描，使用 JavaParser 解析 AST
  - `DatabaseScanner` - 数据库结构扫描，读取系统表
  - `DocumentScanner` - 文档扫描，使用 Tika 解析

---

### 3. 图谱模块

**职责：** 知识图谱的存储、查询、合并

**主要组件：**
- `GraphQueryController` - 查询接口
- `GraphQueryService` - 查询服务
- `GraphMergeService` - 多个版本图谱合并
- `GraphValidatorService` - 图谱质量验证
- `Neo4jSyncService` - 同步到 Neo4j
- `GraphNode` - 节点实体
- `GraphEdge` - 关系实体

**三类图谱：**
1. **代码图谱** - 代码中的类、方法、接口、调用关系
2. **特征图谱** - 业务特征和功能点
3. **业务图谱** - 业务实体和业务流程
4. **数据血缘** - 数据表之间的关系
5. **运行时图谱** - 运行时链路追踪
6. **统一图谱** - 合并后的完整图谱

---

### 4. 事实抽取模块

**职责：** 存储从代码/文档中抽取的事实，关联证据

**主要组件：**
- `FactController` - HTTP 接口
- `Fact` - 事实实体
- `Evidence` - 证据实体
- `NodeEvidence` / `EdgeEvidence` - 关联

---

### 5. LLM Agent 模块

**6 个 Agent 各负其职：**

| Agent | 职责 |
|-------|------|
| `CodeFactAgent` | 从代码中抽取业务事实 |
| `DocUnderstandingAgent` | 解析文档，分割片段，提取信息 |
| `FeatureMappingAgent` | 映射前端 API 到后端 Controller |
| `GraphMergeAgent` | 多轮扫描后，决策哪些节点需要合并 |
| `TestCaseAgent` | 根据图谱事实生成测试用例 |
| `ReviewAgent` | 自动整理证据，检测冲突 |

**统一入口：** `LlmGateway` - 负责 Agent 调度、Prompt 模板管理

---

### 6. 测试模块

**职责：** 生成测试用例，执行测试，收集结果

**主要组件：**
- `TestCaseController` - HTTP 接口
- `TestCaseService` - 测试用例管理
- `TestResultUpdateService` - 测试结果处理
- `DbAssertionExecutor` - 数据库断言执行
- `TestCase` - 测试用例实体
- `TestAssertion` - 断言实体
- `TestResult` - 测试结果实体

---

### 7. 报告模块

**职责：** 生成各种分析报告

**主要组件：**
- `ReportController` - HTTP 接口
- `ReportingService` - 报告生成服务
- 报告类型：
  - 迁移就绪度报告
  - 置信度趋势报告
  - 测试覆盖率报告
  - 图谱质量报告

---

### 8. 向量检索模块

**职责：** 文档片段向量化，语义相似性检索

**主要组件：**
- `VectorController` - HTTP 接口
- `VectorizationService` - 向量化
- `VectorRetrievalService` - 向量检索
- `VectorDocument` - 向量文档实体

**实现：** 使用 pgvector 存储向量，余弦相似度计算

---

### 9. 系统管理模块（新增）

**职责：** 用户管理、字典管理、系统配置

**主要组件：**
- `SystemController` - 完整 HTTP 接口
- `SysUserService` - 用户管理
- `SysDictService` - 字典管理
- `SysConfigService` - 系统配置
- `AuditLogService` - 审计日志查询

---

### 10. 审计日志模块（新增）

**职责：** 记录用户操作，支持安全审计

**实现：**
- `@Log` 注解标记需要记录的方法
- `LogAspect` AOP 切面自动拦截
- 提取请求信息、参数、耗时、异常
- 保存到 `sys_operation_log` 表
- 前端支持分页查询和详情查看

## 🔐 安全架构

### 认证流程

```
用户登录 → 验证用户名密码 → 生成 JWT Token → 返回给前端
    ↓
前端每次请求在 Header 携带 Token
    ↓
JwtAuthenticationFilter 拦截 → 解析 Token → 验证 → 设置 Security Context
    ↓
Controller 处理请求
```

### 密码存储
- 使用 BCrypt 加密存储
- 默认 admin 用户密码 `admin123`（仅开发环境）

### 敏感信息处理
- LogAspect 自动过滤密码、token、密钥等敏感参数
- 不记录日志

### CORS 配置
- Spring Security CORS 配置
- 支持前端跨域访问

## 🚀 异步处理

### 线程池配置

**3 个专用线程池：**
1. `taskExecutor` - 通用任务
2. `ioTaskExecutor` - IO 密集型任务
3. `testExecutor` - 测试执行任务

**拒绝策略：** `CallerRunsPolicy` - 调用者执行，避免任务丢失

**优雅关闭：** 等待任务完成再退出

## 📈 可扩展性设计

### 新增 Agent
1. 在 `io.github.legacygraph.llm.agent` 包新建 Agent 类
2. 实现 Agent 接口
3. 在 `LlmGateway` 中注册
4. 前端添加调用入口即可

### 新增扫描器
1. 实现扫描器接口
2. 在扫描流程中添加类型判断
3. 无需修改核心流程

### 新增报告类型
1. 在 `ReportingService` 添加新的生成方法
2. 前端添加展示页面即可

## 📝 部署架构
