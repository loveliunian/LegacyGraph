### 0.6 CALLS 边恢复方案

**已实施**：扩 `JavaMemberCallResolver` 为双路径模式：

| 路径 | 适用场景 | 逻辑 |
|------|---------|------|
| 正向（原有） | targetClass 已由 AST 解析（注解注入） | god-node 精确匹配 |
| **反向（新增）** | targetClass 为 null（XML 配置注入） | calledMethod 名全局搜索 → god-node(恰好1) → 同包消歧 |

反向匹配置信度 0.85 + PENDING_CONFIRM（略低于正向的 1.0 + CONFIRMED，确保人工可审核）。

**预期效果**：480 条 SERVICE_CALL 中，calledMethod 在全局 Method 索引中唯一命中的调用将被解析为 CALLS 边。对于保证金项目（1,038 个 Method 节点），同名方法占比取决于代码风格——getById/findById 等通用名可能歧义跳过，insert*/update*/delete* 等业务方法名大概率唯一命中。

#### 1.2.1 CALLS 边恢复方案（替代 ServiceCallExtractor 修补）

当前 `ServiceCallExtractor` 的 AST 级别注入推断对注解项目有效，但对 XML 配置项目（如保证金）完全失效。正确路径不是继续修补 AST 推断，而是：

1. **扩 `JavaMemberCallResolver`**：在全局索引阶段，对每个 `SERVICE_CALL` fact（无论 targetClass 是否为 null），利用事实中的 `callerClass` + `calledMethod` + 全局类/方法索引做反向匹配：
   - 若 `calledMethod` 在全局惟一定义→直接解析
   - 若多个同名方法→利用调用方 FQN 的 import/同包做消歧
2. **降低对文件级注入推断的依赖**：`SERVICE_CALL` fact 只需记录"谁调了哪个方法"，不要求 targetClass 已解析；全局解析阶段负责消歧。

### 8.4 架构层面补充建议

- **A. 解析失败分类学**：为未解析调用引入失败原因枚举（`MISSING_TARGET` / `AMBIGUOUS` / `UNRESOLVED_SYMBOL` / `DYNAMIC_DISPATCH`），落到证据属性上，使审计能定位"漏在哪一步"，而非只看 resolved=0。
- **B. 证据链闭环**：每条推断边保留其证据（调用点原文、消歧依据、置信度、来源事实 ID），支持人工从边反查到源码位置，形成可审计闭环（对标 Graphify 的 EXTRACTED/INFERRED/AMBIGUOUS 分级）。
- **C. 多语言分层抽象**：抽取器分「语言 AST 层（Tree-sitter/JavaParser）→ 统一 IR 层 → 图谱构建层」三层，为后续接入 Go/Python/TS 抽取器复用同一 resolver 与消歧逻辑。
- **D. 版本间 diff 能力**：持久化每次扫描快照，支持"两次扫描间新增/删除了哪些边/节点"的结构化对比（当前 §0 靠人工对比），为增量扫描和回归门禁提供基础。
