## 🔴 严重 (Critical) — 16 项

### S1. 命令注入：testRepoConnection 未校验 Git URL ✅ 已修复
- **文件：** `backend/src/main/java/io/github/legacygraph/controller/SourceController.java:304`
- **问题：** `testRepoConnection` 直接使用数据库中的 `repo.getGitUrl()` 调用 `ProcessBuilder`，而同文件的 `testRepoUrl`（356 行）调用了 `isValidGitUrl()` 校验。两者逻辑不一致。
- **攻击路径：** 攻击者在创建仓库时存入恶意 URL（如 `ext::sh -c 'id > /tmp/pwned'`），然后调用 `/repos/{id}/test-connection`，git 的 `ext::` 协议会执行任意命令。
- **修复方案：** 添加 gitUrl 白名单校验（`^[a-zA-Z0-9@:/._~?&=%#+-]+$`），拒绝非法字符。

### S2. JDBC URL 注入：buildJdbcUrl 未过滤用户输入 ✅ 已修复
- **文件：** `SourceController.java:809-824`
- **问题：** `host`、`databaseName` 直接从 `DbConnection` 实体拼接进 JDBC URL，无任何过滤。攻击者可在 `databaseName` 中注入 JDBC 参数：`mydb?allowLoadLocalInfile=true&autoDeserialize=true`，导致 MySQL 服务端读取服务器本地文件（LOAD DATA LOCAL INFILE 攻击）。
- **修复方案：** 添加 JDBC URL 白名单校验（仅允许 `jdbc:postgresql/mysql/mariadb/oracle/sqlserver` 协议），host 和 dbName 正则校验。

### S3. HTTP 响应头注入：Content-Disposition 中的文件名未转义 ✅ 已修复
- **文件：** `SourceController.java:1233-1234`
- **问题：** `doc.getDocName()` 来自用户上传时的 `originalFileName`，直接拼入 HTTP Header。若文件名含 `"\r\n"` 或 `"`，可注入任意 HTTP 头（CRLF Injection）。
- **修复方案：** 使用 `HttpHeaders.setContentDispositionFormData()` 方法安全设置响应头，自动处理编码。

### S4. 文件上传路径遍历 ✅ 已修复
- **文件：** `SourceController.java:927`
- **问题：** `String fileName = UUID.randomUUID() + "_" + originalFileName;` — 虽然 UUID 前缀有一定缓解作用，但 `originalFileName` 可包含 `../` 等路径遍历字符。如果文件名中有 `/`，`Paths.get(uploadDir, fileName)` 会将其解释为子目录分隔符。
- **修复方案：** 使用 `UUID + 原始文件名` 生成安全路径，并验证 `normalizedPath.startsWith(uploadDir)`。

### S5. 明文密码硬编码在 git 仓库 ✅ 已修复

### S9. @Transactional 内 CompletableFuture 事务失效（ProjectService） ✅ 已修复

### M4. ConcurrentHashMap 锁清理存在竞态条件 ✅ 已修复
- **文件：** `EvidenceGraphWriter.java:148-153`
- **问题：** `unlock()` 后检查 `!lock.hasQueuedThreads()` 再 `remove`。在高并发下，`unlock()` 和 `remove()` 之间另一个线程可能 `computeIfAbsent` 拿到新锁，导致两个线程同时持有不同锁执行 MERGE。
- **修复建议：** 使用 Striped Lock（Guava）或固定大小的锁池，避免动态创建/销毁锁的竞态问题。
- **执行结果：** ✅ 改用固定大小 Striped Lock（`Striped.lazyWeakReadWriteLock(64)`），消除动态创建/销毁锁的竞态窗口。编译验证通过。

### M5. @Transactional 与手动 ReentrantLock 组合风险 ✅ 已修复
- **文件：** `EvidenceGraphWriter.java:101-154`
- **问题：** `upsertNode` 标注 `@Transactional`，锁在事务内持有。如果 Neo4j 操作耗时（网络延迟），锁被长时间占用。事务提交在锁释放之后（Spring AOP 代理在方法返回后提交），存在窗口期。

- **执行结果：** ✅ 移除 `upsertNode` 方法的 `@Transactional`，锁释放后再手动开启事务提交 PG 写入。Neo4j MERGE 与 PG 写入拆分为独立步骤。编译验证通过。
