## 2. 高严重度堵点（17 项，按链路顺序）

### 2.1 扫描调度

| ID | 堵点 | 证据 | 修复方案摘要 |
|---|---|---|---|
| H-S01 | 全量扫描时无版本锁，并发触发会互踩 `scan_version` | `task/ProjectScanner.java`、`entity/ScanVersion.java` | 引入 `scan_lock` 行级锁（PG advisory lock），同 project 同时只允许 1 个 active 扫描 |
| H-S02 | 增量扫描断点 `scan_checkpoint` 与 `file_snapshot` 状态不一致 | 新增 `ScanCheckpoint` 实体但消费侧未串接 | 把 checkpoint 提升为扫描主流程状态机唯一来源 |

### 2.2 源码落库

| ID | 堵点 | 证据 | 修复方案摘要 |
|---|---|---|---|
| H-D01 | `file_snapshot` 与 `source_snapshot` 双轨制导致同一文件存两份 | 新增 `SourceSnapshot` 实体与 `ScanCheckpointRepository` | 收口为 `source_snapshot` 单轨，`file_snapshot` 仅作压缩归档 |
| H-D02 | `parse_failure` 写库但前端无查询入口 | `entity/ParseFailure.java` + `repository/ParseFailureRepository.java` 新增未挂服务 | 在 `SourceController` 暴露失败列表 + 重试入口 |
| H-D03 | 密码字段明文风险（旧版） | `V79__db_connection_password_cipher.sql` 已落库 | `LocalAesGcmSecretCipher` 已就位，需把 `DbConnection` 读路径切到 `SecretCipher` |

### 2.3 抽取层

| ID | 堵点 | 证据 | 修复方案摘要 |
|---|---|---|---|
| H-E01 | `JavaMemberCallResolver` 解析 lambda / 方法引用时丢调用边 | `builder/JavaMemberCallResolver.java` 改动 | 引入方法句柄符号表 + lambda 捕获变量追踪 |
| H-E02 | `JavaStructureExtractor` 注解驱动 Bean（Spring / MyBatis）抽取不全 | `extractors/JavaStructureExtractor.java` 改动 | 维护注解→节点类型映射表，先实现 `@Service` / `@Repository` / `@Mapper` |
| H-E03 | `ServiceCallExtractor` 无法识别 Feign / Dubbo / HTTP 调用 | `extractors/ServiceCallExtractor.java` | 接入 `ConcurrencyPropertyExtractor` 的方法调用图做调用边归一 |
| H-E04 | 并发原语（`synchronized` / `Lock` / `CompletableFuture`）未入图 | 新增 `ConcurrencyPropertyExtractor.java` | 已落地，需要把抽取结果写回 Neo4j `Concurrency` 节点 + 边 |

### 2.4 建图层

| ID | 堵点 | 证据 | 修复方案摘要 |
|---|---|---|---|
| H-G01 | `PgEvidenceTxExecutor` 长事务导致锁竞争 | `builder/PgEvidenceTxExecutor.java` 改动 | 拆分为"建图小事务 + 异步对账"两阶段 |
| H-G02 | `Neo4jGraphDao` 写入失败无幂等键，重跑产生重复节点 | `dao/Neo4jGraphDao.java` | 引入 `scan_version + node_key` 唯一约束，先 merge 再 upsert |
| H-G03 | `Neo4jSchemaRepository` 约束变更未在低峰期执行 | `dao/Neo4jSchemaRepository.java` | 启动期只做约束补齐，索引创建走后台 job |

### 2.5 图谱查询

| ID | 堵点 | 证据 | 修复方案摘要 |
|---|---|---|---|
| H-Q01 | 路径查询（`GraphPathReadModel`）深度无上限导致超时 | `service/graph/GraphPathReadModel.java` | 加 depth 软上限（默认 8）+ 超时熔断 |

### 2.6 前端展示

| ID | 堵点 | 证据 | 修复方案摘要 |
|---|---|---|---|
| H-U01 | `GraphViewerOptimized.vue` 万级节点仍卡顿 | `components/graph/GraphViewerOptimized.vue` | 接 L7 聚合 + LOD，节点 > 3000 自动收敛 |

### 2.7 QA 问答

| ID | 堵点 | 证据 | 修复方案摘要 |
|---|---|---|---|
| H-A01 | `RetrievalIntentRouter` 路由后未取证据节点，QA 直接走 LLM 幻觉 | 新增 `service/retrieval/RetrievalIntentRouter.java` | 强制走图谱证据回填，prompt 模板带节点 ID 引用 |
| H-A02 | `QaFeedbackService` 反馈未回写到证据权重 | 新增 `service/feedback/QaFeedbackService.java` | 把 👍 / 👎 写回 `evidence_score`，后续检索加权 |

## 3. 中 / 低严重度堵点（17 项）

### 3.1 扫描调度
- **M-S01**：调度器 cron 解析时区漂移 → 统一存 UTC
- **M-S02**：单步任务 `DocExtractStep` / `FeatureMappingStep` 失败无自动重试
- **L-S01**：扫描进度事件未持久化，刷新页面丢失

### 3.2 源码落库
- **M-D01**：`ScanVersion` 累计统计字段未刷新（`V82__scan_version_cumulative_stats.sql` 待消费）
- **M-D02**：`SourceSnapshotService` 缺少按 project + branch 的幂等 upsert
- **L-D01**：失败原因字段长度不足

### 3.3 抽取层
- **M-E01**：解析策略 `DocumentParsingRouter` 分发优先级硬编码
- **M-E02**：OCR / Layout / Fast 三策略未做超时回退链
- **L-E01**：抽取结果未带定位 span，前端无法跳转源码

### 3.4 建图层
- **M-G01**：`Neo4jProjectionRepository` 投影视图对中文字段名未建索引
- **M-G02**：`Neo4jQueryRepository` 复杂查询无 prepared statement 缓存

### 3.5 图谱查询
- **M-Q01**：`GraphQueryService` 缓存 key 未按 scan_version 隔离
- **M-Q02**：`ImpactAnalysisService` 影响面分析未分层（直接/传递）
- **L-Q01**：ACL 过滤 `AclFilterService` 性能采样缺失

### 3.6 前端展示
- **M-U01**：`UnifiedGraph` 切换三类图谱时全量重渲染
- **L-U01**：版本对比 `GraphDiff` 颜色对比度不足

### 3.7 Wiki
- **M-W01**：文档生成定时任务未触发（缺 `WikiGenerateJob`）
- **L-W01**：文档与图谱状态未双向校验

### 3.8 QA 问答
- **M-A01**：`QaEvaluationView` 评估指标未落库

## 6. 验收口径

每条堵点修复后必须满足：
1. 对应测试用例新增（`backend/src/test` 增量 ≥ 1 个用例）
2. 前端若有改动，需在 `frontend/src/views` 对应页面回归截图
3. `doc/项目升级计划/扫描到展示全链路堵点修复速查.md` 同步勾选该条

## 7. 风险与回滚

- H-D01 收口双轨制期间需保留 `file_snapshot` 只读窗口 7 天
- H-G01 拆分事务需配套回滚脚本（`V__neo4j_orphan_cleanup.sql` 待补）
- H-A01 强制证据回填可能让"闲聊型" QA 命中率下降，需保留意图白名单
