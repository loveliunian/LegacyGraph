# 外部工具对照校验与图谱增强调研方案

> 基于 LegacyGraph 代码库现状（2026-07-09），分析当前两套插件/工具系统的实际运行状态，提出将外部工具接入扫描主流程、与本地抽取器交叉验证、提升图谱准确性和全面性的可行路径。

---

## 1. 现状诊断

### 1.1 当前两套"插件"系统对比

| 维度 | PluginRegistry（元数据注册表） | ToolRegistry（工具执行框架） |
|------|-------------------------------|------------------------------|
| **代码位置** | `io.github.legacygraph.plugin` | `io.github.legacystanding.tool` |
| **设计意图** | 插件元数据注册与发现 | 外部工具（MCP/CLI/本地）的编排调用 |
| **内置数量** | 24 个（5 Scanner + 9 Agent + 3 Tool + 4 GraphView） | 3 个（McpCodeGraphAdapter + CliToolAdapter + LocalFallbackAdapter） |
| **扫描主流程调用** | ❌ 从未被调用 | ⚠️ 仅在 `AI_CODE_UNDERSTANDING` 增强步骤，且默认关闭 |
| **MCP 外部集成** | 仅在元数据中描述 | ✅ `McpCodeGraphAdapter` 已实现 `searchGraph`/`traceCall`/`readSnippet` |
| **实际生产效果** | 零 | 零（`enhancementEnabled=false` + `enableAi=false` 双重门控） |

### 1.2 核心问题

**问题一：PluginRegistry 是"橱窗模型"**

- 24 个插件在启动时注册到内存 Map，前端 `/lg/plugins` 可查询列表
- 但 `ProjectScanner`、`AdapterExecutionService`、`GraphBuilder` 等扫描核心链路**从未引用** `PluginRegistry`
- 效果等价于"有菜单但后厨不看"——插件存在但不参与扫描

**问题二：ToolRegistry 是"增强侧枝"**

- `McpCodeGraphAdapter` 已完整实现 MCP 协议调用（`indexStatus` → `searchGraph` → `queryGraph`）
- 但调用路径被限制在 `AI_CODE_UNDERSTANDING` 步骤（order=8，状态机 ENHANCE 阶段）
- 该步骤受两道门控：
  1. `AiScanConfig.enableAi`（默认 false）
  2. `legacygraph.understanding.scan-enhancement.enabled`（默认 false）
- 即使开启，也只是"扫描后增强"（对已有图谱做深度分析），**不参与图谱构建本身**

**问题三：无"外部工具对照校验"机制**

- 当前扫描结果（节点/边）完全由本地抽取器（`JavaCodeAdapter`、`VueRouteAdapter`、`MyBatisXmlAdapter` 等）生成
- 没有"用外部工具验证本地结果"或"用外部结果补充本地盲区"的闭环
- 导致已发现的缺口（如 CALLS=0、BusinessDomain 100% 孤立）只能靠人工代码修复，而非外部工具自动补全

## 2. 目标定义

### 2.1 短期目标（1-2 周）

将外部工具从"增强侧枝"改造为"扫描验证节点"，在扫描主流程中增加对照校验点：

```
扫描主流程：
  ADAPTER_SCAN → 本地抽取器生成图谱
     ↓
  EXTERNAL_VERIFY → 外部工具对照校验（新增）
     ↓
  GRAPH_BUILD → 融合结果写 Neo4j
```

### 2.2 中期目标（1 个月）

建立"主建 + 外验 + 交叉仲裁"的三层机制：

- **主建层**：本地 ExtractionAdapter（快速、确定性强）
- **外验层**：外部工具对照（高召回、覆盖盲区）
- **仲裁层**：结果比对与置信度加权（解决冲突、去重、补漏）

### 2.3 长期目标（2-3 个月）

形成可插拔的外部工具生态：

- 新增工具只需实现 `ExternalVerificationAdapter` 接口
- 工具结果自动进入图谱融合引擎
- 支持工具间的交叉验证（如 SonarQube 与 Joern 结果互校）

## 3. 外部工具分类调研

### 3.1 工具分类全景

| 类别 | 代表工具 | 能力 | 与 LegacyGraph 的互补性 |
|------|---------|------|------------------------|
| **代码图谱引擎** | Joern、Sourcegraph、Tree-sitter CodeGraph | CPG/控制流/数据流/跨语言符号索引 | LegacyGraph 缺 CFG/DDG 深度分析，可用外部引擎补 |
| **MCP 代码记忆** | codebase-memory MCP Server | 符号搜索、调用链追踪、代码片段读取 | 已有 `McpCodeGraphAdapter`，但只在增强步骤调用 |
| **静态分析平台** | SonarQube、Checkstyle、PMD、SpotBugs | 代码质量规则、坏味道、漏洞、重复代码 | 可提取"方法复杂度""重复代码块"作为图谱属性 |
| **依赖分析** | OWASP Dependency-Check、Snyk、Maven Dependency Plugin | 第三方依赖漏洞、版本冲突、传递依赖 | LegacyGraph 不分析 pom.xml 依赖，可补全 |
| **AST 增强** | JavaParser SymbolSolver（外部实例）、Eclipse JDT | 精确类型解析、跨文件符号链接、重构级分析 | 当前 SymbolSolver 受限于单文件，外部实例可全项目索引 |
| **AI 语义** | OpenAI Code Interpreter、Claude Code、Copilot | 自然语言理解代码意图、生成文档 | 可作为高阶验证层，校验本地抽取的业务概念 |

### 3.2 各工具详细调研

#### 3.2.1 Joern（Code Property Graph）

**能力**：
- 将代码编译为 CPG（控制流图 + 数据流图 + AST + 程序依赖图）
- 支持 C/C++、Java、JavaScript、Python
- 提供 Scala-based 查询语言（类似 Cypher）

**与 LegacyGraph 对照点**：
- Joern 的 `CALL` 边 → 验证 LegacyGraph 的 `CALLS` 边（当前为 0）
- Joern 的 `INHERITS_FROM` → 验证 `EXTENDS`/`IMPLEMENTS` 边
- Joern 的 `REACHING_DEF` → 补充 LegacyGraph 缺失的数据流边

**接入方式**：
- CLI 调用：`joern-parse` → `joern-export` → 解析 CPG JSON
- 或启动 Joern Server（HTTP API），通过 `McpClientFacade` 类似方式调用

**预期效果**：
- CALLS 边从 0 → ≥50（Joern 对 Java 的调用解析非常成熟）
- 新增 DFG（数据流）边类型

#### 3.2.2 Sourcegraph（开源代码搜索）

**能力**：
- 基于 Zoekt（Google 代码搜索算法）的全文索引
- 跨仓库符号搜索、精确跳转定义、引用查找
- 支持 50+ 语言

**与 LegacyGraph 对照点**：
- Sourcegraph 的 `search` API → 验证本地 `Method` 节点是否完整
- `hover` API → 获取方法签名和文档注释，校验本地签名归一化是否正确
- `references` API → 发现本地抽取遗漏的跨文件引用

**接入方式**：
- Sourcegraph 提供 GraphQL API
- 或自托管 Sourcegraph 实例，通过 HTTP 调用

**预期效果**：
- 发现本地 JavaParser 解析失败的文件（Java 8+ 新语法、Lombok 宏等）
- 补全跨文件调用的 targetClass（解决 XML 配置项目 CALLS=0）

#### 3.2.3 Tree-sitter CodeGraph

**能力**：
- 基于 Tree-sitter parser 的通用 AST 解析
- 生成跨语言的代码图谱（支持 Go、Rust、Python、TS 等）
- 轻量级、无需编译（与 Joern 相比）

**与 LegacyGraph 对照点**：
- LegacyGraph 目前只支持 Java/Vue，Tree-sitter 可补全 Python/Go/TS 后端
- 提供统一的 AST → 图谱转换格式

**接入方式**：
- CLI 调用生成 JSON 图谱 → 本地解析融合

#### 3.2.4 SonarQube

**能力**：
- 代码质量度量（圈复杂度、代码重复、测试覆盖率）
- 规则引擎（Bug、漏洞、坏味道）
- 提供 Web API 查询分析结果

**与 LegacyGraph 对照点**：
- SonarQube 的 "Issues" → 可作为图谱节点的 `properties`（如方法复杂度）
- SonarQube 的 "Duplications" → 可生成 `SIMILAR_TO` 边
- SonarQube 的 "Measures" → 可作为节点置信度的加权因子

**接入方式**：
- SonarQube Web API（`api/issues/search`、`api/measures/component`）
- 或自托管 SonarQube Community Edition

#### 3.2.5 codebase-memory MCP Server

**现状**：已有 `McpCodeGraphAdapter` 实现，但只在增强步骤调用。

**改造方向**：
- 将 MCP 调用从"增强后分析"前移到"扫描中验证"
- 具体：在 `JavaMemberCallResolver` 之后，增加 `McpVerificationStep`：
  - 对本地解析为 CALLS 的边，用 MCP `traceCall` 验证
  - 对本地解析失败的 SERVICE_CALL fact，用 MCP `searchGraph` 查找目标
  - 对本地未发现的边，用 MCP `queryGraph` 补充

## 4. 集成方案设计

### 4.1 架构设计

```
┌─────────────────────────────────────────────────────────────────────┐
│                        扫描主流程（ProjectScanner）                    │
│                                                                     │
│  ┌──────────────┐    ┌──────────────┐    ┌──────────────┐          │
│  │ AdapterExecution │ │ ExternalVerify │ │ GraphBuilder   │          │
│  │ Service（本地）   │ │ Service（外验）  │ │（融合写图）     │          │
│  └──────┬───────┘    └──────┬───────┘    └──────┬───────┘          │
│         │                    │                    │                 │
│         ▼                    ▼                    ▼                 │
│  ┌──────────────┐    ┌──────────────┐    ┌──────────────┐          │
│  │ JavaCodeAdapter│   │ JoernAdapter   │   │ EvidenceGraph │          │
│  │ VueRouteAdapter│   │ SonarAdapter   │   │ Writer（MERGE）│         │
│  │ MyBatisAdapter │   │ McpVerifyAdapter│   │               │         │
│  │ ...            │   │ ...            │   │               │         │
│  └──────────────┘    └──────────────┘    └──────────────┘          │
└─────────────────────────────────────────────────────────────────────┘
                              │
                              ▼
                    ┌──────────────────┐
                    │ ResultFusionEngine │
                    │ （结果比对与仲裁）  │
                    └──────────────────┘
```

### 4.2 新增核心组件

#### 4.2.1 ExternalVerificationAdapter 接口

```java
/**
 * 外部验证适配器 —— 在本地抽取完成后，用外部工具对照校验。
 * 与 ExtractionAdapter 的区别：前者"建图"，后者"验图"。
 */
public interface ExternalVerificationAdapter {
    /**
     * 是否支持验证此版本（根据项目语言、框架、可用性判断）
     */
    boolean supports(ScanContext context);

    /**
     * 执行验证，返回外部工具发现的节点/边/差异。
     */
    VerificationResult verify(String projectId, String versionId, ScanContext context);

    /**
     * 健康检查（外部工具是否可达）
     */
    boolean checkHealth();
}
```

#### 4.2.2 VerificationResult 结构

```java
public class VerificationResult {
    // 外部工具确认存在的边（用于验证本地结果）
    private List<VerifiedEdge> confirmedEdges;
    // 外部工具发现但本地缺失的边（补漏）
    private List<VerifiedEdge> missingEdges;
    // 本地存在但外部工具未发现的边（可能误报）
    private List<VerifiedEdge> suspiciousEdges;
    // 外部工具独有的节点属性（如 SonarQube 复杂度）
    private List<VerifiedNodeProperty> nodeProperties;
}
```

#### 4.2.3 ResultFusionEngine（结果融合引擎）

```java
@Service
public class ResultFusionEngine {
    /**
     * 融合本地结果与外部验证结果。
     * 策略：
     * 1. 双方都确认的边 → confidence=1.0, status=CONFIRMED
     * 2. 本地有但外部无 → confidence=0.6, status=PENDING_CONFIRM（标记待审）
     * 3. 外部有但本地无 → confidence=0.85, status=PENDING_CONFIRM（高置信外部结果）
     * 4. 双方冲突 → 按工具优先级仲裁（如 Joern > 本地 > SonarQube）
     */
    public FusionResult fuse(LocalGraph local, List<VerificationResult> externalResults);
}
```

### 4.3 对照校验机制

#### 4.3.1 边级对照（Edge-by-Edge Verification）

| 本地边 | 外部边 | 融合策略 |
|--------|--------|---------|
| ✅ 有 | ✅ 有 | 确认，confidence=1.0 |
| ✅ 有 | ❌ 无 | 降级为 PENDING_CONFIRM，置信度 0.6 |
| ❌ 无 | ✅ 有 | 补充为新边，confidence=0.85（外部工具通常更全） |
| ❌ 无 | ❌ 无 | 无操作 |

#### 4.3.2 节点级对照（Node Property Enrichment）

外部工具独有的节点属性（如 SonarQube 圈复杂度）写入 `GraphNode.properties` JSON：

```json
{
  "sonar.complexity": 12,
  "sonar.duplicatedLines": 45,
  "joern.cpgNodeId": 12345,
  "external.tools": ["sonar", "joern"]
}
```

#### 4.3.3 质量门禁（Quality Gate）

扫描完成后，如果外部验证发现：
- 本地 CALLS 边缺失率 > 30% → 扫描标记为 WARNING
- 本地节点遗漏率 > 10% → 扫描标记为 WARNING
- 本地与外部冲突边 > 5% → 扫描标记为 WARNING

### 4.4 与现有 ToolRegistry 的关系

**不重复造轮子，复用已有基础设施**：

1. `McpCodeGraphAdapter` → 改造为 `McpVerificationAdapter implements ExternalVerificationAdapter`
2. `CliToolAdapter` → 改造为 `JoernCliAdapter implements ExternalVerificationAdapter`
3. `ToolRegistry` → 扩展为支持两种 Adapter 类型：`CodeUnderstandingToolAdapter`（增强）+ `ExternalVerificationAdapter`（验证）

## 5. 实施路径

### Phase 0：验证可行性（1 周）

**任务清单**：
1. **Joern 可行性验证**：选 1 个 Java 项目（保证金），用 Joern CLI 生成 CPG，导出 CALLS 边，与 LegacyGraph 结果对比
2. **MCP 前移验证**：将 `McpCodeGraphAdapter` 从增强步骤前移到验证步骤，测试 `searchGraph` 对本地 CALLS=0 的补全效果
3. **SonarQube 验证**：对同一项目跑 SonarQube 分析，导出方法复杂度数据，写入图谱 properties

**验收标准**：
- Joern 能解析出 ≥10 条 LegacyGraph 未发现的 CALLS 边
- MCP `searchGraph` 能补全 ≥5 个 targetClass 未解析的 SERVICE_CALL

### Phase 1：框架搭建（1 周）

**任务清单**：
1. 新建 `ExternalVerificationAdapter` 接口
2. 新建 `ResultFusionEngine` 融合引擎
3. 改造 `McpCodeGraphAdapter` → `McpVerificationAdapter`
4. 在 `ProjectScanner` 中增加 `EXTERNAL_VERIFY` 步骤（位于 ADAPTER_SCAN 之后、GRAPH_BUILD 之前）
5. 新增 `GraphCompletenessAuditService` 指标：外部验证覆盖率

**验收标准**：
- 编译通过，新增步骤不影响现有扫描流程
- `EXTERNAL_VERIFY` 步骤默认可跳过（向后兼容）

### Phase 2：多工具接入（2 周）

**任务清单**：
1. **JoernAdapter**：Joern CLI 调用 → CPG JSON 解析 → VerificationResult
2. **SonarAdapter**：SonarQube API 调用 → Issue/Measure 提取 → 节点属性 enrichment
3. **SourcegraphAdapter**（可选）：GraphQL API → 符号搜索 → 补全跨文件引用
4. 融合引擎支持多工具结果仲裁（优先级配置）

**验收标准**：
- 至少 2 个外部工具接入并产出有效结果
- 融合后的图谱 CALLS 边 ≥ 本地结果的 2 倍
- GraphCompletenessAuditService 新增"外部验证覆盖率"指标 ≥ 30%

### Phase 3：生产化（1 周）

**任务清单**：
1. 外部工具健康检查与自动降级（工具不可用时跳过，不阻塞扫描）
2. 配置化开关：`legacygraph.external-verification.enabled` + 各工具独立开关
3. 增量验证：只验证变更文件对应的外部结果（避免全量重跑）
4. 监控告警：外部工具失败率、融合冲突率、补全率

## 6. 风险评估

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| 外部工具调用慢（Joern 编译 CPG 需分钟级） | 扫描总时长增加 | 异步验证、缓存 CPG、增量验证 |
| 外部工具不可达（网络/MCP 服务宕机） | 扫描失败 | 健康检查 + 自动降级 + 超时控制 |
| 结果冲突（本地 vs 外部矛盾） | 图谱不一致 | ResultFusionEngine 仲裁规则 + PENDING_CONFIRM 标记 |
| 外部工具误报（如 Joern 解析泛型失败） | 脏数据入图 | 置信度阈值过滤 + 人工审核机制 |

---

## 7. 预期收益

| 指标 | 当前值 | Phase 2 目标 | 驱动工具 |
|------|--------|-------------|---------|
| CALLS 边 | 0 | ≥50 | Joern、MCP |
| EXTENDS/IMPLEMENTS 边 | 少量 | 翻倍 | Joern |
| 方法节点属性丰富度 | 无 | 复杂度/重复度/漏洞数 | SonarQube |
| 跨文件引用解析率 | <30% | ≥60% | Sourcegraph、MCP |
| 图谱完整性评分 | 无 | ≥70 分 | 多工具交叉验证 |
