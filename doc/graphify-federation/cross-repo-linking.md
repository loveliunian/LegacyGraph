# 跨仓库关联文档

## 概述

当系统接入多个代码仓库时，Graphify 联邦层负责发现和管理跨仓库的关联关系。

## 关联规则

### 共享表关联

当多个仓库使用相同的数据库表时，自动建立关联：

- **检测方式**: 扫描各仓库的 SQL 实体/MyBatis Mapper，提取表名
- **关联类型**: `SHARED_TABLE`
- **示例**: 仓库 A 和仓库 B 都操作 `lg_graph_node` 表

### API 关联

当一个仓库调用另一个仓库暴露的 API 时：

- **检测方式**: 扫描 HTTP Client 调用、Feign 接口、RestTemplate 请求
- **关联类型**: `API_DEPENDENCY`
- **示例**: 仓库 A 的 `UserClient` 调用仓库 B 的 `/api/users`

### Topic/消息关联

当仓库通过消息队列通信时：

- **检测方式**: 扫描消息生产者/消费者的 topic 名称
- **关联类型**: `SHARED_TOPIC`
- **示例**: 仓库 A 发布 `user.created` 事件，仓库 B 消费该事件

## 候选关系生成

### 生成流程

```
1. 各仓库独立扫描 → 提取共享资源标识
2. 联邦层聚合 → 匹配相同标识
3. 生成候选关系 → 附带置信度
4. 人工确认 → 变为 confirmed 关系
```

### CrossRepositoryLinker

核心组件，负责：

- `findCandidates()`: 发现跨仓库候选关系
- `confirm()`: 确认候选关系
- `reject()`: 拒绝候选关系
- `getImpactChain()`: 查询影响链路

## 确认流程

1. **自动发现**: 系统定期扫描各仓库，发现新的候选关系
2. **通知审核**: 新候选关系进入审核队列
3. **人工确认**: 审核员确认或拒绝候选关系
4. **记录决策**: 记录确认/拒绝原因，供后续规则优化

## 影响分析

确认的跨仓库关系支持影响分析：

- 输入: 变更的表名/API/topic
- 输出: 受影响的仓库列表和影响路径
- 可视化: 影响链路图
