## 📐 分析方法与判定标准

一个操作"值得缓存"的判定：

1. **读多写少**：读频次远高于写，写后失效成本可接受。
2. **计算/IO 昂贵**：DB 全表扫描、Neo4j 多跳/聚合、JDBC 元数据、外部 API、向量嵌入。
3. **结果可变性低**：在两次写之间结果稳定（图谱只读视图、配置、元数据）。
4. **热点路径**：每次请求/每次 LLM 调用都会触发。

> 已落地的场景 1–5 不再重复；下表仅列**新增**机会。所有建议均遵循既有约定：经 `CacheService` 容错、Redis 故障降级回源、写时失效、统一 `lg:` 前缀。

## 🔴 高优先级（热点路径 / 收益显著）

### A1. LLM 默认提供商 `getActiveDefault()` —— 每次调用都全表扫

**证据**：`LlmProviderService.getActiveDefault()`（第 46 行）执行 `llmProviderRepository.findAll()` 全表加载后在内存筛选；而 `LlmGateway.callWithTemplate` 在**每次** LLM 调用中调用它两次（第 82、147 行）。AI 编排一次扫描会触发数十~上百次 LLM 调用，等于数十~上百次全表扫。

**方案**：`@Cacheable(cacheNames="llm-provider-default")` 缓存默认提供商；`save/setDefault/toggleActive/delete` 时 `@CacheEvict(allEntries=true)`。

**收益**：消除 AI 编排主路径上最高频的 DB 查询之一；同时利好 `LlmProviderController.getActiveDefault` 端点。

**风险**：低。配置变更频率极低，全量失效成本可忽略。

> 注：`LlmGateway` 进程内 `chatModelCache` 仍保留（ChatModel 构建昂贵）；缓存的是**配置对象**，与 ChatModel 缓存互补。

---

### A2. Prompt 模板 `getActiveByCode()` —— 每次渲染都查 DB

**证据**：`PromptTemplateService.getActiveByCode` 每次都 `findActiveByCode` 查库；`PromptTemplateLoader.render` 在**每次** `LlmGateway` 调用时执行（DB 优先，未命中才回退 classpath）。

**方案**：`@Cacheable(cacheNames="prompt-template", key="#templateCode")`；模板 CRUD（`create/update/toggleActive/delete`）时 `@CacheEvict`。

**收益**：AI 编排主路径又一个高频 DB 查询被消除。模板内容极少变更。

**风险**：低。注意 `disableCachingNullValues` —— DB 无模板返回 null 时不缓存，回退 classpath 逻辑不受影响。

---

### A3. 项目概览 `ProjectOverviewService.getOverview()` —— 5+ 查询 + Neo4j 聚合

**证据**：`getOverview` 每次执行：3 张表（repo/db/document）`selectList` + `graphStats`（Neo4j 聚合）+ 最近扫描版本 + 最近审核记录。这是**仪表盘首屏**，用户频繁刷新。

**方案**：`@Cacheable(cacheNames="project-overview", key="#projectId")`，TTL 较短（如 30s–2min，仪表盘可容忍轻微陈旧）；在扫描完成、审核确认/驳回、源接入变更时 `@CacheEvict`。

**收益**：仪表盘打开/刷新从 5+ 查询降为 1 次缓存读。

**风险**：中。需在多个写点失效（扫描、审核、源 CRUD）。可复用 `GraphCacheInvalidator` 思路，或直接在相关写点调用 evict。

---

### A4. 数据库 Schema 元数据 `scanSchema`（JDBC `getTables/getColumns`）　❎ 评估后不实现

**证据**：`SourceController.scanSchema`（第 698 行）通过 JDBC `DatabaseMetaData.getTables` 拉取表元数据。

**结论（落地复核）**：该端点本身是**显式刷新写操作**——它会把 `status=SCANNING→READY`、`tableCount`、`lastScanTime` 持久化到 `lg_db_connection` 行。它的语义就是"主动重新扫描"，缓存它会与用途相悖（用户点击刷新却拿到旧结果）。真正的只读元数据获取（`getColumns` 等）只发生在图谱构建期间的 `DatabaseMetadataExtractor`，属低频后台路径。**故不实现缓存**；若未来新增"只读查看 schema"端点，再按 `dbConnectionId` 缓存并提供手动失效。

## 🟠 中优先级

### B1. 验证报告 `GraphValidatorService.getValidationReport()`

**证据**：每次执行 6 次 `countNodes/countEdges`（Neo4j）+ 2 次 TestResult `count` + `calculateOverallConfidence`。验证页频繁打开。

**方案**：`@Cacheable(cacheNames="validation-report", key="#versionId")`；在 `confirm/reject node/edge`、`updateConfidenceByTestResults` 后 `@CacheEvict`（这些写点已接入 `GraphCacheInvalidator`，可顺带失效）。

**收益**：验证页加载从 6+ Neo4j count 降为缓存读。

**风险**：低。失效点已存在。

---

### B2. 迁移就绪度报告 `generateMigrationReport(projectId)` —— 项目级、未缓存

**证据**：`ReportingService.generateMigrationReport`（第 82 行）调用 `graphStats`+`nodeTypeStats`+`queryLowConfidenceNodes`+`queryDisconnectedNodes`，是重聚合。**当前场景 4 只缓存了 versionId 限定的四类报告，未覆盖此项目级报告**（它只接收 projectId）。

**方案**：`@Cacheable(cacheNames="report-migration-readiness", key="#projectId")`；图谱写入时随 `evictReportCaches` 一并失效。

**收益**：补齐报告缓存的最后一块缺口。

**风险**：低。

---

### B3. 向量检索结果 `semanticSearch` / `findSimilarNodes`

**证据**：`VectorRetrievalService.semanticSearch` 每次都对 query 调 `embeddingModel.embed`（外部/模型调用）+ pgvector 检索。`QaAgent` 与 `VectorController` 高频调用。相同问题会被反复问。

**方案**：按 `versionId + sha256(query) + topK + chunkType` 缓存 Top-K 结果，短 TTL（如 1–5min，向量库会随扫描更新）。

**收益**：热门问题复用 embedding 与检索结果，QA 提速、降 embedding 成本。

**风险**：中。**时效性**——新扫描/向量化后旧结果陈旧；需在 `VectorizationService.embedAndStore` / 重新扫描时按 version 失效。已在原方案列为场景 9，此处确认为中优先级。

---

### B4. 节点详情 `findNodeById` —— 图谱交互高频单点查

**证据**：`findNodeById` 被 `LlmAgentController`（合并决策）、`GraphQueryController`（合并候选）、`GraphValidatorService`（确认/驳回）、`VectorRetrievalService`（QA 邻域）等多处调用。图谱节点在写之前稳定。

**方案**：按 `nodeId` 缓存 `GraphNode`，短 TTL 或写时失效；`updateNode/deleteNode/合并/审核` 时失效对应节点。

**收益**：减少 Neo4j 单点查询，利好合并决策、QA 邻域扩展。

**风险**：中。**失效粒度细**——需在所有写点精确失效；遗漏会导致陈旧。建议 TTL 取短（如 30s）作为兜底，配合写时 evict。

## 🟡 低优先级 / 谨慎评估

### C1. 用户信息 `SysUser`（按 username / id）

**证据**：`AuthController.getCurrentUser/refresh` 按 username 查用户；`JwtAuthenticationFilter` 目前只解析 token 不查库（不构成热点），但若未来从 DB 校验权限则会成为每请求热点。

**方案**：`getById/getByUsername` `@Cacheable`，写（更新/禁用/删除/登录更新）时 `@CacheEvict`。

**收益**：用户量不大时收益有限；**主要价值在于为"强制下线/改密即时生效"配合 JWT 黑名单**。

**风险**：中。**权限/状态变更必须即时失效**，否则禁用用户仍可用。需与场景 2 的 token-version 机制联动。

---

### C2. 审核待办列表 `listPendingReviews`

**证据**：`ReviewController.listPendingReviews` 分页查询待审核记录，审核页频繁翻页刷新。

**方案**：按 `(projectId, versionId?, status, pageNum, pageSize)` 缓存，短 TTL；审核确认/驳回/批量操作后失效。

**收益**：审核列表翻页提速。

**风险**：中。失效点较多（确认/驳回/批量/编排器生成新任务）；建议短 TTL + 写时 evict。

---

### C3. LLM 调用去重表查询 `findCachedRunIdByInputHash` —— 现已冗余

**证据**：`PromptRunRepository.findCachedRunIdByInputHash` 是为"DB 级 LLM 结果去重"设计的查询，但**从未被调用**；场景 1 已用 Redis（`lg:llm:result:{template}:{inputHash}`）实现了更快的缓存层。

**结论**：**无需再加缓存**。建议评估是否保留该查询——若 DB 级去重（跨重启、跨实例持久化）仍有价值，可作为 Redis 缓存的二级回源；否则可标注为废弃。本项不算"新增缓存机会"，而是"清理冗余设计"。

---

### C4. 审计日志 / 分页列表类

`AuditLogController`、`TestCaseController.list`、`ScanVersion` 列表等分页查询：写入频繁、排序随时间变化、缓存命中率低，**不建议缓存**。仅当出现明确热点时再考虑按查询条件短 TTL 缓存。

## 🚀 推荐落地顺序

1. **A1 + A2**（LLM 主路径双热点）：工作量小、收益最高、风险最低，应在场景 1 之后立即补上——它们和 LLM 缓存同属"AI 编排主路径去 DB 化"。
2. **A3 + B2**（仪表盘 + 迁移就绪度报告）：补齐"项目级/概览"读缓存。
3. **A4**（DB Schema 元数据）：JDBC IO 优化，配合手动刷新端点。
4. **B1**（验证报告）：失效点已就绪，接入成本低。
5. **B3 / B4 / C1 / C2**：按实际热点观测结果择机接入，注意时效性与失效正确性。

## ⚠️ 通用注意事项（适用以上全部）

1. **失效正确性优先于命中率**：宁可短 TTL + 写时 evict 双保险，也不要为追求命中率而漏失效，导致读到陈旧图谱/权限。
2. **复用现有失效器**：图谱相关缓存（A3/B1/B2/B4）应统一接入 `GraphCacheInvalidator`，避免散落的 evict 调用遗漏写点。
3. **null 不缓存**：`disableCachingNullValues` 已生效，DB 查无结果不会缓存，回退逻辑（如模板回退 classpath）不受影响。
4. **时效性场景用短 TTL**：向量检索、节点详情、审核列表等会随业务变更的，TTL 取 30s–5min，作为写时 evict 的兜底。
5. **监控**：接入后建议观察各缓存命中率与平均 TTL，命中率持续偏低的应回退为不缓存（避免无效开销）。

## 💡 一句话总结

在场景 1–5 之上，**最该补的是 AI 编排主路径上的两个 DB 热点**——`getActiveDefault()` 与 `getActiveByCode()`（A1/A2），它们和 LLM 结果缓存同属"让 AI 调用尽量不碰库"的闭环；其次是仪表盘概览（A3）与迁移就绪度报告（B2）这两个重聚合读。其余按热点观测择机接入，注意失效正确性与时效性。
