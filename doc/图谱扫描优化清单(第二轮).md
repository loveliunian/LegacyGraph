## A. 速度优化

### A1. ⭐⭐ 修复 LLM 缓存（最大 win，重扫可省 ~747s）— 诊断进行中

**症状**：Redis 里 `lg:` 前缀键 **0 个**（58 个 key 全是 `sys_config`/`sys_dict`，属其他应用）。`AiScanOrchestrator#cachedExtract` 的 `cacheService.put` 疑似静默失败 → 每次重扫未变文档/代码都重调 LLM。第三次扫同样 10 文档，AI_DOC_EXTRACT 仍 535s、AI_CODE_EXTRACT 212s。

**诊断进展**：
- ✅ **已排除序列化器根因**：`BusinessFactExtractionRedisSerializationTest` 复刻 `RedisConfig.jsonSerializer()` 的 ObjectMapper 配置（`activateDefaultTyping(NON_FINAL, As.PROPERTY)`），对填充数据的 `BusinessFactExtraction`（含 BusinessDomain/Features/Roles 等嵌套）做往返，**通过**（序列化 715 字节，反序列化还原为正确类型）。说明 `put` 静默失败不是序列化失败。
- ✅ **已排除扫描清缓存**：`GraphCacheInvalidator.invalidateVersion` 只清 `graph:`/`report-`/`vec:search:`/`project-overview` 前缀，**不动 `llm:`**。LLM 缓存若写入会保留 7 天。0 键 = 从未写入。
- ⏳ **剩余根因（基础设施层）**：`cacheService` 的 get/put 全部静默失败。两种可能：
  1. **后端主机连不上 Redis**（Spring Data Redis 懒连接，启动不报错，首次命令才失败，被 `CacheService` try/catch 吞）。
  2. **`cacheService` 为 null**（bean 未创建 → `@Autowired(required=false)` 注入 null → `cachedExtract` 直接走 `loader.get()`，连 warn 日志都没有）。

**区分方法**（查后端日志）：
- 若有 `Cache get failed (degrade)` / `Cache put failed (ignored)` warn → 是情况 ①（Redis 连不通，修网络/Redis 连接）。
- 若完全没有这类 warn → 是情况 ②（cacheService null，查 CacheService bean 是否创建、RedisConnectionFactory 是否就绪）。

**收益**：修好后未变内容重扫省 DOC_EXTRACT 535s + CODE_EXTRACT 212s ≈ **747s**，且对增量重扫持续生效。

**配套代码改进（建议）**：`CacheService` 现在对所有失败只 `log.warn(key + message)`、无堆栈、无首因告警，导致整层静默降级无人察觉。建议：① 加启动期 Redis PING 健康检查，连不通时 ERROR 级告警；② 首次失败打堆栈，后续降级为 debug，避免日志洪水的同时保留可观测性。

### A2. 启用 embedding（既提速又提质）
dev 配置 `embedding.enabled: false` → `vectorizeContent` 空转、语义检索不可用。启用后功能映射可升级为向量检索（见 B1）。

### A3. AI_DOC_EXTRACT 大文档截断（535s 主因之一）
`DOC_CONTENT_LIMIT=8000` 字符，大文档截断丢信息，单次 LLM ~170s。可按 `VECTOR_CHUNK_SIZE=2000` 分段并行抽取再合并，既全覆盖又并发提速。

### A4. ADAPTER_SCAN 221s 并行化
549 assets，各 `ExtractionAdapter` 若串行可并行。

### A5. LLM 超时 120s + 3 次重试偏长
单次 120s、失败 3 次+2s 退避最坏 ~366s/call。可降到 60-90s + 指数退避，失败快返。

## B. 质量优化

### B1. ⭐ 功能映射用向量语义检索替代 token 匹配
`BusinessGraphBuilder.mapFeaturesToCode`(`:298`) 用 token 重叠度（API 阈值 0.5），产出 1743 映射但大量低置信（0.466=score×0.7），噪声大。**代码注释 `:315` 明说"后续可进一步替换为向量语义检索提升精度"**。启用 embedding（A2）后用 bge-m3 做语义匹配，阈值可收紧到 0.7+。

### B2. claims 落库不稳定
同样内容重扫 claims 562→453（-109 波动），claim key 生成/去重不幂等。查 `KnowledgeClaimService` 的 key 逻辑，确保同内容重扫 claim 数稳定。

### B3. 知识缺口 836 条过噪
提高缺口阈值或按业务域聚合去重，别淹没审核队列。

### B4. Page 节点仅 1 个
`FrontendGraphBuilder` 产出 Page 太少，Feature→Page `EXPOSED_BY` 映射饿死（EXPOSED_BY 仅 6 条）。查前端页面抽取覆盖。

### B5. httpMethod 硬编码 GET（第一轮点 6，未修）
`runTestGenerate`(`:981`) `req.setHttpMethod("GET")` 对所有接口生成 GET 用例，应读 ApiEndpoint 的 method 属性。

### B6. ⭐ 业务对象↔表/代码 名称匹配盲区（已修）
**症状**：`BusinessGraphBuilderTest.testMapBusinessObjectsToTables_MapsToCodeEntities` 在 baseline 即失败；生产保证金项目 MAPS_TO 边仅 9 条，业务层与数据/代码层大量未连通。
**根因**（真实缺陷，非过期测试）：`tokenSimilarity` 的 `min<2` 守卫一刀切否掉所有单 token 集合匹配，且 `matchTokens` 无复数词干——导致单字业务对象名 `order` 永远匹配不上 `orders`/`OrderService`/`OrderMapper`：
- `order` tokens={order}、`orders` tokens={orders} → 交集 0 → 0 分；
- `order` vs `OrderService` → {order} vs {order,service}，交集 1 但 `min<2` 直接返回 0。
**修复**（`BusinessGraphBuilder.java`）：
1. `matchTokens` 加 `stem()`：粗粒度复数归一（ies→y、ses/xes→去 es、s→去 s，仅 len≥4），`orders`/`services`/`mappers` 折叠为 `order`/`service`/`mapper`。
2. `tokenSimilarity` 放宽 `min<2` 守卫：共享 token 长度≥4 且完全包含时返回 1.0，仍拦截 `nx`/`ab` 类短 token 误匹配（原守卫的本意）。
**验证**：`BusinessGraphBuilderTest` 9/9（原 8/9 失败→全绿）；构建器/agent/llm/cache 共 64 测试全过，无回归。
**收益**：单字领域对象（order/account/user/payment/role）能与对应表/Service/Mapper 建边，业务↔数据/代码层连通度显著提升。部署后重扫可见 MAPS_TO/IMPLEMENTED_BY 边数上升。
