# Graphify 试点说明

## 试点目标

1. **验证 Graphify 集成效果**: 在真实项目中测试 Graphify 的代码分析能力
2. **评估图谱质量**: 验证节点召回率、边召回率和准确率
3. **发现潜在问题**: 识别集成过程中的技术风险和性能瓶颈
4. **积累运维经验**: 建立导入、审核、回滚的标准操作流程

## 环境要求

### 基础设施

- **PostgreSQL**: 13+ (存储图谱数据和审核记录)
- **Neo4j**: 5.x (图数据库，存储节点和边)
- **Redis**: 6.x (缓存和会话管理)
- **MinIO**: 存储 graph.json 原始文件

### 应用配置

```yaml
graphify:
  enabled: true
  command: /opt/graphify/bin/graphify
  max-attempts: 3
  review:
    enabled: true
    auto-pass-threshold: 0.9
```

### 权限配置

- 至少 1 个 `GRAPHIFY_ADMIN` (系统管理员)
- 至少 2 个 `GRAPH_REVIEWER` (图谱审核员)
- 至少 1 个 `GRAPH_EVIDENCE_VIEWER` (证据查看者)

## 执行步骤

### Phase 1: 环境准备 (Day 1-2)

1. ✅ 部署 Graphify CLI 工具
2. ✅ 配置数据库连接
3. ✅ 创建试点项目
4. ✅ 分配用户角色

### Phase 2: 首次导入 (Day 3-5)

1. 运行 Graphify 扫描试点项目
2. 检查 graph.json 生成结果
3. 执行导入作业
4. 验证节点和边导入

### Phase 3: 审核流程 (Day 6-10)

1. 审核低置信度边
2. 测试批量审核功能
3. 验证审核规则引擎
4. 收集审核员反馈

### Phase 4: 查询验证 (Day 11-12)

1. 测试 Agent 查询接口
2. 验证权限控制
3. 检查证据追溯
4. 评估查询性能

### Phase 5: 总结评估 (Day 13-14)

1. 统计 benchmark 指标
2. 编写验收报告
3. 整理问题和建议
4. 制定推广计划

## 验收标准

### 功能验收

| 项目 | 标准 | 权重 |
|------|------|:---:|
| 导入成功率 | ≥ 95% | 20% |
| 节点召回率 | ≥ 90% | 20% |
| 边召回率 | ≥ 85% | 20% |
| 审核效率 | 平均 < 30s/边 | 15% |
| 查询响应时间 | P95 < 2s | 15% |
| 权限控制准确性 | 100% | 10% |

### 质量验收

| 项目 | 标准 |
|------|------|
| 误报率 (forbidden edges) | ≤ 2% |
| 审核队列积压 | < 100 条/天 |
| 导入作业失败率 | ≤ 5% |
| 系统可用性 | ≥ 99% |

### 用户体验

- 审核员满意度 ≥ 4/5
- 开发者查询满意度 ≥ 4/5
- 文档完整性评分 ≥ 4/5

## 试点项目选择标准

### 项目规模

- 代码行数: 10K - 100K
- Java 文件数: 50 - 500
- 依赖模块数: 3 - 10

### 项目特征

- 使用 Spring Boot 框架
- 有明确的模块边界
- 包含数据库操作 (MyBatis/JPA)
- 有 REST API 接口

### 团队配合

- 项目负责人支持试点
- 至少 1 名开发者参与测试
- 至少 2 名审核员参与审核

## 风险与缓解

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| Graphify 工具不稳定 | 导入失败 | 设置重试机制，准备回滚方案 |
| 审核队列积压 | 影响进度 | 增加审核员，优化审核规则 |
| 图谱质量不达标 | 影响查询 | 调整阈值，增加人工审核 |
| 性能问题 | 用户体验差 | 优化查询，增加缓存 |

## 成功指标

试点成功的判定条件（满足所有）：

1. ✅ 所有功能验收项达标
2. ✅ 所有质量验收项达标
3. ✅ 用户体验评分 ≥ 4/5
4. ✅ 无严重级别 (P0/P1) 问题
5. ✅ 团队愿意继续使用
