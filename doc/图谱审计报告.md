## 五、关键链路断点（P0 - 影响核心功能）

### 5.1 Method → Service/Mapper CALLS 边

**问题**: Method 节点无任何 CALLS 出边  
**数量**: 0 条（应为 972 条 Controller Methods + 5173 条 Service Methods）  
**状态**: ✅ **已修复，nodeKey 格式对齐**

#### 根因分析

**代码层面**：
- `JavaServiceCallAdapter.java:43` 显式排除 Controller 文件
- `JavaCodeAdapter.java` 只建了 `ApiEndpoint → Method` (HANDLED_BY) 边，没有提取 CALLS 边

**数据层面**：
- `GraphBuilder.buildServiceCallGraph` 尝试建方法级 CALLS 边时，依赖 `findExistingNode` 查找 Method 节点
- 如果 Method 节点尚未创建（扫描顺序问题）或 `nodeKey` 格式不一致，边创建会静默跳过

#### 修复方案

✅ **已修复 1**: `JavaCodeAdapter.extract()` 增加 `ServiceCallExtractor` 调用，为 Controller 文件提取 CALLS 边
（注：`JavaControllerExtractor` 仅负责 API 端点抽取，不承载 CALLS 提取；CALLS 提取在 `JavaCodeAdapter` 中编排）

✅ **已修复 2**: `nodeKey` 格式对齐，**caller 与 target 双侧**均带参数签名
- caller 侧：`callerMethodSignature = methodName(paramTypes)`（来自方法声明形参）
- target 侧：`calledMethodSignature = methodName(argTypes)`（基于调用实参推断：字面量 / 方法形参 / 注入字段 / 本地变量 / 对象创建 / 强转；不可推断时回退为 null，避免误匹配重载）

**修改文件**:
- `JavaCodeAdapter.java` — 在 `extract()` 中调用 `ServiceCallExtractor` 为 Controller 文件提取 CALLS 边并写图
- `ServiceCallExtractor.java` — 生成 `callerMethodSignature`（形参）与 `calledMethodSignature`（实参推断），归一化规则与 `JavaStructureExtractor` 的 Method nodeKey 一致
- `GraphBuilder.java` — `buildServiceCallGraph` 双侧均用签名拼装 nodeKey（对齐 `类全限定名.方法名(参数类型)`）

**验证**:
- ✅ `mvn compile` 编译通过
- ✅ `mvn test` 测试通过（`ServiceCallExtractorTest` 7 项 + `ServiceCallSignatureTest` 3 项，验证 `calledMethodSignature` 对齐与回退）
- ⏳ 待重新扫描目标项目验证方法级 CALLS 边数量

### 7.1 SqlStatement → Table READS/WRITES 缺失

**问题**: 230 条 SqlStatement（29%）无表关联  
**影响**: 无法完整追踪 SQL 影响的表  
**状态**: ✅ **已修复，增强 SQL 解析和 include 展开**

#### 根因分析

1. **JSqlParser 解析失败**:
   - 复杂 SQL（子查询、CTE、动态表名）无法被 JSqlParser 正确解析
   - MyBatis 动态 SQL 片段（`<if>`、`<where>`、`<foreach>`）语法不完整

2. **include 片段未展开**:
   - MyBatis XML 中 `<include refid="xxx"/>` 引用的 SQL 片段未被展开
   - 导致 SQL 语句不完整，解析失败

3. **表名不匹配**:
   - SQL 中使用别名（`SELECT * FROM user u`）
   - 表名大小写不一致
   - Schema 前缀缺失（`public.user` vs `user`）

#### 修复方案

✅ **已修复 1**: `SqlTableExtractor.extractTables()` 增加正则回退提取
- 当 JSqlParser 解析失败时，使用正则表达式提取 `FROM`、`JOIN`、`INSERT INTO`、`UPDATE`、`DELETE FROM` 后的表名
- 过滤 SQL 保留词（SELECT、WHERE、AND 等）避免误提取

✅ **已修复 2**: `GraphBuilder.buildMapperSqlGraph()` 优先使用 `expandedSql`
- MyBatis XML 解析时，`<include>` 片段会被展开并存储到 `expandedSql` 字段
- 解析 SQL 表关系时优先使用展开后的完整 SQL，而非原始片段

**修改文件**:
- `SqlTableExtractor.java:46-74` — 增加正则回退提取逻辑
- `SqlTableExtractor.java:251-332` — 新增 `extractTablesWithRegex()` 和 `isReservedWord()` 方法
- `GraphBuilder.java:227-228` — 优先使用 `stmt.getExpandedSql()`

**验证**:
- ✅ `mvn compile` 编译通过
- ✅ `mvn test` 测试通过
- ⏳ 待重新扫描目标项目验证 READS/WRITES 边数量提升

### 7.2 Mapper → SqlStatement EXECUTES 缺失

**问题**: 47 个 Mapper（40%）无 SQL 关联  
**影响**: 无法追踪 Mapper 调用的 SQL  
**状态**: ✅ **已修复，支持注解方式 Mapper 和扩展文件匹配**

#### 根因分析

1. **MyBatis XML 扫描覆盖不足**:
   - `MyBatisXmlAdapter.isMapperFile()` 只匹配 `*Mapper.xml`，部分 XML 文件命名不符合规则
   - 某些项目使用 `*Dao.xml` 或 `*Repository.xml` 命名

2. **注解方式 Mapper 未覆盖**:
   - Mapper 方法使用 `@Select`/`@Insert`/`@Update`/`@Delete` 注解而非 XML
   - 当前 `MyBatisXmlExtractor` 仅解析 XML 文件

3. **XML 中 `<id>` 与 Java 方法名不匹配**:
   - 部分项目 XML 定义与 Java 接口不一致

#### 已修复

✅ **已修复 1**: `expandedSql` 优先使用
- `GraphBuilder.buildMapperSqlGraph()` 优先使用展开后的 SQL 解析表关系
- 减少因 `<include>` 片段未展开导致的解析失败

✅ **已修复 2**: 扩展 `isMapperFile()` 匹配规则
- 增加 `*Dao.xml`、`*Repository.xml` 匹配
- 支持更多 MyBatis XML 文件命名约定

✅ **已修复 3**: 支持注解方式 Mapper
- 新增 `MyBatisAnnotationExtractor` 解析 `@Select/@Insert/@Update/@Delete` 注解
- 新增 `MyBatisAnnotationAdapter` 集成到扫描流程
- 覆盖使用注解而非 XML 的 Mapper 方法
