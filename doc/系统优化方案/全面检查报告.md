### 2.3 状态机脱节

**问题 1：`ENHANCE` 状态冲突**
- `AiScanOrchestrator.java:266-272` 串行组执行 KnowledgeGapStep 调 `updateStep(ENHANCE)`，紧接着 UnderstandingEnhancementStep 又调 `updateStep(ENHANCE)` → 前端读 `currentStep` 永远看不到 GAP→UNDERSTANDING 推进过程。

**问题 2：`ScanTask.taskStatus` 是 String（无枚举）**
- 全工程 18+ 字面量 `"RUNNING"/"QUEUED"/"SUCCESS"/"FAILED"/"WARNING"/"SKIPPED"/"REJECTED"/"CANCELLED"`，无编译期拦截。
- **建议**：新增 `common/ScanTaskStatus.java` 枚举 + `completeTask` / `failTask` 工厂方法。

**问题 3：`EXTRACT_FACTS` / `INDEX` / `COMPLETE` 3 个枚举值无 Step 映射**
- 靠 orchestrator 直接写 status 而非由 Step 推进。

### 2.4 事务边界风险

**漏标 @Transactional**（与同类方法不一致）：
- `BusinessGraphBuilder.mapBusinessProcessesToDomains` (line 1107)
- `BusinessGraphBuilder.mapBusinessObjectsToMappers` (line 1170)
- `BusinessGraphBuilder.mapBusinessRulesToRuleNodes` (line 1226)

**Cross-store 一致性**：
- `BusinessGraphBuilder.buildEdgePOJO` 路径（line 1071-1090）不走 `EvidenceGraphWriter` → 批量映射边不写 PG `lg_graph_edge` 表 → 前端 RLS 直接查 PG 看不到这些边。
- 修复：要么 `buildEdgePOJO` 走 `EvidenceGraphWriter`，要么文档明确"批量映射边仅活在 Neo4j 中"。

**mergeEdgesBatch 单批大小无 top-N**：
- `mapBusinessObjectsToMappers` / `mapBusinessRulesToRuleNodes` 是 B×M 双层循环，无 top-N 截断 → 500×300 = 15 万次相似度计算 + Cypher 单批过大 OOM/超时。

## 8. 总结：下一轮实施的 5 大原则

1. **先把孤儿接通再谈优化**：3 个孤儿 + IN_DOMAIN 字面量是 P0 必修，胜过任何性能调优。
2. **命名空间必须统一**：`legacygraph.*` vs `legacy-graph.*` 必须先统一，再补 yml 配置项。
3. **文档与代码同步版本化**：每次 merge 强制跑 `mvn verify` + 文档数字校验脚本，把"文档与代码不一致"作为构建失败项。
4. **状态机原子化**：`ENHANCE` 拆分 + `ScanTaskStatus` 枚举化是后续所有状态显示功能的前提。
5. **QA 答案可信度 > 答案数量**：当前 9.0/10 的关键是抗幻觉双层防御；后续优化要保"宁可拒答不幻觉"原则。
