**优先级**: 🔴 显著  
**预期收益**: 大图首屏响应稳定、QA 召回链路降延迟、写图吞吐提升、减少 Neo4j/Redis/PG 无效往返

> - ✅ §1.1 图谱窗口查询 + 邻域展开 — `GraphQueryService` + `Controller` + `Neo4jProjectionRepository`
> - ✅ §1.2 Neo4j 索引 — `Neo4jIndexInitializer.java`
> - ✅ §1.3 Neo4j 连接池 — `application.yml`

>

## 一、图谱全量接口改为窗口查询与邻域展开

### 1.1 `getUnifiedGraph` 当前按版本全量返回节点和边

**位置**:
- `backend/src/main/java/io/github/legacygraph/controller/GraphQueryController.java:213-231`
- `backend/src/main/java/io/github/legacygraph/service/graph/GraphQueryService.java:329-390`
- `frontend/src/views/graph/UnifiedGraph.vue:599-627`

**代码证据**:
- Controller 文档和实现都是“获取统一图谱全量数据”，只接收 `versionId/minConfidence/statusFilter`，没有 `limit/cursor/nodeTypes/sourceTypes/viewport`。
- `GraphQueryService.getUnifiedGraphUncached()` 并行查询 `queryNodesProjection(...)` 和 `queryEdgesProjection(...)` 后，把完整 `nodes/edges` 放进返回体。
- `UnifiedGraph.vue` 调用 `graphApi.getUnifiedGraph(...)` 后，把全部节点映射为 VueFlow 节点，并用 `Math.random()` 生成初始位置；筛选主要发生在前端 computed 中。

**问题**:
- 大版本图谱会把后端查询、网络传输、浏览器内存、布局计算绑在一次请求里。
- 前端筛选虽然交互灵活，但后端仍已查出全量节点/边，不能降低数据库和网络成本。
- `RiskDetail.vue`、`TestCaseEditor.vue` 等页面为了查单个节点或下拉选项也复用 `getUnifiedGraph`，属于浅接口：调用者只需要很小的数据，却必须理解并消费全图结构。
- `Math.random()` 布局导致刷新后节点位置不稳定，不利于缓存布局和用户定位。

**优化方案**:

1. 新增轻量汇总接口：
```http
GET /lg/projects/{projectId}/graph/summary?versionId=...
```
返回 `nodeCount/edgeCount/typeCounts/statusCounts/sourceTypeCounts`，用于顶部统计和筛选面板。

2. 新增窗口查询接口：
```http
GET /lg/projects/{projectId}/graph/window
  ?versionId=...
  &nodeTypes=Service,Controller
  &sourceTypes=CODE_AST,LLM_INFERRED
  &status=CONFIRMED,PENDING_CONFIRM
  &minConfidence=0.5
  &cursor=...
  &limit=500
```
返回：
```json
{
  "nodes": [],
  "edges": [],
  "nextCursor": "...",
  "hasMore": true,
  "totalNodeCount": 12345,
  "totalEdgeCount": 67890
}
```

3. 新增节点邻域接口：
```http
GET /lg/projects/{projectId}/graph/nodes/{nodeId}/neighborhood
  ?versionId=...
  &depth=1
  &edgeTypes=CALLS,USES_TABLE
  &limit=300
```
用于点击节点后增量展开，避免首屏加载全图。

4. 替换下拉/详情页面的全图读取：
- `RiskDetail.vue`：改用 `GET /graph/nodes/{nodeId}` 或 `/graph/nodes/search?id=...`。
- `TestCaseEditor.vue`：改用 `GET /graph/node-options?types=ApiEndpoint,Feature,Table&limit=200`。

5. 前端布局稳定化：
- 初始位置从 `hash(node.id)` 或后端 layout cache 派生，不再用 `Math.random()`。
- `GraphViewerOptimized` 保留 worker 布局能力，但输入从“全量图”改为“窗口图 + 邻域增量”。

**预期收益**:
- 大图首屏从”全量加载”变成”统计 + 首批 300-500 节点”，降低超时和浏览器卡顿风险。
- 后端接口更深：调用者只表达视图意图，筛选、分页、邻域展开由图谱查询模块承接。
- 下拉和详情页面不再被全图接口拖慢。

**API 兼容性与迁移方案**:
```
旧接口: GET /lg/projects/{projectId}/graph/unified
       → 保留但标记 @Deprecated，1-2 个小版本后移除
       → 内部委托到 /graph/window（limit 设大值，向下兼容）

新接口: GET /lg/projects/{projectId}/graph/summary      (新增)
       GET /lg/projects/{projectId}/graph/window        (新增)
       GET /lg/projects/{projectId}/graph/nodes/{nodeId}/neighborhood (新增)

迁移路径:
  v1.0  旧接口正常使用 + 新接口上线（灰度）
  v1.1  前端适配新接口，旧接口标记 @Deprecated
  v1.2  移除旧接口
```

### 1.2 Neo4j 索引优化（配合窗口查询）

窗口查询的核心 Cypher 模式依赖 `projectId + versionId + nodeType` / `nodeKey` / `edgeType` 的快速过滤。如果没有对应索引，即使做了窗口分页，Neo4j 内部仍然是全表扫描。

**创建索引**（详见 [01-数据库查询优化.md §3.2](./01-数据库查询优化.md#32-neo4j-索引优化)）:
```cypher
CREATE INDEX node_project_version_type IF NOT EXISTS
FOR (n:Node) ON (n.projectId, n.versionId, n.nodeType);

CREATE INDEX node_project_version_key IF NOT EXISTS
FOR (n:Node) ON (n.projectId, n.versionId, n.nodeKey);

CREATE INDEX edge_project_version_type IF NOT EXISTS
FOR ()-[e:Edge]-() ON (e.projectId, e.versionId, e.edgeType);
```

> 建索引应与窗口查询接口上线同步执行，否则窗口查询的实际性能提升有限。

### 1.3 Neo4j 连接池调优

**位置**: `application.yml` Neo4j driver 配置

窗口查询 + 邻域展开使 Neo4j 查询频率和并发度上升，连接池需要相应调整：

```yaml
spring:
  neo4j:
    pool:
      max-connection-pool-size: 50       # 从默认值提升，支持图谱窗口并发查询
      connection-acquisition-timeout: 10000  # 10s 超时，快速失败
      max-connection-lifetime: 3600s     # 1 小时连接最大存活
```

### 1.4 图谱响应 GZIP 压缩

**位置**: `application.yml`

窗口查询虽然减少了单次返回的节点数，但 JSON 响应仍可能较大（节点属性多）。开启 HTTP 层 GZIP 压缩可显著缩小传输体积：

```yaml
server:
  compression:
    enabled: true
    mime-types: application/json,application/xml,text/html,text/plain
    min-response-size: 1024   # 超过 1KB 才压缩
```

**预期收益**: JSON 文本压缩率 5-10x，大图首屏网络传输耗时降低 50-70%。

## 二、QA 召回链路 fan-out/fan-in

### 2.1 `HybridRetrievalService.retrieve` 多路召回串行

**位置**:
- `backend/src/main/java/io/github/legacygraph/service/qa/HybridRetrievalService.java:31-78`
- `backend/src/main/java/io/github/legacygraph/service/qa/VectorRetrievalService.java:105-139`
- `backend/src/main/java/io/github/legacygraph/agent/EnhancedQaAgent.java:120-240`

**代码证据**:
- `EnhancedQaAgent.answerStream()` 在改写/HyDE 后调用 `hybridRetrievalService.retrieve(...)`。
- `HybridRetrievalService.retrieve()` 顺序执行：主查询向量检索 -> 每个 query variant 向量检索 -> 关键词检索。
- `VectorRetrievalService.semanticSearch()` 虽有 `vec:search` 缓存，但 miss 后每个 query 都单独 `embeddingModel.embed(query)` 并查 pgvector。

**问题**:
- 查询改写产生多个变体时，召回耗时接近多个 embedding + pgvector 查询的总和。
- 同一问题在高并发下同时缓存 miss，会重复向量化和重复查库。
- 主查询、HyDE 查询、关键词检索之间大多可并行，当前接口没有利用这个天然并行度。

**优化方案**:

```java
List<String> variants = Stream.concat(Stream.of(query), queryVariants.stream())
    .filter(StringUtils::hasText)
    .distinct()
    .limit(maxVariants)
    .toList();

List<CompletableFuture<List<VectorDocument>>> vectorFutures = variants.stream()
    .map(q -> CompletableFuture.supplyAsync(
        () -> vectorRetrievalService.semanticSearch(projectId, effectiveVersionId, q, topKFor(q), null),
        qaRetrievalExecutor
    ).completeOnTimeout(List.of(), 2, TimeUnit.SECONDS)
     .exceptionally(ex -> List.of()))
    .toList();

CompletableFuture<List<VectorDocument>> keywordFuture = CompletableFuture.supplyAsync(
    () -> keywordSearch(projectId, effectiveVersionId, query, topK / 2),
    qaRetrievalExecutor
).completeOnTimeout(List.of(), 800, TimeUnit.MILLISECONDS);

CompletableFuture.allOf(Stream.concat(vectorFutures.stream(), Stream.of(keywordFuture))
    .toArray(CompletableFuture[]::new)).join();
```

同时补充：
- `VectorRetrievalService` 增加 single-flight：同一个 `vec:search` key 在 miss 时只有一个线程执行 embedding 和 pgvector 查询。
- 配置 `qa.retrieval.max-variants`、`qa.retrieval.timeout-ms`、`qa.retrieval.max-concurrency`，避免并发化后打爆 embedding 服务。
- 合并结果时记录来源权重：主查询 > HyDE > 改写变体 > 关键词。

**预期收益**:
- 召回阶段延迟从 `T主查询 + ΣT变体 + T关键词` 降到接近 `max(T主查询, T变体, T关键词)`。
- 缓存 miss 风暴时减少重复 embedding 调用。

## 三、QA 图扩展改为多源 BFS 批量展开

### 3.1 `expandGraph` 多 seed、多 hop 下 Neo4j 往返放大

**位置**:
- `backend/src/main/java/io/github/legacygraph/agent/EnhancedQaAgent.java:304-368`
- `backend/src/main/java/io/github/legacygraph/dao/Neo4jQueryRepository.java:79-91`
- `backend/src/main/java/io/github/legacygraph/dao/Neo4jQueryRepository.java:344-359`

**问题**:
- 当前逻辑按 seed 节点逐个调用 `findNeighborNodeIds(projectId, seed.getId())`，再对每个 seed 的邻居调用 `findNodesByIds(...)`。
- 多个 seed 或深度大于 1 时，会形成多次 Neo4j 网络往返。
- 代码中每层循环使用 `new ArrayList<>(seedNodes)`，需要重点复核是否推进到下一层 frontier；如果未推进，`depth > 1` 时会重复扩展原始 seed，既浪费查询，也影响多跳上下文完整性。

**优化方案**:

新增 DAO 方法：
```java
Map<String, Set<String>> findNeighborNodeIdsBySources(
    String projectId,
    Collection<String> sourceNodeIds,
    Collection<String> edgeTypes,
    int perNodeLimit
);
```

批量 BFS：
```java
Set<String> visited = new LinkedHashSet<>(seedIds);
Set<String> frontier = new LinkedHashSet<>(seedIds);

for (int level = 1; level <= depth && !frontier.isEmpty(); level++) {
    Map<String, Set<String>> neighbors = neo4jGraphDao.findNeighborNodeIdsBySources(
        projectId, frontier, edgeTypes, perNodeLimit);
    Set<String> next = neighbors.values().stream()
        .flatMap(Set::stream)
        .filter(visited::add)
        .limit(maxExpandedNodes)
        .collect(Collectors.toCollection(LinkedHashSet::new));
    frontier = next;
}

List<GraphNode> expanded = neo4jGraphDao.findNodesByIds(new ArrayList<>(visited));
```

**预期收益**:
- 每层从 `seedCount * 2` 次查询降到约 2 次查询。
- 多跳语义更清楚，避免重复展开原始 seed。
- QA 上下文稳定，便于按节点类型/边类型控制 token 预算。

## 四、写图 outbox 批量化与 worker 有界并发

### 4.1 `GraphWriteIntentWorker` 批量拉取但串行处理

**位置**:
- `backend/src/main/java/io/github/legacygraph/task/GraphWriteIntentWorker.java:41-80`
- `backend/src/main/java/io/github/legacygraph/builder/EvidenceGraphWriter.java:497-555`

**代码证据**:
- `processPendingIntents()` 每 30 秒拉一批 `pending`，但随后 `for` 循环逐个 `processIntent(entity)`。
- `writeIntent()` 内部继续逐条遍历 `nodeClaims`、`edgeClaims`、`evidenceRecords`，分别调用 `upsertNode`、`upsertEdge`、`insertOrIgnore`。
- `FactRepository.batchUpsert` 已经证明本项目接受批量 upsert 模式，因此写图链路不应停留在逐条远程 MERGE。

**问题**:
- 一个 intent 里有大量 claims 时，Neo4j/PG 往返按 claim 数线性增长。
- Worker 批量拉取后串行处理，吞吐受单线程限制。
- Evidence 独立事务能隔离失败，但逐条事务成本高，且缺少批次级成功/失败统计。

**优化方案**:

1. Worker 有界并发：
```java
for (GraphWriteIntentEntity entity : pending) {
    futures.add(CompletableFuture.runAsync(() -> processIntent(entity), graphWriteExecutor));
}
CompletableFuture.allOf(futures.toArray(CompletableFuture[]::new)).join();
```
并通过配置限制：
```yaml
legacygraph:
  graph-write:
    worker-concurrency: 4
    batch-size: 50
```

2. `EvidenceGraphWriter` 增加批量接口：
```java
BatchNodeUpsertResult batchMergeNodes(List<GraphNodeClaim> claims);
BatchEdgeUpsertResult batchMergeEdges(List<GraphEdgeClaim> claims);
int batchInsertEvidence(List<EvidenceRecord> records);
```

3. Neo4j 使用 `UNWIND $nodes MERGE`：
```cypher
UNWIND $nodes AS row
MERGE (n:Node {
  projectId: row.projectId,
  versionId: row.versionId,
  nodeType: row.nodeType,
  nodeKey: row.nodeKey
})
ON CREATE SET n.id = row.id, n.createdAt = row.createdAt
SET n += row.properties, n.updatedAt = row.updatedAt
RETURN n.id AS id, row.id AS requestId
```

4. 保留当前失败隔离语义：
- 批量返回每条 claim 的 `success/error/created`。
- 对失败 claim 写入 intent 处理结果，而不是直接把整个 intent 标成成功。
- Evidence 失败继续标记 `INCOMPLETE`，交给 `GraphWriteReconciler`。

**预期收益**:
- 单个 intent 的 Neo4j 往返从 `N节点 + M边` 降到 `1-数批节点 + 1-数批边`。
- Worker 吞吐受配置控制，不再被单线程串行限制。
- 批次指标可直接定位写图瓶颈。

## 五、向量检索缓存加入模型版本与主动失效

### 5.1 `vec:search` key 未显式包含 embedding/corpus 版本

**位置**:
- `backend/src/main/java/io/github/legacygraph/service/qa/VectorRetrievalService.java:52-55`
- `backend/src/main/java/io/github/legacygraph/service/qa/VectorRetrievalService.java:105-139`

**问题**:
- 当前 key 来源为 `projectId|versionId|topK|chunkType|query` 的 SHA256，未显式包含 embedding 模型、向量语料版本、分片策略版本。
- 文档重新向量化后，如果 versionId 不变，只依赖短 TTL 失效，仍可能短时间返回旧召回结果。
- Redis 优化文档已提到 `VectorRetrievalService` 缓存失效不完整，这里补齐落地边界。

**优化方案**:

```java
private String searchCacheKey(SearchCacheContext ctx) {
    String raw = String.join("|",
        ctx.projectId(),
        ctx.versionId(),
        ctx.embeddingModel(),
        ctx.corpusVersion(),
        ctx.chunkStrategyVersion(),
        String.valueOf(ctx.topK()),
        String.valueOf(ctx.chunkType()),
        ctx.query()
    );
    return "vec:search:" + ctx.projectId() + ":" + ctx.versionId() + ":" + sha256(raw);
}
```

向量写入完成后触发：
```java
applicationEventPublisher.publishEvent(
    new VectorCorpusUpdatedEvent(projectId, versionId, embeddingModel, corpusVersion)
);
```

监听器执行：
```java
cacheService.evictByPrefix("vec:search:" + projectId + ":" + versionId + ":");
```

**VectorCorpusUpdatedEvent 定义**:
```java
package io.github.legacygraph.event;

public record VectorCorpusUpdatedEvent(
    String projectId,
    String versionId,
    String embeddingModel,    // 如 "text-embedding-3-small"
    String corpusVersion      // 如 "v2" 或时间戳 "2026-07-08T12:00:00Z"
) {}
```

**预期收益**:
- 模型升级、分片策略调整、向量重建后不复用旧结果。
- 失效范围从全局 `vec:search:*` 收敛到项目和版本粒度。
