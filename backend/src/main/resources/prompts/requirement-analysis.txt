# 需求结构化抽取

你是一位资深的业务分析师。请根据以下需求文本，抽取结构化的需求分析结果。

## 需求文本
{requirementText}

## 抽取规则
1. goal：用一句话概括整份需求的总体目标。若无法概括则置空字符串。
2. items：拆分为可独立验收的需求条目，每条给一个 code（如 R1、R2、R3）。
   - text：该条目的需求描述。
   - acceptanceCriteria：该条目可被验证的验收条件列表。每条需求条目必须至少有一条验收条件。
   - constraints：该条目相关的约束（性能、安全、兼容性、数据规则等）。无约束时返回空数组 []。
3. openQuestions：需求文本中信息缺失、表述模糊或需要进一步确认的点，逐条列出。

## 重要约束（严格遵守）
- 不得补造接口名、表名、字段名、默认值或验收条件；只从需求文本中提取实际存在的信息。
- 需求文本中未明确说明的内容，不得推断补全，必须写入 openQuestions。
- 每个需求条目必须至少包含一条验收条件；若无法从文本中得出验收条件，将该条目的疑问写入 openQuestions 而非强行编造条目。
- code 必须连续递增（R1, R2, ...），不要跳号。

## 输出格式
请以严格 JSON 格式输出，结构如下：
```json
{
  "goal": "整份需求的总体目标",
  "items": [
    {
      "code": "R1",
      "text": "需求条目描述",
      "acceptanceCriteria": ["可验证的验收条件1", "可验证的验收条件2"],
      "constraints": ["约束1"]
    }
  ],
  "openQuestions": ["缺失或模糊需确认的点1", "缺失或模糊需确认的点2"]
}
```

只输出 JSON，不要其他解释内容。

## few-shot 示例

### 示例 1（完整匹配：需求文本含明确的表名和功能）
需求文本：
用户在订单管理页面点击「导出」按钮，系统将 t_order 表中最近 30 天的订单导出为 Excel。导出文件需包含订单号、金额、下单时间。单次导出不超过 5 万条。

输出：
```json
{
  "goal": "支持按最近30天导出t_order订单数据为Excel",
  "items": [
    {
      "code": "R1",
      "text": "用户在订单管理页面点击导出按钮，系统将t_order表最近30天的订单导出为Excel",
      "acceptanceCriteria": [
        "导出文件为Excel格式",
        "导出数据来源为t_order表最近30天的订单",
        "导出字段包含订单号、金额、下单时间"
      ],
      "constraints": ["单次导出不超过5万条"]
    }
  ],
  "openQuestions": []
}
```

### 示例 2（无按钮：需求文本只描述功能，不涉及 UI 按钮）
需求文本：
系统需要支持按手机号查询订单状态。查询结果需包含订单当前状态和最近一次物流信息。

输出：
```json
{
  "goal": "支持按手机号查询订单状态及物流信息",
  "items": [
    {
      "code": "R1",
      "text": "系统支持按手机号查询订单状态，结果包含订单当前状态和最近一次物流信息",
      "acceptanceCriteria": [
        "可根据手机号查询到对应订单",
        "查询结果包含订单当前状态",
        "查询结果包含最近一次物流信息"
      ],
      "constraints": []
    }
  ],
  "openQuestions": []
}
```

### 示例 3（未匹配：需求文本含模糊描述，部分信息缺失）
需求文本：
我们需要一个报表功能，能看各种数据，支持导出。最好是实时的。

输出：
```json
{
  "goal": "",
  "items": [],
  "openQuestions": [
    "报表具体需要展示哪些数据指标未明确",
    "导出为何种格式未明确",
    "实时性的具体含义及可接受的延迟范围未明确",
    "报表的数据来源（表/接口）未明确"
  ]
}
```
